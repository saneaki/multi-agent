## 3. モデル特性詳細

### 3.1 Sonnet 4.6 — 「自己レビューの王」

**強み**:
- 最小差分主義 — 元コード構造を最大限尊重
- 自己懸念検出精度が3モデル中最高 (実装詳細レベルまで自分で発見)
- 改善提案に diff コードを併記 — 実装直結度が高い
- バランスの良さ (ボリューム/精度/速度すべて中位)

**弱み**:
- エッジケース見落とし (例: B1 で macOS 非互換を後出し)
- 構造的概念化能力は中位

**ベストフィット**:
- 既存コードへの最小侵襲な改修
- 実装後の精密な自己レビュー
- 短時間で本番投入したい場合
- QC レポート、コードレビュー

---

### 3.2 Opus 4.6 + Extended Thinking — 「思考の深さの王」

**強み**:
- 設計概念化能力 (例: 「二層配達モデル」のように設計核を命名)
- 網羅性 (改善提案数・評価軸数・テストケース数・引用数すべて最多)
- 一次情報の正確性 (二次情報の誤りを GitHub API で訂正)
- フェーズ間知見転送 (過去分析の発見を後続実装で構造的解決)
- 防御的設計 (defense-in-depth, 多パターン後方互換)
- 運用 UX 追加価値 (仕様外の改善を自発提案)

**弱み**:
- トークンコストが3モデル中最高 (Sonnet の 1.9倍、Codex の 5.7倍)
- 過剰設計の傾向 (シンプル案件にも複数代替案検討)
- 応答速度が遅い (Extended Thinking ON のため)

**ベストフィット**:
- アーキテクチャ設計・大規模リファクタリング
- 既存コード深層解析・依存関係マップ作成
- ライブラリ選定・技術判断
- 複雑な状態遷移を伴う実装
- 長期保守性が最重要なタスク
- フェーズ跨ぎの一貫性が必要な多段プロジェクト

---

### 3.3 Codex 5.3 — 「構造化の隠れた王」

**強み**:
- トークン効率最良 (出力ボリューム最少だが結論品質は同等)
- **構造的設計**: 関数分離・環境変数による外部調整・責務分離
- 静的解析の正確性 (bash 構文・関数境界判定が正確)
- WebSearch なしでの調査能力 (`raw.githubusercontent.com` から直接取得)
- read-only 制約遵守

**弱み**:
- 分析系では深度に劣る (主要関数のみ・主要環境変数のみ)
- 改善提案の具体性が低い (方針止まり、diff 併記なし)
- 自己懸念検出件数が少ない
- 設計概念化が弱い

**ベストフィット**:
- 単一ファイルのコード生成 (関数追加・既存パターン適用)
- 構造的責務分離が重要な実装 (関数抽出が自然)
- トークンコストを抑えたい繰り返しタスク
- 一次情報源 URL が明確な調査
- read-only 制約下での修正版生成

**🌟 重要な注意 — 特性逆転**:
> Codex はフェーズ1の分析系では「最簡素」だが、フェーズ2の実装系では「最も構造的」と逆転する。Codex を分析系で評価して足切りすると、実装系で発揮される構造化能力を見逃す危険がある。**実装タスクでは Codex を必ず候補に入れよ**。

## 4. 並列実行パターン

複数足軽の並列配備時のハイブリッド活用パターン:

| パターン | 用途 | 例 |
|---------|------|-----|
| **3者並列 (品質横並び)** | 重要決定の事前比較 | 技術選定、実証実験 (cmd_463 型) |
| **Opus + Sonnet** | 設計と実装の二段 | Opus が設計→Sonnet が実装 |
| **Sonnet + Codex** | 実装の冗長性 | 同じ仕様を2モデルで書いて比較 |
| **Opus 単独** | 一貫性必須の長期タスク | フェーズ間知見転送が必要なケース |
| **Codex 単独** | 高頻度の小タスク | 関数追加・テスト追加・docstring 追加 |

## 5. アンチパターン (避けるべき組合せ)

| アンチパターン | なぜ駄目か |
|---------------|-----------|
| Codex に L4-L5 の概念化タスク | 設計核命名能力が低く、結論が方針止まりになる |
| Opus に L1-L2 の単純検索 | トークンコスト過剰、納期に対して oversized |
| Sonnet に 1000行超の依存解析 | Opus の概念化能力に届かず、改善提案が局所的に留まる |
| 全タスクを単一モデルに集約 | 特性逆転 (Codex Phase1↔Phase2) を見逃す |
| 同一足軽1人に長期連続タスク | コンテキスト圧迫リスク。並列分散すべし |

## 6. cmd 設計時のチェックリスト

家老が cmd を書く際は、以下を必ず判定せよ:

- [ ] **task の主目的**: 分析 / 実装 / 設計 のいずれか
- [ ] **想定 Bloom レベル**: L1-L6 (詳細は `instructions/karo.md` 参照)
- [ ] **行数規模**: 〜100 / 100-300 / 300-800 / 800-1500 / 1500+
- [ ] **エッジケース重要度**: 低 / 中 / 高
- [ ] **後方互換必須度**: 低 / 中 / 高
- [ ] **概念化価値**: 設計核を命名する価値があるか
- [ ] **トークンコスト感度**: 高頻度繰返しか / 一度限りか
- [ ] **時間制約**: 緊急 / 通常 / 余裕あり

これらをもとに本ガイドの **§1 デフォルト割当ルール** または **§2 Routing Matrix** を参照し、最適モデルを選択せよ。

## 8. 実証データの要約

cmd_463 の実証結果から、以下の数値的事実が確認された:

| 軸 | Sonnet | Opus | Codex | 倍率 (max/min) |
|----|:---:|:---:|:---:|:---:|
| 設計書ボリューム平均 (b2) | 282行 | 528行 | 93行 | **5.7x** |
| 評価軸数 (a1) | 5 | 8 | 5 | 1.6x |
| 改善提案件数 (a2) | 5 | 12 | 5 | 2.4x |
| URL引用数 (a1) | 10 | 20 | 9 | 2.2x |
| 自己懸念検出件数 (b2) | 4 | 9 | 2 | 4.5x |
| AC 充足率 | 100% | 100% | 100% | 1.0x |
| QC PASS 率 | 100% | 100% | 100% | 1.0x |

**結論**: 全モデルが本番適用可能だが、「網羅性 vs 効率性 vs 構造性」のトレードオフが明確に存在する。タスク要件に応じて最適な軸を選択せよ。
