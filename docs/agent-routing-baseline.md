## 1.5 ロードバランシングは routing baseline より優先する

<!-- cmd_471 (2026-04-08) で追加。Sonnet 偏重防止 + Opus/Codex 稼働率向上。 -->
<!-- 出典: cmd_468 フェーズ1 (家老が3調査タスクを Sonnet 1〜3号に集中→Opus/Codex が13〜31分アイドル化) -->

### 原則

- 本ガイドの **§1 デフォルト割当ルール** および **§2 Routing Matrix** は「**タスク種別ごとの理論最適**」を示すマトリクスである
- しかし現実の足軽配分は **「現状の負荷分布」も加味** しなければならない
- **ロードバランシングは routing baseline より優先する** (= 理論最適でも待機足軽が多い場合は再分配せよ)

### 理由

タスク種別だけで決めると Sonnet 1〜3号 (Sonnet 最適タスクが多い) に集中し、Opus 4/5号と Codex 6/7号がアイドル化する。並列度が落ち、結果として cmd 全体の所要時間が延びる。

### 適用ルール (要約)

詳細は `instructions/karo.md` 「足軽ロードバランシングルール」セクション参照。

| ID | ルール |
|----|--------|
| (a) | タスク割当前に全足軽の現アイドル時間を確認すること |
| (b) | **5分以上アイドル**の Opus/Codex 足軽が居れば、Sonnet 最適タスクでも Opus/Codex に優先的に振る (品質80%超担保できる範囲で) |
| (c) | Sonnet 最適度 ≥4 (5段階) かつ Opus/Codex で品質劣化リスク高い場合のみ例外。例外時は task YAML の `notes` に「Sonnet選定理由」を明記 |
| (d) | 並列タスクは **モデル多様化必須** (全員 Sonnet 禁止)。3並列なら最低 Sonnet1+Opus1+Codex1 等 |

### 調査系の軍師シフト回避

調査系 cmd (WebSearch / 分析 / 比較 / 設計検討) の **第一候補は Opus 4/5号**。軍師に直接振るのは「**Opus 足軽が全員稼働中** かつ **締切タイト**」の例外時のみ。軍師は QC + 統合 + 戦況分析に集中させる。詳細は `instructions/gunshi.md` 「調査タスク受諾基準」セクション参照。

### 違反例 (cmd_468 フェーズ1, 2026-04-08)

- 家老が3調査タスクを全て Sonnet 1〜3号に割当 → Opus 4/5号と Codex 6/7号が **13〜31分アイドル化**
- 軍師は調査+QC兼務で **1h22m 停滞**
- 教訓: routing baseline の機械的適用は **負荷分布の片寄りを生む**。タスク種別 + 負荷分布の二軸判断が必須。

### 適用実例

- **cmd_470 フェーズ1** (2026-04-08): Sonnet1 + Opus1 + Codex1 の3並列配分が本ルール初適用例
- **cmd_471** (2026-04-08, 本cmd): karo.md / gunshi.md / routing-baseline.md / global_context.md にルール明文化。本cmd 自身も Opus 4 または Opus 5号で先取り適用 (Sonnet 禁止指定)

## 3. モデル特性詳細

### 3.1 Sonnet 4.6 — 「自己レビューの王」

**強み**:
- 最小差分主義 — 元コード構造を最大限尊重
- 自己懸念検出精度が3モデル中最高 (実装詳細レベルまで自分で発見)
- 改善提案に diff コードを併記 — 実装直結度が高い
- バランスの良さ (ボリューム/精度/速度すべて中位)
- トークン費用対効果最良 (cmd_466 平均 24K tokens / Opus の 1/4)
- **(cmd_466 C1/C2) QC・テスト計画で bats 直結度・後方互換 CI gate 化提案など実装直結価値**

**弱み**:
- エッジケース見落とし (例: cmd_463 B1 で macOS 非互換を後出し)
- 構造的概念化能力は中位
- **自己報告精度の限界** (cmd_466 D1: md 275/171 と自己申告 vs 実測 167/145 → 約 39% 過大申告)
  - cmd_463 B1 の「macOS 非互換を後出し」と同質の自己検証精度の限界
  - **運用ルール**: Sonnet 成果物は `wc -l`・`grep -c "^## "`・`jq empty`・`bash -n` で軍師 QC 時に必ず実測検証せよ
  - Opus/Codex は cmd_466 全 8 タスクで自己報告 = 実測値で完全一致 (Sonnet のみ発生する事象)

**ベストフィット**:
- 既存コードへの最小侵襲な改修
- 実装後の精密な自己レビュー
- 短時間で本番投入したい場合
- QC レポート、コードレビュー
- **(cmd_466 新) コードレビュー短〜中 / テスト計画 (20件以下) / n8n WF 設計 (〜300行)**

---

### 3.2 Opus 4.6 + Extended Thinking — 「思考の深さの王」

**強み**:
- 設計概念化能力 (例: 「二層配達モデル」「Border-aware」「Daily Frog Guard」のように設計核を命名)
- 網羅性 (改善提案数・評価軸数・テストケース数・引用数すべて最多)
- 一次情報の正確性 (二次情報の誤りを GitHub API で訂正。cmd_466 D2: 5 件訂正)
- フェーズ間知見転送 (過去分析の発見を後続実装で構造的解決)
- 防御的設計 (Defense-in-Depth, 多パターン後方互換。cmd_466 D1: 7 層)
- 運用 UX 追加価値 (仕様外の改善を自発提案)
- 自己報告精度完全一致 (cmd_466 全 8 タスクで自己報告 = 実測値)

**弱み**:
- トークンコストが3モデル中最高 (cmd_466 実証: Sonnet の **約 4 倍**、Codex の **約 2.7 倍**)
  - cmd_466 平均: Sonnet 24K / Codex 35K / **Opus 95K** tokens
  - 最大値: Opus 130K (cmd_466 D2) — Sonnet の 5.2 倍
- 過剰設計の傾向 (シンプル案件にも複数代替案検討)
- 応答速度が遅い (Extended Thinking ON のため)

**ベストフィット**:
- アーキテクチャ設計・大規模リファクタリング
- 既存コード深層解析・依存関係マップ作成
- ライブラリ選定・技術判断
- 複雑な状態遷移を伴う実装
- 長期保守性が最重要なタスク
- フェーズ跨ぎの一貫性が必要な多段プロジェクト
- **(cmd_466 新) 深層コードレビュー (400行+) / テスト計画 (30件+edge case) / n8n WF 設計 (9ノード+代替案) / 深層調査 (8軸+一次情報訂正)**

**🚨 Opus 選択ルール (cmd_466 費用対効果分析より)**:

> **「Opus を選ぶなら、以下の 4 要素のうち 2 つ以上が価値を持つ場合のみ」**
>
> 1. **概念化価値**: 設計核命名 (Border-aware / Daily Frog Guard / 二層配達モデル等) が後続運用価値を生む
> 2. **一次情報訂正**: 二次情報の誤りを GitHub API 等で訂正する必要がある
> 3. **代替案体系比較**: A/B/C を 5 軸 × 5 段階でスコアリングする価値がある
> 4. **フェーズ間知見転送**: 過去の cmd の発見を後続タスクに構造的に適用する
>
> - **1 つだけ** → Sonnet または Codex で代替可能
> - **2 つ以上** → Opus の 4 倍コストを正当化できる
> - **3 つ以上** → Opus 必須

**L3-L4 の通常タスクには Opus は過剰投資**。Sonnet または Codex で十分。

---

### 3.3 Codex 5.3 — 「構造化の隠れた王」

**強み**:
- トークン効率最良 (出力ボリューム最少だが結論品質は同等。cmd_466 平均 35K)
- **構造的設計**: 関数分離・環境変数による外部調整・責務分離
- 静的解析の正確性 (bash 構文・関数境界判定が正確)
- WebSearch なしでの調査能力 (`raw.githubusercontent.com` / `curl` から直接取得)
- read-only 制約遵守
- 自己報告精度完全一致 (cmd_466 全 8 タスクで自己報告 = 実測値)

**弱み**:
- 分析系では深度に劣る (主要関数のみ・主要環境変数のみ)
- 改善提案の具体性が低い (方針止まり、diff 併記なし)
- 自己懸念検出件数が少ない
- 設計概念化が弱い
- 深層調査 (8軸以上) では Opus に大差

**ベストフィット**:
- 単一ファイルのコード生成 (関数追加・既存パターン適用)
- 構造的責務分離が重要な実装 (関数抽出が自然)
- トークンコストを抑えたい繰り返しタスク
- 一次情報源 URL が明確な調査
- read-only 制約下での修正版生成
- **(cmd_466 新) 短い WF 設計 (3-6ノード) / 短いコードレビュー (関数分離提案)**

**🌟 重要な注意 — 特性逆転 (cmd_463 + cmd_466 で継続観察)**:

> Codex の特性は「**情報圧縮 vs 情報拡張**」の軸で理解せよ。
>
> | タスク性質 | Codex の振る舞い | 結果 |
> |-----------|----------------|------|
> | **分析系 (A1/A2/D2)** | 情報圧縮して最簡素に要点提示 | 深度不足 / 足切りされる |
> | **実装系 (B1/B2)** | 関数分離・責務分離で構造的に拡張 | 最も構造的・移植性最高 |
> | **設計系 (D1 短 / C1 短)** | 最小構造 + 外部設定化 | 最も移植性高い設計 |
> | **レビュー系 (C1 長 / C2 長)** | 情報圧縮が弱みとして顕在化 | Opus の深度に劣る |
>
> **運用ルール**:
> 1. **分析系で評価して足切りするな** — 実装系・設計系で発揮される構造化能力を見逃す
> 2. **実装タスクでは必ず Codex を候補に入れよ** (cmd_463 B1/B2 → cmd_466 D1 短で一貫観察)
> 3. **短い設計書・短いコードレビューでは Codex が第一候補** (関数分離・外部設定化が価値を生む)
> 4. **深層タスク (L4+) では Codex を避けよ** (情報圧縮が弱みになる)

## 4. 並列実行パターン

複数足軽の並列配備時のハイブリッド活用パターン:

| パターン | 用途 | 例 |
|---------|------|-----|
| **3者並列 (品質横並び)** | 重要決定の事前比較 | 技術選定、実証実験 (cmd_463 型) |
| **cmd_466 型 (実証比較研究)** | routing baseline 再検証・新タスク種別の追加検証 | 同一タスクを 3 モデル並列で実行し、品質・行数・トークン・自己報告精度を比較 |
| **Opus + Sonnet** | 設計と実装の二段 | Opus が設計→Sonnet が実装 |
| **Sonnet + Codex** | 実装の冗長性 | 同じ仕様を2モデルで書いて比較 |
| **Opus + Codex** | 概念化と構造化の補完 | Opus が設計核命名→Codex が関数分離実装 |
| **Opus 単独** | 一貫性必須の長期タスク | フェーズ間知見転送が必要なケース |
| **Codex 単独** | 高頻度の小タスク | 関数追加・テスト追加・docstring 追加 |

**cmd_466 型の適用条件**:
- 新タスク種別が routing baseline 未検証の場合
- 既存 baseline の陳腐化懸念があり再検証が必要な場合
- 3 モデルの特性が逆転する可能性がある場合 (特に Codex)
- 結論に 3 モデル一致が求められる調査系 (推奨の正当性強化)

**cmd_466 型の運用手順**:
1. 同一タスクの task YAML を 3 つ作成 (各足軽に Sonnet/Opus/Codex 割当)
2. 並列実行 → 軍師 QC で各モデルの成果物を比較
3. ルーティングマトリクスに結果を反映 → docs/agent-routing-baseline.md 更新

## 5. アンチパターン (避けるべき組合せ)

| アンチパターン | なぜ駄目か |
|---------------|-----------|
| Codex に L4-L5 の概念化タスク | 設計核命名能力が低く、結論が方針止まりになる |
| Opus に L1-L2 の単純検索 | トークンコスト過剰、納期に対して oversized |
| Sonnet に 1000行超の依存解析 | Opus の概念化能力に届かず、改善提案が局所的に留まる |
| 全タスクを単一モデルに集約 | 特性逆転 (Codex Phase1↔Phase2) を見逃す |
| 同一足軽1人に長期連続タスク | コンテキスト圧迫リスク。並列分散すべし |

## 6. cmd 設計時のチェックリスト

家老が cmd を書く際は、以下を必ず判定せよ:

- [ ] **task の主目的**: 分析 / 実装 / 設計 のいずれか
- [ ] **想定 Bloom レベル**: L1-L6 (詳細は `instructions/karo.md` 参照)
- [ ] **行数規模**: 〜100 / 100-300 / 300-800 / 800-1500 / 1500+
- [ ] **エッジケース重要度**: 低 / 中 / 高
- [ ] **後方互換必須度**: 低 / 中 / 高
- [ ] **概念化価値**: 設計核を命名する価値があるか
- [ ] **トークンコスト感度**: 高頻度繰返しか / 一度限りか
- [ ] **時間制約**: 緊急 / 通常 / 余裕あり

これらをもとに本ガイドの **§1 デフォルト割当ルール** または **§2 Routing Matrix** を参照し、最適モデルを選択せよ。
