# Notion 案件DB・タスクDB テンプレート提案

作成日: 2026-02-13

## 前提: v2.0 WF の同期動作

```
タスクページの本文ブロック → そのまま案件ページの対応履歴トグル内にコピーされる
```

**タスクページに書いた内容がそのまま案件の履歴記録になる。** テンプレート設計はこの前提に立つ。

### 現在のタスクDB プロパティ

| プロパティ | 型 | 値 |
|-----------|-----|-----|
| タスク名 | title | - |
| 状態 | status | 未着手/中断/進行中/完了 |
| 案件同期 | status | しない/予定/同期済 |
| タスク内容 | select | - |
| Memo | rich_text | - |
| 担当者 | people | - |
| 完了日時 | formula | - |
| 案件 | relation | → 案件DB |

### 対応履歴の自動追記フォーマット

- 案件ページ内の「対応履歴」heading_1 トグル内に追記
- heading_3 トグル: `YYYYMMDDHHmm　タスク内容　担当者　メモ`
- トグル内容 = タスクページの本文ブロック（コピー不可ブロック除去済み）

---

## 案A: 転写最適型 — 自動化との親和性を最大化

**設計思想**: タスクページの本文が「そのまま美しい履歴」になるよう逆算設計する

### タスクDB テンプレート

```
## 対応内容
（ここに実施した内容を書く — これが案件の対応履歴にそのまま転写される）

## 結果・所見
-

## 次のアクション
- [ ]
```

### 案件DB テンプレート

```
# 案件概要
| 項目 | 内容 |
|------|------|
| 依頼者 |  |
| 相手方 |  |
| 案件種別 |  |
| 受任日 |  |

# 対応履歴
（← WFが自動追記。手動記入不要）
```

**長所**: テンプレートがシンプル。タスク完了→履歴蓄積が自然に回る。運用負荷が最も低い。

**短所**: 案件ページに手動で管理する情報が少なく、案件の「全体像」を俯瞰しにくい。対応履歴が時系列で積み上がるだけで、論点別の整理がない。

---

## 案B: 二層構造型 — 案件=戦略俯瞰、タスク=戦術記録

**設計思想**: 案件ページは「今どうなっているか」を俯瞰する場。タスクは個別の作業記録。対応履歴は時系列ログとして自動蓄積。

### タスクDB テンプレート

```
## 作業記録
（実施内容 — 対応履歴に転写される）

## 確認事項・気づき
-

## 添付・参照
-
```

### 案件DB テンプレート

```
# 案件概要
| 項目 | 内容 |
|------|------|
| 依頼者 |  |
| 相手方 |  |
| 案件種別 |  |
| 受任日 |  |
| 担当 |  |

# 現在のステータス
（← 手動更新。案件の現況を常に最新化する）
- 進行段階:
- 直近の論点:
- 次回期日:

# 論点整理
## 論点1:
- 当方主張:
- 相手方主張:
- 証拠:

# 期日・マイルストーン
- [ ]

# 関連書類
| 書類名 | 種別 | 日付 | 備考 |
|--------|------|------|------|
|  |  |  |  |

# 対応履歴
（← WFが自動追記）
```

**長所**: 案件の全体像（論点・期日・書類）と時系列記録（対応履歴）が分離され、それぞれが読みやすい。案件ページを開けば「今何が問題で、これまで何をしたか」が一目で分かる。

**短所**: 「現在のステータス」「論点整理」は手動更新が必要。更新を怠ると情報が古くなる。テンプレートのセクション数が多く、軽微な案件には過剰。

---

## 案C: 類型別テンプレート型 — 案件種別で分岐

**設計思想**: 案件の種別（訴訟・契約・相談等）ごとに必要な情報が異なる。テンプレートを類型化し、各類型に最適化されたセクションを用意する。

### タスクDB テンプレート（共通）

```
## 対応内容
（対応履歴に転写される）

## 方針メモ
（転写されるが、内部検討の記録として有用）

## チェック
- [ ] 依頼者への連絡要否
- [ ] 期日への影響確認
```

### 案件DB テンプレート — 訴訟類型

```
# 案件概要
| 項目 | 内容 |
|------|------|
| 事件番号 |  |
| 裁判所 |  |
| 依頼者 |  |
| 相手方 |  |
| 代理人 |  |
| 受任日 |  |

# 請求の趣旨・争点
-

# 期日一覧
| 回 | 日時 | 種別 | 内容・結果 |
|----|------|------|-----------|
|  |  |  |  |

# 証拠一覧
| 号証 | 標目 | 立証趣旨 | 備考 |
|------|------|---------|------|
|  |  |  |  |

# 対応履歴
（← WFが自動追記）
```

### 案件DB テンプレート — 契約・顧問類型

```
# 案件概要
| 項目 | 内容 |
|------|------|
| 依頼者 |  |
| 相手方 |  |
| 契約種別 |  |
| 受任日 |  |

# 検討事項
- [ ]

# 関連書類
| 書類名 | 版 | 日付 | 備考 |
|--------|---|------|------|
|  |  |  |  |

# 対応履歴
（← WFが自動追記）
```

**長所**: 各案件に過不足ないテンプレート。訴訟なら期日・証拠一覧、契約なら検討事項・版管理など、実務に直結するセクションが最初から用意される。

**短所**: テンプレートの種類が増え、管理コストが上がる。Notion側でDB templateを複数設定する必要がある。新しい類型の案件が来るたびにテンプレート追加が必要。

---

## 三案比較

| 評価軸 | A: 転写最適 | B: 二層構造 | C: 類型別 |
|--------|:---------:|:---------:|:--------:|
| 運用負荷 | ◎ 最軽量 | △ 手動更新あり | △ テンプレ管理 |
| 情報の網羅性 | △ 最小限 | ◎ 俯瞰+詳細 | ◎ 類型特化 |
| WF連携の自然さ | ◎ 転写前提 | ○ 良好 | ○ 良好 |
| 案件の俯瞰性 | △ 履歴のみ | ◎ ステータス常時把握 | ○ 類型に依存 |
| 導入の容易さ | ◎ 即時 | ○ 1テンプレ | △ 類型ごとに設計 |
| 拡張性 | ○ | ○ | △ 類型増で複雑化 |

## 組み合わせ案

案件DBはB案、タスクDBはA案のシンプルな形、といった組み合わせも有効。
