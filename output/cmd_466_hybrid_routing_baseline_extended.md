## 1. 背景・目的

### 1.1 cmd_463 の成果と残課題

cmd_463 では調査系 (A1/A2) と実装系 (B1/B2) の 4 タスク種別 × 3モデルで routing baseline を策定した。しかし、cmd_463 の実証範囲には以下の残課題があった:

### 1.2 cmd_466 の北極星

> **「3モデル × 4タスク種 (C1/C2/D1/D2) = 12件の並列実行で cmd_463 で策定した 3 経験則を再検証し、routing baseline の精度を高める」**

### 1.3 アプローチ

4 タスク種 × 3 モデル = **12 並列タスク**を実施。同一 task description / 同一制約条件下で各モデルに同じ仕事をさせ、cmd_463 と同一の評価軸 (出力ボリューム/評価軸数/URL引用数/自己評価精度/AC 充足率/QC PASS 率) で横並び評価した。

**cmd_463 との差異**:
- cmd_463 は「4 タスク種 × 3 モデル = 12 並列」(A1/A2/B1/B2)
- cmd_466 は「4 タスク種 × 3 モデル = 12 並列」(C1/C2/D1/D2)
- **両者合算で 8 タスク種 × 3 モデル = 24 データポイント**で routing baseline を策定

> - Time Border: JST 00:00 で昨日/本日切替
> - Section Border: markdown `## ✅ 本日の戦果` rename + 旧昨日削除
> - State Border: Frog/streak/completed カウンタの 0 リセット
>
> **不変条件**: 3 境界の atomic 切替が rotation の本質的設計目標

この「設計概念命名」は cmd_463 A2 (inbox_watcher.sh を「二層配達モデル」と命名) の再現であり、**L5-L6 レベルの抽象化能力**が Opus に存在することを追検証した。

#### 5.2.2 Opus の「設計概念命名 + Defense-in-Depth 7層」

Opus は設計核を **"Daily Frog Guard"** と命名し、Guard (受動的防御層) + Daily (不変周期) + Frog (本システム固有概念継承) の三項で抽象化した。後続 WF (Daily Streak Guard / Daily Inbox Hygiene Guard 等) への**設計流用を可能にする抽象化**である。

**代替設計 9 案**を 3 軸 (データソース / 通知 / エラー処理) × 各 3 案で 5 評価軸 × 5 段階でスコアリング:

| 軸 | 採用案 | スコア |
|----|--------|--------|
| データソース | 案 A streaks.yaml | **24/25** |
| 通知 | 案 A inbox | **24/25** |
| エラー処理 | 案 A ErrorTrigger+ntfy | **23/25** |

**Defense-in-Depth 7層**:
- L1 入力 (streaks.yaml 不存在/YAML破損) → try/catch + regex 二段抽出
- L2 判定 (frog 値の型ゆれ/日付ずれ) → 三重 null 判定 + last_date 整合チェック
- L3 出力 (同時書込競合/重複) → atomic tmp+rename + dateKey 重複防止
- L4 通知失敗 (ntfy 不達) → appendFileSync fallback
- L5 監視 (n8n 自体停止) → 範囲外 (将来 WF)
- L6 時刻 (サーバ UTC) → new Date(Date.now() + 9*3600*1000) 明示変換

## 8. Codex 特性逆転の統合観察

### 8.1 特性逆転の定義

> **「分析系では最簡素、実装系・設計系では最も構造的」という Codex 固有の振る舞い**

cmd_463 で初観察された現象を、cmd_466 の 4 新タスク種別 (C1/C2/D1/D2) で追検証した結果、**全 4 種別で特性逆転が継続**することが確認された。

### 8.2 タスク種別ごとの Codex 立ち位置マトリクス

| タスク種別 | 期待される Codex 立ち位置 (cmd_463時点) | 実際の cmd_466 観察 | 特性逆転? |
|-----------|-----------------------------------|---------------------|:---:|
| C1 コードレビュー | 分析系 → 「最簡素」予想 | **関数分離候補 5 関数抽出 + 環境変数化提案** | ✅ 逆転 |
| C2 テスト計画 | 分析系 → 「最簡素」予想 | **関数境界マップ 4 関数 + 比較容易均質構造** | ✅ 逆転 |
| D1 n8n WF 設計 | 設計系 → 「最簡素」予想 | **3ノード最小構造 + 5関数分離 + 外部設定化** | ✅ 逆転 |
| D2 調査業務 | 分析系 → 「最簡素」予想 | **141行必要十分 + curl+GitHub API 直取得** | (境界ケース) |

**D2 は境界ケース**: D2 (調査業務) では Codex は「141 行で要件充足」という簡素寄りの立ち位置を維持したが、同時に「一次情報直取得」「必要十分な構造」という構造化特性も維持した。**簡素性 ≠ 非構造化**であることを示している。

### 8.3 特性逆転の本質

cmd_463 と cmd_466 の観察を統合すると、特性逆転の本質は以下のように整理できる:

1. **Codex は「情報圧縮」に強く、「情報拡張」に弱い**
   - 情報圧縮: 既存コードの関数境界を見つけて分離する (構造化)
   - 情報拡張: 新規概念を命名して抽象化する (概念化)

2. **分析系で「最簡素」に見える理由**: 評価軸・URL 引用・比較表のセル数など「情報拡張」が価値を持つ指標で Codex のスコアが低く見える

3. **実装系・設計系で「最も構造的」に見える理由**: 既存コードや明確な要件から関数境界・環境変数・責務分離を抽出する「情報圧縮」が直接価値を生む

4. **D2 での境界ケース**: 調査系だが、一次情報源 (GitHub API) が明確なため「情報圧縮」戦術 (curl 直取得) が有効に機能

### 8.4 運用ルールへの反映

**新規運用ルール**: **「Codex 評価は単一カテゴリで判断するな」**

- 分析系単体で Codex を評価して「簡素すぎ」と却下すると、実装系・設計系での構造化能力を見逃す
- **Codex の価値は「情報圧縮タスク」で発揮される**: 関数抽出、環境変数化、最小構造設計、要件充足のための必要十分な出力
- ルーティング時は「情報圧縮タスクか情報拡張タスクか」で判断せよ

## 11. Sonnet 運用時の新規注意点

### 11.1 自己報告精度の限界

cmd_466 D1 (d1_sonnet) で以下の自己報告誤差が観察された:

### 11.2 運用ルール追加

**新規ルール**: Sonnet 使用時は成果物の以下を**必ず実測検証**せよ:

1. 行数 (`wc -l`)

4. JSON 構文 (`jq empty` または `python -c "import json; json.load(open(...))"`)
5. bash 構文 (`bash -n`)

Sonnet は実装直結度が高く最小差分主義で優秀だが、**自己検証精度の限界**を認識し、軍師 QC で必ず実測値と突き合わせる必要がある。
