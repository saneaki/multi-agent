## 8. /plan (20点)

### (1) 概要

実装計画を作成し、ユーザーの承認を得てからコード作業を開始するコマンドです。複雑な機能追加やシステム変更を行う前に、設計図を描いて「この方針で進めていいですか？」と確認を取る仕組みです。

### (2) 内部ロジック

1. **planner agentの起動**: 専門の計画担当エージェント（コマンド定義ファイルでは「planner agent」と記載）が動き出します
2. **要件の再定義**: ユーザーの依頼内容を明確な言葉で言い直します
3. **段階分割**: 実装作業をPhase 1、Phase 2...と段階に分けます
4. **依存関係の洗い出し**: どの部品がどの部品に依存するかを整理します
5. **リスク評価**: 起こりうる問題を「HIGH（高）」「MEDIUM（中）」「LOW（低）」で分類します
6. **複雑度の見積もり**: 作業全体の難易度と時間を推定します
7. **承認待ち**: 計画を提示し、ユーザーが「yes」「proceed」などと返すまでコードを一切書きません

### (3) 使い方

チャットで `/plan` と入力すると発動します。その後、何を作りたいかを説明すると、上記の計画が提示されます。

- `/plan` のみ → 計画作成を開始
- 計画が提示されたら `yes` または `proceed` → 実装開始
- `modify: [変更内容]` → 計画の修正を依頼
- `different approach: [別の方法]` → 別のアプローチを提案

### (4) 実践例（弁護士業務での活用）

**シーン**: 法律事務所のWebサイトに「相談予約のリアルタイム通知機能」を追加したい

1. `/plan` と入力し、「相談予約が入ったら即座にSlackとメールで通知したい」と説明
2. planner agentが計画を提示:
   - Phase 1: データベースに通知テーブルを追加（通知の履歴を保存する場所を用意）
   - Phase 2: 通知サービスの作成（メールやSlackに送信する部品を作る）
   - Phase 3: 予約システムとの連携（予約が入ったタイミングで通知を送る仕組み）
   - Phase 4: 管理画面の追加（通知の設定や履歴を見る画面）
   - リスク評価: メール配信の信頼性（HIGH）、Slack API制限（MEDIUM）
   - 見積もり: 合計10〜15時間
3. 計画を確認し、「Phase 4は後回しでいい」などと修正依頼も可能
4. `yes` で承認 → 実装開始

**メリット**: いきなりコードを書いて「思ってたのと違う」となるリスクを防げます。

### (5) Tips/注意点

- **コード着手は承認後のみ**: planner agentは承認なしでコードを書きません。「とりあえず試しに作ってみて」という使い方はできません。
- **複雑な案件で威力を発揮**: 1ファイルだけの小さな修正では不要。複数ファイルにまたがる変更や、設計判断が必要な場合に使いましょう。
- **他コマンドとの連携**: 計画承認後は `/tdd`（テスト駆動開発）でテストを先に書く、実装後は `/code-review`（コード品質チェック）で品質を確認、という流れが推奨されています。
- **修正は何度でも可**: 最初の計画が気に入らなければ、修正依頼を出して練り直せます。

## 9. /pub (20点)

### (1) 概要

ドキュメント（説明書類）の更新、変更のコミット（確定）、リモートリポジトリ（ネット上の保管庫）へのプッシュ（送信）を自動で行うコマンドです。コード変更後の「公開準備」を一気にやってくれます。

### (2) 内部ロジック

**Phase 1: ドキュメント更新チェック**
1. `git diff --name-only` と `git status` でどのファイルが変わったか確認
2. ソースコード（.ts/.tsx/.js/.py/.goなど）が変わっていたら、README.mdやCLAUDE.mdも更新が必要か判断
3. 必要なら README.md や CLAUDE.md を更新
4. **日本語ドキュメントの同期（必須）**: agents/、commands/、rules/、skills/ 配下の `.md` ファイルが変更されていたら、対応する `docs/ja/` 配下の日本語ファイルも同じ構造で更新します
5. **日本語同期の検証（ゲート）**: すべての既存日本語ファイルが更新されたか確認。漏れがあれば Phase 2 に進みません

**Phase 2: 変更内容のレビュー**
6. `git status` で変更・未追跡ファイルを一覧表示
7. `git diff` で変更差分を確認
8. `git log --oneline -5` で最近のコミットメッセージのスタイルを確認
9. 除外対象の特定: セッションデータ（sessions/）、キャッシュファイル（*-cache.json）、OSの一時ファイル（nul, .DS_Store）、機密情報（.env, credentials）を .gitignore に追加
10. セキュリティチェック: ハードコードされたAPIキー（コード内に直接書かれた秘密の文字列）やトークン（認証用の鍵）がないか確認

**Phase 3: コミット＆プッシュ**
11. `git add` で意味のある変更だけをステージング（コミット対象に追加）
12. 変更内容を分析し、適切なコミットタイプ（feat/fix/docs/refactor/chore/perf/test/ci）を選択し、日本語でコミットメッセージを作成
13. `git push` でリモートに送信（新しいブランチなら `-u` フラグ付き）

### (3) 使い方

- `/pub` → 全フロー実行（ドキュメント更新 → コミット → プッシュ）
- `/pub --dry-run` → 変更内容だけ見せて、コミット・プッシュはしない（予行演習）
- `/pub --skip-docs` → ドキュメント更新チェックをスキップ
- `/pub --amend` → 直前のコミットに追加で修正を入れる（コミット履歴を上書き）

### (4) 実践例（弁護士業務での活用）

**シーン**: 事務所Webサイトの「お問い合わせフォーム」のバリデーション（入力チェック）ロジックを修正した

1. コードを修正後、`/pub` と入力
2. Phase 1で README.md が「新しいバリデーション機能」について触れていないことを検出 → 自動で README.md に説明を追加
3. Phase 1で agents/tdd-guide.md が変更されていた場合、docs/ja/agents/tdd-guide.md も同じ構造で更新
4. Phase 2で git diff を確認し、`.env` ファイルが除外対象になっているか確認
5. Phase 3で `fix: お問い合わせフォームのメールアドレス検証を強化` というコミットメッセージを自動生成 → コミット → プッシュ完了

**メリット**: ドキュメント更新忘れや、機密情報の誤コミットを防げます。

### (5) Tips/注意点

- **日本語ドキュメント同期は絶対**: agents/、commands/、rules/、skills/ のファイルを編集したら、対応する docs/ja/ のファイルも必ず更新されます。これが漏れるとPhase 2に進めません。
- **--dry-run で予行演習**: 本番のプッシュ前に `/pub --dry-run` で「何がコミットされるか」を確認する習慣をつけると安全です。
- **機密情報の誤コミット防止**: Phase 2のセキュリティチェックで、コード内にAPIキーやパスワードが直接書かれていないか自動でスキャンします。
- **日本語コミットメッセージ**: このプロジェクトではコミットメッセージを日本語で書くルールになっています（`git-workflow.md` の設定）。

## 10. /verify (18点)

### (1) 概要

プロジェクトの品質を総合的に検証するコマンドです。ビルド（プログラムの組み立て）、型チェック（データ型の正しさ確認）、lint（コードスタイル検査）、テスト実行、console.log監査（デバッグ用出力の検出）、git状態確認を順番に実行し、「PR（プルリクエスト、変更提案）を出していいか？」を判定します。

### (2) 内部ロジック

以下の順番で実行されます（1つでも失敗すると即座に停止）:

1. **ビルドチェック**: プロジェクトのビルドコマンド（例: `npm run build`）を実行。失敗したらエラーを報告して停止。
2. **型チェック**: TypeScriptやその他の型チェッカーを実行し、データ型の不整合を検出。エラーがあればファイル名と行番号を報告。
3. **Lintチェック**: linter（コードスタイル検査ツール、例: ESLint）を実行し、警告とエラーを報告。
4. **テストスイート**: すべてのテストを実行し、「何件中何件成功したか」「カバレッジ（テストがコードの何%をカバーしているか）」を報告。
5. **console.log監査**: ソースファイル内に `console.log`（本番コードに残すべきでないデバッグ出力）がないか検索し、見つかった場所を報告。
6. **Git状態**: 未コミットの変更や、最後のコミット以降に変わったファイルを表示。

最後に以下の形式でレポートを出力:

```
VERIFICATION: [PASS/FAIL]

Build:    [OK/FAIL]
Types:    [OK/X errors]
Lint:     [OK/X issues]
Tests:    [X/Y passed, Z% coverage]
Secrets:  [OK/X found]
Logs:     [OK/X console.logs]

Ready for PR: [YES/NO]
```

### (3) 使い方

- `/verify` または `/verify full` → 全チェック実行（デフォルト）
- `/verify quick` → ビルド＋型チェックのみ（高速）
- `/verify pre-commit` → コミット前に必要なチェック
- `/verify pre-pr` → PR作成前の全チェック＋セキュリティスキャン

### (4) 実践例（弁護士業務での活用）

**シーン**: 法律事務所のWebサイトに「顧問契約管理機能」を追加し、GitHub上でレビュー依頼（PR）を出す前

1. `/verify` と入力
2. 実行結果:
   ```
   VERIFICATION: FAIL

   Build:    OK
   Types:    2 errors
             - src/contracts/types.ts:15 - Property 'endDate' does not exist
             - src/api/contracts.ts:42 - Type 'string' is not assignable to type 'Date'
   Lint:     OK
   Tests:    45/47 passed, 78% coverage
   Secrets:  OK
   Logs:     3 console.logs
             - src/contracts/ContractForm.tsx:120
             - src/contracts/ContractForm.tsx:145
             - src/api/contracts.ts:88

   Ready for PR: NO
   ```
3. 型エラー2件と `console.log` 3件が検出されたので、これらを修正
4. 再度 `/verify` → すべてPASS → PRを作成

**メリット**: PR作成前に品質問題を検出でき、レビュアー（コードを確認する人）に迷惑をかけません。

### (5) Tips/注意点

- **ビルド失敗で即停止**: ビルドが通らないとそこで検証が止まります。ビルドエラーを最優先で直しましょう。
- **console.logは本番NGルール**: このプロジェクトでは `console.log` を本番コードに残さないルールがあります（`coding-style.md` 参照）。デバッグ時に使ったら、コミット前に削除しましょう。
- **quick vs full**: 開発中の小刻みなチェックは `/verify quick`、PR前の最終確認は `/verify full` と使い分けると効率的です。
- **カバレッジ80%ルール**: このプロジェクトではテストカバレッジ80%以上が必須です（`testing.md` 参照）。これを下回ると「Ready for PR: NO」になります。

## 11. /code-review (18点)

### (1) 概要

未コミットの変更内容を、セキュリティと品質の観点から包括的にレビューするコマンドです。コミット前に「危険なコードがないか」「品質基準を満たしているか」を自動チェックし、重大な問題があればコミットをブロックします。

### (2) 内部ロジック

1. **変更ファイルの取得**: `git diff --name-only HEAD` で変更されたファイルのリストを取得
2. **各ファイルをチェック**: 以下の3つの観点で検査

   **セキュリティ問題（CRITICAL: 最重要）**:
   - ハードコードされた認証情報（APIキー、トークン、パスワード）
   - SQLインジェクション脆弱性（データベースへの不正な命令を許す欠陥）
   - XSS脆弱性（クロスサイトスクリプティング、悪意あるスクリプト実行を許す欠陥）
   - 入力検証の欠如（ユーザーが送った値をチェックせず使う危険性）
   - 安全でない依存関係（既知の脆弱性を持つライブラリ）
   - パストラバーサルリスク（ファイルシステム上の意図しない場所へアクセスされる危険性）

   **コード品質（HIGH: 高優先度）**:
   - 50行を超える関数（長すぎて理解しづらい）
   - 800行を超えるファイル（大きすぎて保守が困難）
   - ネスト（入れ子）が4階層より深い（複雑で読みにくい）
   - エラーハンドリングの欠如（エラー発生時の処理が書かれていない）
   - `console.log` 文（本番コードに残すべきでない）
   - TODO/FIXMEコメント（未解決の課題が残っている）
   - 公開API（外部から使う関数）のJSDoc欠如（説明が書かれていない）

   **ベストプラクティス（MEDIUM: 中優先度）**:
   - ミューテーション（既存データの直接書き換え、イミュータブル＝書き換えない方式を推奨）
   - 絵文字の使用（コードやコメント内での絵文字は非推奨）
   - 新規コードのテスト欠如（テストが書かれていない）
   - アクセシビリティ問題（視覚障害者などが使いづらい画面設計）

3. **レポート生成**:
   - 各問題に対して「重大度（Severity）」「ファイル名と行番号」「問題の説明」「修正案」を記載
   - CRITICAL（最重要）またはHIGH（高優先度）問題があれば、コミットをブロック

### (3) 使い方

チャットで `/code-review` と入力すると、現在の未コミット変更がレビューされます。引数やオプションはありません。

### (4) 実践例（弁護士業務での活用）

**シーン**: 法律事務所の「案件管理システム」に顧客情報の検索機能を追加した

1. コード編集後、コミット前に `/code-review` と入力
2. レポートが出力:
   ```
   CODE REVIEW REPORT

   [CRITICAL] src/api/clients.ts:45
   - SQL injection vulnerability
   - Issue: User input directly interpolated in SQL query
   - Fix: Use parameterized queries: db.query('SELECT * FROM clients WHERE name = ?', [name])

   [HIGH] src/api/clients.ts:82
   - Missing error handling
   - Issue: Async function has no try-catch block
   - Fix: Wrap in try-catch and return user-friendly error message

   [HIGH] src/components/ClientSearch.tsx:120
   - Function exceeds 50 lines (actual: 78 lines)
   - Fix: Extract filter logic into separate helper function

   [MEDIUM] src/components/ClientSearch.tsx:45
   - Mutation pattern detected
   - Issue: clients.push(newClient) mutates array
   - Fix: Use immutable pattern: [...clients, newClient]

   BLOCKED: Cannot commit with CRITICAL or HIGH issues.
   ```
3. 特にCRITICALのSQLインジェクション脆弱性は危険なので最優先で修正
4. すべての問題を修正後、再度 `/code-review` → PASS → コミット可能

**メリット**: 顧客データを扱う法律事務所では、セキュリティ脆弱性は致命的です。このコマンドで事前に防げます。

### (5) Tips/注意点

- **セキュリティ最優先**: CRITICAL問題は絶対に修正してからコミットしましょう。特にハードコードされたAPIキーやSQLインジェクションは、そのまま公開すると情報漏洩につながります。
- **コミット前の習慣化**: プロジェクトの `agents.md` では「コード変更後は必ず code-reviewer agent を使う」とされています。手動でやるより `/code-review` コマンドの方が簡単です。
- **自動ブロック機能**: CRITICAL/HIGH問題があるとコミットができません。「とりあえずコミットして後で直そう」という誘惑を防げます。
- **ベストプラクティス違反も検出**: ミューテーション（配列の直接書き換え等）や、絵文字の使用なども検出します。コードスタイルの統一に役立ちます。

## 12. /revise-claude-md (18点)

### (1) 概要

現在のセッション（対話のやりとり）から学んだことを振り返り、CLAUDE.md（プロジェクト固有の指示書）に追記するコマンドです。「今回使ったコマンド」や「遭遇した問題の解決策」を残しておくことで、次回以降のClaude Codeセッションがより効率的に動けるようになります。

### (2) 内部ロジック

**Step 1: 振り返り**
- 今回のセッションで「どんなコンテキスト（背景情報）があれば、Claudeがもっと効率的に動けたか？」を考えます
- 具体的には:
  - 使用したBashコマンド（シェル操作）
  - 従ったコードスタイルのパターン
  - うまくいったテスト方法
  - 環境や設定の癖（特殊な動作）
  - 遭遇した警告や注意点

**Step 2: CLAUDE.mdファイルを探す**
- `find . -name "CLAUDE.md" -o -name ".claude.local.md"` でファイルを検索
- 追記先を判断:
  - `CLAUDE.md`: チーム全体で共有（Gitにコミットされる）
  - `.claude.local.md`: 個人用（Gitには含まれない）

**Step 3: 追記内容を起草**
- 簡潔に、1行1概念で記述
- フォーマット: `<コマンドまたはパターン>` - `<簡潔な説明>`
- 避けるべきもの:
  - 冗長な説明
  - 自明な情報（誰でも知っていること）
  - 一度きりの修正（再発しなさそうなもの）

**Step 4: 提案内容を表示**
```
### Update: ./CLAUDE.md

**Why:** [1行で理由]

\`\`\`diff
+ [追加内容 - 簡潔に]
\`\`\`
```

**Step 5: 承認後に適用**
- ユーザーが承認したファイルのみ編集
- 承認がなければ何もしない

### (3) 使い方

チャットで `/revise-claude-md` と入力すると、現在のセッションから学んだことを CLAUDE.md に反映するプロセスが始まります。引数やオプションはありません。

### (4) 実践例（弁護士業務での活用）

**シーン**: 法律事務所のn8n自動化プロジェクトで、「Google DriveのOCR処理が日本語PDFでうまくいかない」問題を解決した

1. 問題解決後、`/revise-claude-md` と入力
2. Claudeが振り返り:
   - 「Google Drive APIのOCR言語設定で `ja` を明示的に指定する必要があった」
   - 「n8nのHTTP Requestノードで `ocrLanguage=ja` パラメータを追加」
   - 「この設定がないと英語OCRとして処理され、日本語が文字化けする」
3. 提案内容が表示:
   ```
   ### Update: ./CLAUDE.md

   **Why:** Google Drive OCRで日本語PDFを正しく処理するために必要

   \`\`\`diff
   + ## Google Drive API
   + - `ocrLanguage=ja` - OCRで日本語を認識させる場合は必須（デフォルトは英語）
   \`\`\`
   ```
4. 承認すると CLAUDE.md に追記される
5. 次回のセッションで、Claudeは最初から「日本語OCRには `ocrLanguage=ja` が必要」という知識を持って動けるようになる

**メリット**: プロジェクト固有の「癖」や「解決済みの問題」を蓄積でき、同じ問題に何度もぶつからなくて済みます。

### (5) Tips/注意点

- **簡潔さが重要**: CLAUDE.mdはClaude Codeのプロンプト（指示）の一部として読み込まれます。長すぎるとコンテキスト（文脈理解の容量）を圧迫するので、1行で済む説明を心がけましょう。
- **チーム共有 vs 個人用**: チーム全員に役立つ情報なら `CLAUDE.md`、自分の環境だけの設定なら `.claude.local.md` に書きましょう。
- **許可されたツールのみ**: このコマンドでは Read, Edit, Glob ツールのみ使えます（コマンド定義の `allowed-tools` で制限）。
- **承認制**: 勝手にファイルを書き換えません。必ず提案内容を見せて、ユーザーが「OK」と言ってから編集します。
- **一度きりの問題は書かない**: 再発しなさそうな問題（例: 特定のライブラリのバージョン1.2.3でのみ起きたバグ）は、書いても将来役立たないので省略しましょう。
