# Sランク スキル・コマンド 詳細マニュアル

**作成日**: 2026年2月13日
**対象読者**: 弁護士（法律事務所経営者、技術者ではない方）
**目的**: Claude Codeの最高評価ツール20件について、仕組み・使い方・実践例を解説するマニュアル

---

## このマニュアルについて

Claude Codeには多数のスキル（自動で動く補助機能）とコマンド（手動で呼び出す機能）が搭載されています。本マニュアルでは、実用度評価でSランク（16点以上/20点満点）を獲得した上位20件について、以下の5項目で詳しく解説します。

- **(1) 概要**: 何をするツールか
- **(2) 内部ロジック**: どういう仕組みで動くか
- **(3) 使い方**: 具体的な呼び出し方
- **(4) 実践例**: 弁護士業務での具体的ユースケース
- **(5) Tips/注意点**: 効果的に使うためのコツ

**カタログとの違い**: 以前のレポート（cmd_132）は「一覧＋スコアリング」でしたが、本マニュアルは各ツールの**内部動作の解説**と**実務での活用方法**に踏み込んだ「マニュアル」です。

## カテゴリA: スキル（自動で動く補助機能）

スキルは、特定の条件を満たしたときに自動で起動する補助機能です。利用者が明示的に呼び出す必要がない場合もあります。

### 1. strategic-compact（戦略的コンテキスト圧縮）— 19点

#### (1) 概要

AIが一度に処理できる情報量（コンテキストウィンドウ）には上限があります。このスキルは、作業の区切りの良いタイミングで「今までの会話を要約して整理する」ことを提案するツールです。適切なタイミングで圧縮することで、重要な情報を失わずに長時間の作業を続けられます。

#### (2) 内部ロジック

1. **ツール呼び出しを監視**: Claude Codeが「ファイル編集」や「ファイル作成」を行うたびに、内部でカウンターが増えます。
2. **閾値判定**: 50回のツール使用に達すると、「そろそろ /compact（圧縮コマンド）を実行しませんか？」と提案します。
3. **定期リマインダー**: その後も25回ごとに再度提案を行います。
4. **実装方式**: `suggest-compact.sh` というシェルスクリプトがPreToolUse Hook（ツール使用前に動く仕組み）として動作します。

このスキルは自動で圧縮するのではなく、「提案だけ」を行います。最終判断は利用者に委ねられます。

#### (3) 使い方

このスキルは設定済みであれば自動で提案が表示されます。利用者が行うのは以下の操作だけです:

- **提案が表示されたとき**: `/compact` とコマンドを入力すると、会話履歴が要約され、コンテキストがリフレッシュされます。
- **無視したいとき**: 何もせず作業を続ければOKです。次の25回後に再度提案されます。

手動でいつでも `/compact` を実行することもできます。

#### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース**: 複雑な訴訟案件で準備書面を作成中。契約書の分析、判例検索、証拠整理、書面起案を1セッションで続けている。

1. **契約書分析**: Claude Codeに契約書を読み込ませ、争点を整理（20回のツール呼び出し）
2. **判例検索**: 関連判例をWebで検索し、要旨を整理（25回のツール呼び出し）
3. **証拠整理**: 証拠書類のファイル名をリネームし、証拠説明書を作成（30回のツール呼び出し）
4. **ここで提案が表示**: 「/compact を実行しませんか？」
5. **圧縮を実行**: `/compact` を入力。契約書の詳細や判例の全文は要約され、争点と証拠リストだけが残る。
6. **書面起案**: 軽くなったコンテキストで準備書面を起案開始。新しい作業に集中できる。

**効果**: 長時間の作業で会話が肥大化しても、区切りの良いタイミングで整理することで、AIが「何が重要か」を見失わずに済みます。

#### (5) Tips/注意点

- **圧縮のタイミング**: 「調査フェーズが終わって、実行フェーズに入る直前」がベスト。調査結果の詳細は要約し、これからやることだけを残す。
- **圧縮しすぎ注意**: 作業の途中で圧縮すると、細かい指示やメモが消えてしまう可能性があります。大きな区切りまで待ちましょう。
- **手動でもOK**: 提案を待たずに、自分で「今がタイミングだ」と思ったら `/compact` を実行してもOKです。
- **代替手段**: 圧縮の代わりに「メモリに保存」する手もあります。重要な情報は `/remember` コマンドでセッションを超えて保存できます。

### 2. claude-md-improver（CLAUDE.md改善ツール）— 19点

#### (1) 概要

CLAUDE.mdとは、プロジェクトやフォルダごとに置く「Claude Code向けの説明書」です。このファイルに書かれた内容は、Claude Codeが毎回自動で読み込みます。claude-md-improverは、このCLAUDE.mdの品質を評価し、不足している情報を追加するツールです。

#### (2) 内部ロジック

1. **発見フェーズ**: プロジェクト内の全CLAUDE.mdファイルを検索します。
2. **品質評価フェーズ**: 各ファイルを6つの基準で採点します:
   - コマンド/ワークフローの記載（20点）
   - アーキテクチャの明確さ（20点）
   - 非自明なパターンの記載（15点）
   - 簡潔さ（15点）
   - 最新性（15点）
   - 実行可能性（15点）
3. **レポート出力**: 各ファイルの評価をA〜F判定で表示します。
4. **改善提案**: 「このセクションを追加すべき」という具体的な提案を出します。
5. **承認後に更新**: 利用者が承認したら、CLAUDE.mdを更新します。

このスキルは「読み取り専用」ではなく、承認後に実際にファイルを**書き換える**点が特徴です。

#### (3) 使い方

**手動起動**: 「CLAUDE.mdをチェックして」「CLAUDE.mdを改善して」と依頼すると、このスキルが起動します。

**基本的な流れ**:
1. 「CLAUDE.mdをチェックして」と指示
2. 品質レポートが表示される
3. 改善提案が提示される
4. 「この提案で更新してください」と承認
5. CLAUDE.mdが更新される

#### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース**: 法律事務所で顧問先企業向けの契約書レビューツール（スクリプト）を開発中。

**状況**: プロジェクトフォルダに契約書分析スクリプト（Python）がいくつかあるが、CLAUDE.mdは古いまま。新しいメンバーがClaude Codeを使って修正しようとしても、どのスクリプトが何をするのか分からない。

**対処**:
1. 「CLAUDE.mdをチェックして」と依頼
2. **評価結果**: D判定（30点）。理由: 実行コマンド未記載、各スクリプトの役割未記載、契約書の種類ごとの処理パターン未記載
3. **改善提案が表示**: 「実行方法」「スクリプト構成」「注意点」セクションの追加が提案される
4. **承認して更新**: 提案を承認すると、CLAUDE.mdが自動で更新される
5. **効果**: 次回からClaude Codeを起動すると、自動で更新されたCLAUDE.mdが読み込まれ、各スクリプトの使い方が分かる

**ポイント**: CLAUDE.mdは「人間向けのREADME」ではなく「AI向けの取扱説明書」です。人間が読んで分かることでも、AIに明示的に書いておかないと見落とされることがあります。

#### (5) Tips/注意点

- **定期メンテナンスが重要**: プロジェクトが進化するとCLAUDE.mdは古くなります。月1回くらいでこのスキルを実行して鮮度を保ちましょう。
- **簡潔さを保つ**: 詳細すぎるCLAUDE.mdは読み込みに時間がかかり、コンテキストウィンドウを圧迫します。「本当に必要な情報だけ」を残す方針で。
- **個人設定は別ファイルで**: チーム共有のCLAUDE.mdには個人の好みを書かず、`.claude.local.md` に分けましょう。
- **ショートカット**: セッション中にClaude Codeが学んだことを即座にCLAUDE.mdに追加したいときは `#` キーを押すと、自動で追記してくれます。

### 3. pull-merge-pub（Pull→Merge→Push統合ワークフロー）— 18点

#### (1) 概要

ソースコード管理ツール（Git）で、リモート（GitHub等）の最新版を取得（pull）し、ローカルの変更とマージ（merge）し、最終的にリモートに送信（push）するまでの一連の流れを安全に実行するスキルです。コンフリクト（衝突）が起きたときは自動で止まり、利用者に判断を求めます。

#### (2) 内部ロジック

**フェーズ1: Pull準備**
1. 現在のブランチ名とリモート設定を確認
2. ローカルに未コミットの変更があるかチェック
3. 変更がある場合は「コミットするか、stash（一時退避）するか」を判断
4. `git pull origin <branch>` を実行

**フェーズ2: コンフリクト判定**
1. pull結果を解析: 変更なし/自動マージ成功/コンフリクト発生
2. コンフリクトが軽微（設定ファイルの値だけ等）→ 自動解決
3. コンフリクトが深刻（ソースコードのロジック部分等）→ 停止して利用者に確認

**フェーズ3: /pub実行**
1. ドキュメントの翻訳同期チェック
2. `.gitignore` の整理
3. シークレット（APIキー等）のスキャン
4. コミット（変更を記録）
5. プッシュ（リモートに送信）

#### (3) 使い方

- 「リモートの最新版をpullしてマージして、pubして」と指示
- または「pull-merge-pubワークフローを実行して」と指示

**典型的な流れ**:
1. ローカルで作業した内容がある状態で上記の指示を出す
2. Claude Codeがローカルの変更を確認し、必要に応じてコミットを促す
3. pullが実行される
4. コンフリクトがなければ自動で進み、/pubまで完了
5. コンフリクトがあれば停止し、「どちらを優先しますか？」と聞かれる

#### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース**: 事務所の複数の弁護士が同じ判例データベース管理スクリプトを共同で更新している。

**状況**: あなたはローカルで「最高裁判例の検索機能」を追加。同僚は同じ期間に「下級審判例のフォーマット修正」をリモートにプッシュ済み。

**操作**:
1. 「リモートの最新版をpullしてマージして、pubして」と指示
2. **Pull準備**: ローカルに未コミットの変更を検出
3. **コミット確認**: 「先にコミットしますか？」→「はい」と応答
4. **Pull実行**: リモートから同僚の変更を取得
5. **コンフリクト判定**: 別ファイル → 自動マージ成功。同じ箇所 → 停止して確認
6. **/pub実行**: マージ完了後、自動でドキュメント同期・コミット・プッシュが実行される

**効果**: 手動で `git pull`, `git merge`, `git commit`, `git push` を順番に実行する手間が省け、コンフリクトが起きても安全に対処できます。

#### (5) Tips/注意点

- **pull前に必ずコミット**: ローカルの未保存の変更がある状態でpullすると、変更が上書きされるリスクがあります。このスキルはそれを防ぎます。
- **コンフリクトは慎重に**: 深刻なコンフリクトは自動解決せず、必ず利用者に確認します。
- **force pushは禁止**: このスキルは `git push --force` を使いません。
- **定期的なpull習慣**: チーム開発では、朝一番にpullする習慣をつけると、大きなコンフリクトを避けられます。

### 4. security-review（セキュリティレビュー）— 17点

#### (1) 概要

セキュリティ上の脆弱性（セキュリティホール）をチェックするツールです。認証（ログイン処理）、ユーザー入力の扱い、APIキーの管理など、10項目の観点で「危険なコードがないか」を自動で検査します。

#### (2) 内部ロジック

このスキルは10項目のチェックリストを持っています:

1. **シークレット管理**: APIキーやパスワードがソースコードに直接書かれていないか
2. **入力バリデーション**: ユーザーが入力したデータを信頼せず、必ず検証しているか
3. **SQLインジェクション対策**: データベースクエリがパラメータ化されているか
4. **認証・認可**: ログイン処理とアクセス権限のチェックが正しいか
5. **XSS対策**: ユーザーが入力したHTMLをそのまま表示していないか
6. **CSRF対策**: なりすまし攻撃への対策があるか
7. **レート制限**: 大量のリクエストに耐えられるか
8. **機密データの露出**: エラーメッセージにパスワード等が含まれていないか
9. **ブロックチェーン関連**: ウォレット署名検証が正しいか（該当する場合のみ）
10. **依存パッケージ**: 古いライブラリに脆弱性がないか

問題を発見したら「CRITICAL（緊急）」「HIGH（高）」「MEDIUM（中）」でランク付けし、修正案を提示します。

#### (3) 使い方

**自動起動**: 認証機能、API連携、ユーザー入力処理を書いたとき、このスキルが自動で起動します。

**手動起動**: 「セキュリティレビューして」と指示するだけです。

**レビュー後の流れ**:
1. 問題リストが表示される
2. CRITICALとHIGHは**必ず修正**
3. MEDIUMは可能な限り修正
4. 修正後、再度レビューを実行して問題が消えたことを確認

#### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース1: 顧問先スタートアップのWebアプリ開発支援**

1. 開発中のソースコードをClaude Codeで開く
2. 「セキュリティレビューして」と指示
3. **検出された問題**:
   - **CRITICAL**: パスワードが平文でデータベースに保存されている
   - **HIGH**: ログイン失敗時にメアド存在確認の情報漏洩
   - **MEDIUM**: レート制限がなく、総当たり攻撃に脆弱
4. **修正指示**: Claude Codeに「CRITICALとHIGHを修正して」と指示
5. **再レビュー**: 問題なしと確認

**ケース2: 事務所内の案件管理ツール**

1. 「セキュリティレビューして」と指示
2. **CRITICAL**: データベース接続パスワードがソースコードに直接書かれている
3. **修正**: 環境変数に移動し、`.gitignore` に追加
4. **効果**: GitHubに誤ってプッシュしても、パスワードは漏れない

#### (5) Tips/注意点

- **CRITICALは即修正**: 情報漏洩や不正アクセスのリスクがあります。
- **デプロイ前に必ず実行**: 本番環境にリリースする前に、必ずセキュリティレビューを通してください。
- **過信は禁物**: このスキルは完璧ではありません。重要なシステムでは専門のセキュリティ診断サービスの利用も検討してください。

### 5. legal-document-namer（法律文書命名規則）— 17点

#### (1) 概要

法律事務所の事件フォルダで使うファイル名を、フォルダ種別ごとのルールに従って自動生成するツールです。「02 協議」「03 調停・審判」「04 訴訟」などのフォルダごとに異なる命名パターンがあり、それを手動で覚える必要がなくなります。証拠番号（甲01, 乙02等）の連番生成機能もあります。

#### (2) 内部ロジック

このスキルは3つのモジュールで構成されています:

**1. LegalTemplatePreset（テンプレート管理）**:
- フォルダ種別（02協議, 03調停審判, 04訴訟, 05送付書類, 07手持ち資料, 08メール）ごとに命名パターンを保持
- 例: `{提出日}_{受信者}へ_{文書名}` というテンプレートに対し、情報を渡すと `20260209_先方代理人へ_和解案` を生成

**2. DateHelper（日付処理と枝番）**:
- 今日の日付をYYYYMMDD形式で取得
- 日付のバリデーション（不正な日付を検出）
- 同名ファイルがあるときの枝番自動付与（`_2`, `_3`）

**3. EvidenceNumberer（証拠番号生成）**:
- 当事者（甲/乙）と開始番号・生成数を指定すると、連番リストを生成
- 例: 甲01〜甲05の連番を自動生成
- 証拠ファイル名も生成: `20260209_甲01_契約書`

#### (3) 使い方

**基本的な使い方**:
1. 「02協議フォルダの発信文書ファイル名を生成して」と指示
2. Claude Codeが「提出日は？」「受信者は？」「文書名は？」と順番に聞いてくる
3. 回答すると、命名ルールに従ったファイル名が生成される

**証拠番号の使い方**:
- 「甲号証を5件分、01から生成して」→ `甲01〜甲05` が生成される
- 「甲05の証拠ファイル名を、契約書という名前で作って」→ `20260209_甲05_契約書` が生成される

**一括リネーム**:
- 複数ファイルに対して「このフォルダのファイルを全部、04訴訟の書証ルールでリネームして」と指示すると、一括で処理されます。

#### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース: 民事訴訟で提出する証拠書類の整理**

**手動での作業**（従来）:
1. 提出日をYYYYMMDD形式で手入力
2. 証拠番号を手動で割り当て
3. ファイル名を1件ずつ変更
4. 途中で同じ番号を2回使ってしまうミスが発生
5. 50件分で1時間以上かかる

**このスキルを使った場合**:
1. 「04訴訟フォルダの書証ファイル名を、甲01から50件分生成して」と指示
2. Claude Codeが証拠リストを生成
3. 「このリストを使って、フォルダ内のファイルをリネームして」と指示
4. 自動でリネーム実行: `契約書.pdf` → `20260209_甲01_契約書.pdf`
5. 所要時間: 5分

**効果**: 時間が大幅短縮（1時間 → 5分）、命名ミスがゼロになります。

#### (5) Tips/注意点

- **日付は必ずYYYYMMDD**: `20260209` 形式のみ対応。`2026-02-09` はエラーになります。
- **「相手方」は使用禁止**: 命名ルールでは「先方本人」「先方代理人」を使います。
- **裁判所名は支部まで**: 「東京家裁」ではなく「東京家裁立川支部」のように明記してください。
- **枝番は自動**: 同じ日に同じ名前のファイルを作ると、自動で `_2`, `_3` が付きます。
- **証拠番号は2桁ゼロ埋め**: 甲1ではなく甲01です。ファイル名の並び順が正しくなります。

### 6. n8n-expression-syntax（n8n式構文）— 16点

#### (1) 概要

n8n（ノーコード自動化ツール）で、動的なデータを扱うための「式」の書き方を検証・修正するスキルです。n8nでは `{{$json.email}}` のような形式でデータを参照しますが、記法を間違えると動きません。このスキルは「よくある間違い」を検出し、正しい書き方を提案します。

#### (2) 内部ロジック

**1. 基本ルールの検証**:
- 式は必ず `{{` と `}}` で囲まれているか
- ノード名は引用符で囲まれているか
- フィールド名にスペースがあるときは角括弧記法を使っているか

**2. よくある間違いのパターン検出**:
- カッコ忘れ: `$json.email` → `{{$json.email}}`
- Webhookの`.body`忘れ: `{{$json.name}}` → `{{$json.body.name}}`
- Codeノードでの不要なカッコ: `'={{$json.email}}'` → `$json.email`
- 引用符忘れ: `{{$node.HTTP Request}}` → `{{$node["HTTP Request"]}}`

**3. 修正案の提示**: 問題箇所を指摘し、正しい構文を提示します。

#### (3) 使い方

**自動起動**: n8nのワークフロー設定ファイルを編集すると、このスキルが自動で構文チェックを行います。

**手動起動**: 「このn8n式を検証して」とエラーが出ている式を貼り付けると、どこが間違っているかを教えてくれます。

#### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース: 顧問先企業の問い合わせフォームをn8nで自動化**

1. Slackノードのメッセージ欄に `新しい問い合わせ: {{$json.name}}` と書いた
2. テスト実行すると、`name`フィールドが空白で表示される
3. **原因**: Webhookノードのデータは `.body` 配下にあるのに、`{{$json.name}}` と書いている
4. **修正**: `{{$json.body.name}}` に修正
5. Slackに正しく通知が届くようになる

#### (5) Tips/注意点

- **Webhookは必ず `.body`**: n8nのWebhookノードは、ユーザーデータを `.body` の下に格納します。
- **Codeノードでは `{{}}` 不要**: JavaScriptコードを書くCodeノードでは、直接書きます。
- **ノード名は大文字小文字を区別**: 正確にコピーしてください。
- **式エディタでテスト**: n8nの各フィールドには「式エディタ」があり、リアルタイムで結果がプレビューされます。

### 7. iterative-retrieval（段階的コンテキスト取得）— 16点

#### (1) 概要

大規模なプロジェクトで、「どのファイルが関連しているか分からない」状況で、段階的に関連ファイルを絞り込んでいくスキルです。最初に広く浅くファイルを探し、関連度の高いものだけを深掘りし、最大3回のサイクルで最適なコンテキストを構築します。

#### (2) 内部ロジック

4つのフェーズを最大3回繰り返します:

**フェーズ1: DISPATCH（初回検索）**: 大まかなキーワードでファイルを広範囲に検索

**フェーズ2: EVALUATE（関連度評価）**: 各ファイルに関連度スコア（0〜1）を付ける
- 0.8〜1.0（高）: 目的の機能を直接実装している
- 0.5〜0.7（中）: 関連パターンや型定義を含む
- 0.2〜0.4（低）: 間接的に関連
- 0〜0.2（なし）: 無関係、除外

**フェーズ3: REFINE（検索条件の精緻化）**: 高関連度ファイルのキーワードを使って次の検索を改善

**フェーズ4: LOOP（繰り返し判定）**: 高関連度ファイルが3件以上あり、重要な欠落がなければ終了

#### (3) 使い方

このスキルは**自動で動作**します。利用者が特別に呼び出す必要はありません。大規模プロジェクトで「この機能を実装して」と指示したとき、Claude Codeが内部で使用します。

#### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース: 契約書管理システムのバグ修正**

**状況**: 100個以上のファイルで構成されたシステムで、「トークンの有効期限が切れてもログアウトされない」バグが発生。どのファイルを修正すべきか不明。

**サイクル1**: キーワード "token", "auth", "expiry" で検索 → `auth.ts`, `session.ts` 等を発見
**サイクル2**: "middleware", "jwt", "verify" で追加検索 → `auth-middleware.ts` を発見（ここにバグがあった）

**結果**: 100ファイルのプロジェクトでも、関連する4ファイルだけを10秒で特定。手動で探す時間（30分〜1時間）が大幅に短縮されます。

#### (5) Tips/注意点

- **大規模プロジェクト向け**: 10ファイル未満では効果が薄いです。100ファイル以上で真価を発揮します。
- **ファイルの命名規則に依存**: ファイル名が機能を表していない場合、精度が下がります。
- **3サイクルで見つからない場合**: 手動でヒントを与えましょう。

### 8. /plan（実装計画の作成と承認）— 20点

#### (1) 概要

実装計画を作成し、ユーザーの承認を得てからコード作業を開始するコマンドです。複雑な機能追加やシステム変更を行う前に、設計図を描いて「この方針で進めていいですか？」と確認を取る仕組みです。

#### (2) 内部ロジック

1. **planner agentの起動**: 専門の計画担当エージェントが動き出します
2. **要件の再定義**: ユーザーの依頼内容を明確な言葉で言い直します
3. **段階分割**: 実装作業をPhase 1、Phase 2...と段階に分けます
4. **依存関係の洗い出し**: どの部品がどの部品に依存するかを整理します
5. **リスク評価**: 起こりうる問題をHIGH/MEDIUM/LOWで分類します
6. **複雑度の見積もり**: 作業全体の難易度と時間を推定します
7. **承認待ち**: 計画を提示し、ユーザーが「yes」「proceed」と返すまでコードを一切書きません

#### (3) 使い方

チャットで `/plan` と入力すると発動します。

- `/plan` のみ → 計画作成を開始
- 計画提示後 `yes` または `proceed` → 実装開始
- `modify: [変更内容]` → 計画の修正を依頼
- `different approach: [別の方法]` → 別のアプローチを提案

#### (4) 実践例（弁護士業務での活用）

**シーン**: 法律事務所のWebサイトに「相談予約のリアルタイム通知機能」を追加したい

1. `/plan` と入力し、「相談予約が入ったら即座にSlackとメールで通知したい」と説明
2. planner agentが計画を提示:
   - Phase 1: データベースに通知テーブルを追加
   - Phase 2: 通知サービスの作成
   - Phase 3: 予約システムとの連携
   - Phase 4: 管理画面の追加
   - リスク評価: メール配信の信頼性（HIGH）、Slack API制限（MEDIUM）
3. 計画を確認し、「Phase 4は後回しでいい」などと修正依頼も可能
4. `yes` で承認 → 実装開始

**メリット**: いきなりコードを書いて「思ってたのと違う」となるリスクを防げます。

#### (5) Tips/注意点

- **コード着手は承認後のみ**: planner agentは承認なしでコードを書きません。
- **複雑な案件で威力を発揮**: 1ファイルだけの小さな修正では不要です。
- **他コマンドとの連携**: 計画承認後は `/tdd` でテストを先に書き、実装後は `/code-review` で品質確認、という流れが推奨されています。
- **修正は何度でも可**: 最初の計画が気に入らなければ、修正依頼を出して練り直せます。

### 9. /pub（ドキュメント更新→コミット→プッシュ）— 20点

#### (1) 概要

ドキュメントの更新、変更のコミット（確定）、リモートリポジトリへのプッシュ（送信）を自動で行うコマンドです。コード変更後の「公開準備」を一気にやってくれます。

#### (2) 内部ロジック

**Phase 1: ドキュメント更新チェック**
1. どのファイルが変わったか確認
2. ソースコードが変わっていたら、README.mdやCLAUDE.mdも更新が必要か判断
3. 日本語ドキュメントの同期チェック（docs/ja/ 配下のファイル）
4. 同期の検証ゲート: 漏れがあればPhase 2に進みません

**Phase 2: 変更内容のレビュー**
5. 変更・未追跡ファイルを一覧表示
6. 除外対象の特定（セッションデータ、キャッシュ、機密情報）
7. セキュリティチェック（ハードコードされたAPIキーやトークンの検出）

**Phase 3: コミット＆プッシュ**
8. 意味のある変更だけをステージング
9. 適切なコミットタイプを選択し、日本語でコミットメッセージを作成
10. リモートに送信

#### (3) 使い方

- `/pub` → 全フロー実行
- `/pub --dry-run` → 変更内容だけ見せて、コミット・プッシュはしない（予行演習）
- `/pub --skip-docs` → ドキュメント更新チェックをスキップ
- `/pub --amend` → 直前のコミットに追加で修正を入れる

#### (4) 実践例（弁護士業務での活用）

**シーン**: 事務所Webサイトの「お問い合わせフォーム」のバリデーションロジックを修正した

1. コードを修正後、`/pub` と入力
2. Phase 1で README.md が新機能について触れていないことを検出 → 自動で追加
3. Phase 2で `.env` ファイルが除外対象になっているか確認
4. Phase 3で `fix: お問い合わせフォームのメールアドレス検証を強化` というコミットメッセージを自動生成 → コミット → プッシュ完了

**メリット**: ドキュメント更新忘れや、機密情報の誤コミットを防げます。

#### (5) Tips/注意点

- **日本語ドキュメント同期は絶対**: 対象ファイルを編集したら、対応するdocs/ja/のファイルも必ず更新されます。
- **--dry-run で予行演習**: 本番のプッシュ前に確認する習慣をつけると安全です。
- **機密情報の誤コミット防止**: Phase 2でAPIキーやパスワードが直接書かれていないか自動スキャンします。

### 10. /verify（品質の総合検証）— 18点

#### (1) 概要

プロジェクトの品質を総合的に検証するコマンドです。ビルド、型チェック、lint（コードスタイル検査）、テスト実行、console.log監査、git状態確認を順番に実行し、「PR（プルリクエスト）を出していいか？」を判定します。

#### (2) 内部ロジック

以下の順番で実行されます（1つでも失敗すると即座に停止）:

1. **ビルドチェック**: プロジェクトのビルドコマンドを実行
2. **型チェック**: データ型の不整合を検出
3. **Lintチェック**: コードスタイル検査
4. **テストスイート**: すべてのテストを実行、カバレッジを計測
5. **console.log監査**: デバッグ出力が残っていないか検索
6. **Git状態**: 未コミットの変更を表示

最終レポート:
```
VERIFICATION: [PASS/FAIL]
Build:    [OK/FAIL]
Types:    [OK/X errors]
Lint:     [OK/X issues]
Tests:    [X/Y passed, Z% coverage]
Secrets:  [OK/X found]
Logs:     [OK/X console.logs]
Ready for PR: [YES/NO]
```

#### (3) 使い方

- `/verify` → 全チェック実行（デフォルト）
- `/verify quick` → ビルド＋型チェックのみ（高速）
- `/verify pre-commit` → コミット前チェック
- `/verify pre-pr` → PR作成前の全チェック＋セキュリティスキャン

#### (4) 実践例（弁護士業務での活用）

**シーン**: 「顧問契約管理機能」を追加し、レビュー依頼を出す前

1. `/verify` と入力
2. 型エラー2件とconsole.log 3件が検出される
3. これらを修正
4. 再度 `/verify` → すべてPASS → PRを作成

**メリット**: PR作成前に品質問題を検出でき、レビュアーに迷惑をかけません。

#### (5) Tips/注意点

- **ビルド失敗で即停止**: ビルドが通らないとそこで検証が止まります。
- **console.logは本番NGルール**: デバッグ時に使ったら、コミット前に削除しましょう。
- **quick vs full**: 開発中は `/verify quick`、PR前は `/verify full` と使い分けると効率的です。
- **カバレッジ80%ルール**: テストカバレッジ80%以上が必須です。

### 11. /code-review（セキュリティ・品質レビュー）— 18点

#### (1) 概要

未コミットの変更内容を、セキュリティと品質の観点から包括的にレビューするコマンドです。コミット前に「危険なコードがないか」「品質基準を満たしているか」を自動チェックし、重大な問題があればコミットをブロックします。

#### (2) 内部ロジック

変更ファイルを以下の3つの観点で検査します:

**セキュリティ問題（CRITICAL）**: ハードコードされた認証情報、SQLインジェクション、XSS脆弱性、入力検証の欠如、パストラバーサルリスク

**コード品質（HIGH）**: 50行超の関数、800行超のファイル、4階層超のネスト、エラーハンドリング欠如、console.log文

**ベストプラクティス（MEDIUM）**: ミューテーション、テスト欠如、アクセシビリティ問題

CRITICAL/HIGH問題があれば、コミットをブロックします。

#### (3) 使い方

チャットで `/code-review` と入力するだけです。

#### (4) 実践例（弁護士業務での活用）

**シーン**: 「案件管理システム」に顧客情報の検索機能を追加した

1. コミット前に `/code-review` と入力
2. **CRITICAL**: SQLインジェクション脆弱性を検出
3. **HIGH**: エラーハンドリング欠如、関数が長すぎる
4. **MEDIUM**: 配列の直接書き換え（ミューテーション）
5. すべて修正後、再度 `/code-review` → PASS → コミット可能

**メリット**: 顧客データを扱う法律事務所では、セキュリティ脆弱性は致命的です。このコマンドで事前に防げます。

#### (5) Tips/注意点

- **セキュリティ最優先**: CRITICAL問題は絶対に修正してからコミットしましょう。
- **コミット前の習慣化**: 変更後は必ず `/code-review` を実行する習慣をつけましょう。
- **自動ブロック機能**: 「とりあえずコミットして後で直そう」という誘惑を防げます。

### 12. /revise-claude-md（セッション学習のCLAUDE.md反映）— 18点

#### (1) 概要

現在のセッションから学んだことを振り返り、CLAUDE.md（プロジェクト固有の指示書）に追記するコマンドです。「今回使ったコマンド」や「遭遇した問題の解決策」を残しておくことで、次回以降のセッションがより効率的に動けるようになります。

#### (2) 内部ロジック

**Step 1: 振り返り** — セッション中に使用したコマンド、従ったコードスタイル、うまくいったテスト方法、環境の癖を分析

**Step 2: CLAUDE.mdファイルを探す** — `CLAUDE.md`（チーム共有）と `.claude.local.md`（個人用）を検索

**Step 3: 追記内容を起草** — 簡潔に、1行1概念で記述。冗長な説明や一度きりの修正は避ける

**Step 4: 提案内容を表示** — 変更箇所と理由を表示

**Step 5: 承認後に適用** — ユーザーが承認したファイルのみ編集

#### (3) 使い方

チャットで `/revise-claude-md` と入力するだけです。

#### (4) 実践例（弁護士業務での活用）

**シーン**: n8n自動化プロジェクトで、日本語PDFのOCR処理問題を解決した

1. 問題解決後、`/revise-claude-md` と入力
2. Claudeが「Google Drive APIのOCR言語設定で `ja` を明示的に指定する必要があった」を抽出
3. 提案内容が表示される
4. 承認すると CLAUDE.md に追記される
5. 次回のセッションで、Claudeは最初から「日本語OCRには `ocrLanguage=ja` が必要」という知識を持って動ける

**メリット**: プロジェクト固有の「癖」を蓄積でき、同じ問題に何度もぶつからなくて済みます。

#### (5) Tips/注意点

- **簡潔さが重要**: CLAUDE.mdはプロンプトの一部として読み込まれます。長すぎるとコンテキストを圧迫します。
- **チーム共有 vs 個人用**: 全員に役立つ情報は `CLAUDE.md`、個人環境の設定は `.claude.local.md` に。
- **承認制**: 勝手にファイルを書き換えません。必ず提案内容を見せて、OKと言ってから編集します。
- **一度きりの問題は書かない**: 再発しなさそうな問題は省略しましょう。

### 13. /learn（再利用パターンの抽出・保存）— 17点

#### (1) 概要

Claude Codeで作業した内容の中から、将来また使える「知恵」を抽出して保存するツールです。作業中に解決した問題、工夫した方法、発見したコツ等を「スキル」として記録し、次回同じ状況になったときにすぐ使えるようにします。

#### (3) 使い方

チャットで `/learn` と入力するだけです。

#### (4) 実践例（弁護士業務での活用）

**シーン**: 法律文書PDFをOCRで処理するとき、特定の裁判所フォーマットで毎回同じエラーが出る

1. エラーを解決した後、`/learn` を実行
2. Claude Codeが回避策を抽出提案
3. 確認してOKなら保存
4. 次回同じ裁判所の書面を処理するとき、Claude Codeが自動で参照しエラーを回避

#### (5) Tips/注意点

- **些細な修正は対象外**: 単なる誤字脱字は抽出しません。
- **一度きりの問題は対象外**: 再現性のあるパターンのみです。
- **1スキル1パターン**: 複数の問題を混ぜると後で使いにくくなります。

### 14. /orchestrate（複数エージェント連携）— 17点

#### (1) 概要

複雑なタスクを遂行するとき、複数の専門エージェントを順番に呼び出して作業させるツールです。「設計担当→テスト担当→品質確認担当→セキュリティ確認担当」のように、流れ作業を自動化します。

#### (2) 内部ロジック

4種類の定義済みワークフロー:
- **feature（機能追加）**: planner → tdd-guide → code-reviewer → security-reviewer
- **bugfix（バグ修正）**: explorer → tdd-guide → code-reviewer
- **refactor（設計改善）**: architect → code-reviewer → tdd-guide
- **security（セキュリティ重視）**: security-reviewer → code-reviewer → architect

各エージェント間で「引き継ぎ文書」を作成し、成果物を受け渡します。独立した確認作業は並列実行（同時に複数エージェントを動かす）も可能です。

#### (3) 使い方

```
/orchestrate [workflow-type] [タスク説明]
```

例:
- `/orchestrate feature "ユーザー認証機能を追加"`
- `/orchestrate bugfix "ログイン画面で日本語入力が効かない"`
- `/orchestrate security "個人情報保護の監査"`

#### (4) 実践例（弁護士業務での活用）

**シーン**: 契約書テンプレート自動生成システムに「条項カスタマイズ機能」を追加する

```
/orchestrate feature "契約書テンプレートに条項カスタマイズ機能を追加"
```

1. **Planner**: 設計を立てる
2. **TDD Guide**: テストコードを先に書き、実装を追加
3. **Code Reviewer**: コードの可読性・構造をレビュー
4. **Security Reviewer**: 契約書データの漏洩リスク・アクセス権限を確認

**最終レポート**: 変更ファイル、テスト結果、セキュリティ確認結果が統合されて出力されます。

#### (5) Tips/注意点

- **featureワークフローは必ずplannerから開始**: 設計をスキップすると手戻りが発生します。
- **認証・個人情報を扱う場合はsecurity-reviewerを必ず含める**: 弁護士業務では機密情報を扱うため、省略厳禁です。
- **並列実行で時間短縮**: 独立した確認作業は同時実行できます。

### 15. /sessions（セッション履歴管理）— 16点

#### (1) 概要

Claude Codeで過去に行った作業セッションを一覧表示したり、読み込んだり、覚えやすい別名を付けたりするツールです。「あの時どうやって解決したっけ？」を素早く振り返ることができます。

#### (2) 内部ロジック

セッション履歴は `~/.claude/sessions/` にMarkdown形式で保存されています:

1. **list（一覧表示）**: 全セッションを日付順で表示。フィルタや件数制限も可能
2. **load（読み込み）**: セッションIDまたは別名で内容を表示
3. **alias（別名付け）**: 覚えやすい名前を付ける
4. **info（詳細情報）**: 統計情報を表示
5. **aliases（別名一覧）**: 登録済みの別名を全て表示

#### (3) 使い方

```
/sessions list                          # 全セッション表示
/sessions list --limit 10              # 最新10件のみ
/sessions load contract-template       # 別名で読み込み
/sessions alias 2026-02-01 today-work  # 別名を付ける
/sessions info today-work              # 詳細情報表示
```

#### (4) 実践例（弁護士業務での活用）

**シーン**: 過去に類似案件で作成した「契約書雛形改訂」作業を参照したい

1. 作業時に別名を付けておく: `/sessions alias 2026-01-15 contract-template-revision`
2. 後日、類似案件が来たとき: `/sessions load contract-template-revision`
3. 過去の判断を即座に振り返る

**活用例**: 顧問先ごとの分類、ツール設定手順の記録、月次定型業務の参照

#### (5) Tips/注意点

- **セッションIDは短縮可能**: 最初の4〜8文字だけで識別できます。
- **よく参照するセッションには必ず別名を付ける**: 長いIDを毎回入力するのは非効率です。
- **意味が分かる名前に**: `session1` ではなく `contract-review-202601` のように。

### 16. /checkpoint（作業の節目記録）— 16点

#### (1) 概要

プロジェクトの途中経過を「チェックポイント」として記録し、後で「あの時と比べてどう変わったか」を検証できるツールです。

#### (2) 内部ロジック

**チェックポイント作成（create）**:
1. 現在の状態を検証（ビルドエラーやテスト失敗がないか確認）
2. Gitでスナップショットを作成
3. ログに記録

**チェックポイント検証（verify）**:
1. 指定したチェックポイントを読み込む
2. 現在との差分を比較（追加ファイル数、テスト結果の変化、カバレッジの変化）
3. 比較レポートを作成

#### (3) 使い方

```
/checkpoint create "feature-start"     # 記録
/checkpoint verify "feature-start"     # 検証
/checkpoint list                       # 一覧
/checkpoint clear                      # 古いものを削除
```

#### (4) 実践例（弁護士業務での活用）

**シーン**: 法律文書管理システムの大規模改修

1. 改修開始前: `/checkpoint create "before-refactor"`
2. コア機能完成後: `/checkpoint create "core-done"`
3. PR前: `/checkpoint verify "before-refactor"` → 全体の変更量を確認
4. テストが増え、カバレッジが上がり、ビルドも通っている → 安心してリリース判断

#### (5) Tips/注意点

- **feature開始時には必ずcreate**: 後で「最初はどうだったっけ？」を防ぎます。
- **PR前には必ずverify**: レビュアーへの説明材料になります。
- **名前は段階が分かるように**: `checkpoint1` ではなく `feature-start`, `core-done` のように。

### 17. /pull-build（リモート同期＋自動ビルド）— 16点

#### (1) 概要

Gitリポジトリのリモートから最新の変更を取得し、変更があった場合に自動でビルドを行うツールです。

#### (2) 内部ロジック

1. **現在の状態確認**: ローカルに未コミットの変更があるかチェック
2. **プル実行**: リモートから変更を取得
3. **変更ファイルの判定**: `.py`, `.spec`, `requirements.txt` 等が変更されていればビルド実行
4. **ビルド実行**: PyInstallerで実行ファイルを生成
5. **結果サマリー**: プル結果とビルド結果を表示

**ビルドをスキップする変更**: ドキュメントのみ（`.md`, `.txt`）、設定ファイルのみ（`.json`, `.yaml`）

#### (3) 使い方

```
/pull-build
```

引数は不要です。

#### (4) 実践例（弁護士業務での活用）

**シーン**: PDF結合ツールを事務所の複数PCで共有開発

1. PC-Aでバグ修正 → `git commit` → `git push`
2. PC-Bで `/pull-build` → 自動で変更取得＋ビルド → 最新版の実行ファイルが生成
3. 事務所PCでもすぐに修正済みバージョンが利用可能

#### (5) Tips/注意点

- **ドキュメントのみ変更ならビルドスキップ**: 不要なビルドは自動で省かれます。
- **未コミット変更がある場合は確認**: 手元の作業内容があると、プル前に確認が入ります。
- **PyInstaller専用**: 現在はPythonプロジェクトビルドに対応しています。

### 18. legal-office-research（法律リサーチ総合スキル）— 17点

#### (1) 概要

法律に関わるリサーチ業務を3つのパターンで網羅する統合スキルです。特定地域の法律事務所競合調査、法制度の要件調査と適用分析、裁判例の網羅的調査と立証戦略の提言を行います。

#### (2) 内部ロジック

**パターン1: 法律事務所競合調査（3段階）**
1. 弁護士ドットコムで事務所一覧取得（弁護士会名簿との照合も必要）
2. 各事務所の公式HPから詳細確認
3. 不動産サイトから物件規模推定（推定根拠がない場合は「不明」と明記）

**パターン2: 法制度・要件調査 + 適用分析（5段階）**
1. 事実関係の整理
2. 制度の根拠法・概要の特定（e-Gov法令検索等）
3. 要件の体系的整理（主体要件、客体要件、手続要件、実体要件）

**パターン3: 裁判例・行政判断例の網羅的調査 + 立証戦略（5段階）**
1. 判断基準の体系的整理
2. 裁判例の網羅的収集（**捏造防止ルール**: WebSearchで確認できた判例のみ記載）
3. 裁判例の傾向分析
4. 本件への適用分析
5. 立証戦略の提言（証拠優先順位、想定反論と対応方針）

#### (3) 使い方

- **パターン1**: 法律事務所開業の市場分析が必要な時
- **パターン2**: 法制度が依頼者に適用できるか判断する時
- **パターン3**: 法的論点について提出資料の基礎を作成する時

#### (4) 実践例（弁護士業務での具体的ユースケース）

**実践例1: 倉敷市への法律事務所開業検討（パターン1）**
倉敷駅周辺・水島地区・岡山市南部の法律事務所を調査し、弁護士数の分布、取扱分野の空白地帯、参入機会を特定できます。

**実践例2: 未払賃金立替払制度の適用可能性判断（パターン2）**
「事実上の倒産」認定の要件を整理し、依頼者の状況を当てはめ、判断マトリクスで各要件の充足見込みを評価。具体的なアクションプランを提示できます。

**実践例3: リモートワーク労働者性の立証戦略（パターン3）**
昭和60年労基法研究会報告の5要素を整理し、最高裁判例を収集。振込記録が最強の間接証拠であることを示し、想定反論への対応方針を提言できます。

#### (5) Tips/注意点

- **推定根拠の明記ルール**: 物件面積等の推定根拠がない場合は「不明」と明記します。
- **捏造防止ルール**: 裁判例はWebSearchで確認できたもののみ記載。推測で裁判例を創作すると致命的な誤りとなります。
- **品質レベルの違い**: パターン2は依頼者説明レベル、パターン3は提出資料レベルです。
- **並列実行の推奨**: 地域別に分担すれば調査時間を1/3に短縮できます。

### 19. n8n-drive-notion-sync（Google Drive→Notion自動連携）— 16点

#### (1) 概要

Google DriveにアップロードされたファイルをトリガーとしてNotion DBに自動登録する、n8nワークフロー構築パターンです。ファイル命名規則を元に書面種別・証拠番号・案件名を自動抽出します。

#### (2) 内部ロジック

1. **Google Drive Trigger**: 特定フォルダ配下でファイル作成を5分間隔で監視
2. **Function Node**: 正規表現で `YYYYMMDD_種別番号_書面名.拡張子` を分解
3. **IF Node**: パース成功なら書面DBに登録、失敗なら未分類DBに登録
4. **Notion Node**: 書面名・種別・証拠番号・提出日・Driveリンクを登録
5. **Error Trigger**: パース失敗時にGoogle Chat通知やリトライを実行

**ファイル命名規則の具体例**:
| 書面種別 | ファイル名例 |
|---------|-------------|
| 甲号証 | `20260212_甲1_売買契約書.pdf` |
| 甲号証（枝番） | `20260212_甲1-2_契約書別紙.pdf` |
| 準備書面 | `20260220_準備書面1_原告第一準備書面.pdf` |
| 訴状 | `20260210_訴状_損害賠償請求.pdf` |

#### (3) 使い方

- **通常**: 命名規則に従ってファイルをアップロードすると自動でNotion登録
- **命名規則違反**: 未分類DBに振り分け → 週次レビューで修正
- **バックフィル**: Split In Batchesで10ファイルずつ一括登録

#### (4) 実践例（弁護士業務での具体的ユースケース）

**実践例1: 訴訟案件の書面管理自動化**
`20260210_訴状_損害賠償請求.pdf` をアップロード → 自動でNotion書面DBに登録 → 案件ページから関連書面を一覧確認

**実践例2: 証拠管理の枝番対応**
甲1-2のような枝番にも対応。証拠番号順にソートしてエクスポート可能

**実践例3: エラーハンドリング**
命名規則違反ファイルは未分類DBに登録 + Google Chat通知 → 即座に修正可能

#### (5) Tips/注意点

- **命名規則の厳守**: ワークフローの成否は命名規則に100%依存します。
- **Notion API Rate Limit**: 3回/秒の制限。Wait Node（300ms待機）を挿入してください。
- **セキュリティ**: Google Drive OAuth2認証は読取専用スコープ（`drive.readonly`）を推奨します。

## よくある質問（FAQ）

### Q1: スキルとコマンドの違いは何ですか？

**スキル**は、特定の条件を満たしたときに自動で起動する補助機能です。利用者が意識しなくても裏で動きます。例えば、セキュリティレビュー（security-review）は認証機能を書いたときに自動で起動します。

**コマンド**は、チャットで `/コマンド名` と入力して手動で呼び出す機能です。利用者が「今やりたい」と判断したタイミングで実行します。例えば、`/plan` は「この機能を作る前に計画を立てよう」と思ったときに実行します。

### Q2: 技術的な知識がなくても使えますか？

はい、使えます。本マニュアルのツールはすべて、「〜して」「〜を確認して」のような自然な日本語で指示するだけで動作します。内部で何が起きているかを理解する必要はありません。ただし、出力結果（エラーメッセージや改善提案）を読んで判断する場面はあります。判断に迷ったときは、Claude Codeに「これはどういう意味？」と聞けば、平易な言葉で説明してくれます。

### Q3: 最初に覚えるべきツールはどれですか？

以下の3つから始めることをお勧めします:
1. **/plan** — 何かを作る前に計画を立てる習慣をつける
2. **/pub** — 変更を安全にGitHubに反映する
3. **legal-document-namer** — 訴訟書面のファイル名整理を効率化する

この3つだけで、日常業務の大部分をカバーできます。慣れてきたら `/verify`、`/code-review`、`/sessions` に広げていきましょう。

### Q4: ツール同士を組み合わせるコツはありますか？

はい、以下の組み合わせが効果的です:

**開発ワークフロー**: `/plan` → `/orchestrate feature` → `/verify` → `/pub`
（計画 → 実装 → 検証 → 公開）

**品質管理ワークフロー**: `/code-review` → security-review → `/verify`
（コードレビュー → セキュリティチェック → 総合検証）

**知識蓄積ワークフロー**: 作業完了 → `/learn` → `/revise-claude-md`
（パターン抽出 → CLAUDE.md更新）

### Q5: Google Chatへの通知はどうやって設定しますか？

Google Chat Webhookを使います。Google Chatのスペース（チャットルーム）設定から「Webhook」を追加すると、専用のURLが発行されます。このURLに対してHTTPリクエストを送ると、チャットにメッセージが表示されます。n8n-automation-patternsのエラーハンドリングや、定期レポート送信で利用されています。
