# Sランクスキル 詳細解説（Part 1）

Claude Codeの高評価スキル7件について、仕組み・使い方・実践例を解説します。

## 1. strategic-compact（戦略的コンテキスト圧縮）- 19点

### (1) 概要

AIが一度に処理できる情報量（コンテキストウィンドウ）には上限があります。このスキルは、作業の区切りの良いタイミングで「今までの会話を要約して整理する」ことを提案するツールです。適切なタイミングで圧縮することで、重要な情報を失わずに長時間の作業を続けられます。

### (2) 内部ロジック

1. **ツール呼び出しを監視**: Claude Codeが「ファイル編集」や「ファイル作成」を行うたびに、内部でカウンターが増えます。
2. **閾値判定**: 50回のツール使用に達すると、「そろそろ /compact（圧縮コマンド）を実行しませんか？」と提案します。
3. **定期リマインダー**: その後も25回ごとに再度提案を行います。
4. **実装方式**: `suggest-compact.sh` というシェルスクリプトがPreToolUse Hook（ツール使用前に動く仕組み）として動作します。

このスキルは自動で圧縮するのではなく、「提案だけ」を行います。最終判断は利用者に委ねられます。

### (3) 使い方

このスキルは設定済みであれば自動で提案が表示されます。利用者が行うのは以下の操作だけです:

- **提案が表示されたとき**: `/compact` とコマンドを入力すると、会話履歴が要約され、コンテキストがリフレッシュされます。
- **無視したいとき**: 何もせず作業を続ければOKです。次の25回後に再度提案されます。

手動でいつでも `/compact` を実行することもできます。

### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース**: 複雑な訴訟案件で準備書面を作成中。契約書の分析、判例検索、証拠整理、書面起案を1セッションで続けている。

1. **契約書分析**: Claude Codeに契約書を読み込ませ、争点を整理（20回のツール呼び出し）
2. **判例検索**: 関連判例をWebで検索し、要旨を整理（25回のツール呼び出し）
3. **証拠整理**: 証拠書類のファイル名をリネームし、証拠説明書を作成（30回のツール呼び出し）
4. **👉 ここで提案が表示**: 「/compact を実行しませんか？」
5. **圧縮を実行**: `/compact` を入力。契約書の詳細や判例の全文は要約され、争点と証拠リストだけが残る。
6. **書面起案**: 軽くなったコンテキストで準備書面を起案開始。新しい作業に集中できる。

**効果**: 長時間の作業で会話が肥大化しても、区切りの良いタイミングで整理することで、AIが「何が重要か」を見失わずに済みます。

### (5) Tips/注意点

- **圧縮のタイミング**: 「調査フェーズが終わって、実行フェーズに入る直前」がベスト。調査結果の詳細は要約し、これからやることだけを残す。
- **圧縮しすぎ注意**: 作業の途中で圧縮すると、細かい指示やメモが消えてしまう可能性があります。大きな区切りまで待ちましょう。
- **手動でもOK**: 提案を待たずに、自分で「今がタイミングだ」と思ったら `/compact` を実行してもOKです。
- **代替手段**: 圧縮の代わりに「メモリに保存」する手もあります。重要な情報は `/remember` コマンドでセッションを超えて保存できます。

## 2. claude-md-improver（CLAUDE.md改善ツール）- 19点

### (1) 概要

CLAUDE.mdとは、プロジェクトやフォルダごとに置く「Claude Code向けの説明書」です。このファイルに書かれた内容は、Claude Codeが毎回自動で読み込みます。claude-md-improverは、このCLAUDE.mdの品質を評価し、不足している情報を追加するツールです。

### (2) 内部ロジック

1. **発見フェーズ**: プロジェクト内の全CLAUDE.mdファイルを検索します（`find . -name "CLAUDE.md"`）。
2. **品質評価フェーズ**: 各ファイルを6つの基準で採点します:
   - コマンド/ワークフローの記載（20点）
   - アーキテクチャの明確さ（20点）
   - 非自明なパターンの記載（15点）
   - 簡潔さ（15点）
   - 最新性（15点）
   - 実行可能性（15点）
3. **レポート出力**: 各ファイルの評価をA〜F判定で表示します。
4. **改善提案**: 「このセクションを追加すべき」という具体的な提案を出します。
5. **承認後に更新**: 利用者が承認したら、Edit toolを使ってCLAUDE.mdを更新します。

このスキルは「読み取り専用」ではなく、承認後に実際にファイルを**書き換える**点が特徴です。

### (3) 使い方

**手動起動**: 「CLAUDE.mdをチェックして」「CLAUDE.mdを改善して」と依頼すると、このスキルが起動します。

**自動起動**: `/pub` コマンド（ドキュメント同期→コミット→プッシュの一括処理）の中で自動的にCLAUDE.mdの品質チェックが行われることもあります。

**基本的な流れ**:
1. 「CLAUDE.mdをチェックして」と指示
2. 品質レポートが表示される
3. 改善提案が提示される
4. 「この提案で更新してください」と承認
5. CLAUDE.mdが更新される

### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース**: 法律事務所で顧問先企業向けの契約書レビューツール（スクリプト）を開発中。

**状況**: プロジェクトフォルダに契約書分析スクリプト（Python）がいくつかあるが、CLAUDE.mdは古いまま。新しいメンバーがClaude Codeを使って修正しようとしても、どのスクリプトが何をするのか分からない。

**対処**:
1. 「CLAUDE.mdをチェックして」と依頼
2. **評価結果**: D判定（30点）。理由:
   - 実行コマンドが書かれていない（ビルド方法不明）
   - 各スクリプトの役割が未記載（アーキテクチャ不明）
   - 契約書の種類ごとの処理パターンが書かれていない（非自明なパターン欠落）
3. **改善提案が表示**:
   - 「## 実行方法」セクションに `python analyze_contract.py` を追加
   - 「## スクリプト構成」セクションに各ファイルの役割を記載
   - 「## 注意点」に「業務委託契約は analyze_outsourcing.py、秘密保持契約は analyze_nda.py を使う」と追記
4. **承認して更新**: 提案を承認すると、CLAUDE.mdが自動で更新される
5. **効果**: 次回から新しいメンバーがClaude Codeを起動すると、自動で更新されたCLAUDE.mdが読み込まれ、各スクリプトの使い方が分かる

**ポイント**: CLAUDE.mdは「人間向けのREADME」ではなく「AI向けの取扱説明書」です。人間が読んで分かることでも、AIに明示的に書いておかないと見落とされることがあります。

### (5) Tips/注意点

- **定期メンテナンスが重要**: プロジェクトが進化するとCLAUDE.mdは古くなります。月1回くらいでこのスキルを実行して鮮度を保ちましょう。
- **簡潔さを保つ**: 詳細すぎるCLAUDE.mdは読み込みに時間がかかり、コンテキストウィンドウを圧迫します。「本当に必要な情報だけ」を残す方針で。
- **個人設定は別ファイルで**: チーム共有のCLAUDE.mdには個人の好みを書かず、`.claude.local.md` に分けましょう。
- **ショートカット**: セッション中にClaude Codeが学んだことを即座にCLAUDE.mdに追加したいときは `#` キーを押すと、自動で追記してくれます。

## 3. pull-merge-pub（Pull→Merge→Push統合ワークフロー）- 18点

### (1) 概要

ソースコード管理ツール（Git）で、リモート（GitHub等）の最新版を取得（pull）し、ローカルの変更とマージ（merge）し、最終的にリモートに送信（push）するまでの一連の流れを安全に実行するスキルです。コンフリクト（衝突）が起きたときは自動で止まり、利用者に判断を求めます。

### (2) 内部ロジック

**フェーズ1: Pull準備**
1. 現在のブランチ名とリモート設定を確認（`git branch -a`, `git remote -v`）
2. ローカルに未コミットの変更があるかチェック（`git status --short`）
3. 変更がある場合は「コミットするか、stash（一時退避）するか」を判断
4. `git pull origin <branch>` を実行

**フェーズ2: コンフリクト判定**
1. pull結果を解析:
   - `Already up to date.` → 変更なし、そのまま進む
   - `Fast-forward` → 自動マージ成功、差分を確認
   - `CONFLICT` → コンフリクト発生
2. コンフリクトが軽微（設定ファイルの値だけ等）→ 自動解決
3. コンフリクトが深刻（ソースコードのロジック部分等）→ 停止して利用者に確認

**フェーズ3: /pub実行**
1. ドキュメントの翻訳同期チェック
2. `.gitignore` の整理
3. シークレット（APIキー等）のスキャン
4. コミット（変更を記録）
5. プッシュ（リモートに送信）

### (3) 使い方

**基本的な呼び出し方**:
- 「リモートの最新版をpullしてマージして、pubして」と指示
- または「pull-merge-pubワークフローを実行して」と指示

**典型的な流れ**:
1. ローカルで作業した内容がある状態で上記の指示を出す
2. Claude Codeがローカルの変更を確認し、必要に応じてコミットを促す
3. pullが実行される
4. コンフリクトがなければ自動で進み、/pubまで完了
5. コンフリクトがあれば停止し、「どちらを優先しますか？」と聞かれる

### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース**: 事務所の複数の弁護士が同じ判例データベース管理スクリプト（Pythonプロジェクト）を共同で更新している。

**状況**: あなたはローカルで「最高裁判例の検索機能」を追加。同僚は同じ期間に「下級審判例のフォーマット修正」をリモートにプッシュ済み。

**操作**:
1. 「リモートの最新版をpullしてマージして、pubして」と指示
2. **Pull準備**: Claude Codeがローカルに未コミットの変更（検索機能追加）を検出
3. **コミット確認**: 「先にコミットしますか？」→「はい」と応答
4. **Pull実行**: リモートから同僚の変更（フォーマット修正）を取得
5. **コンフリクト判定**:
   - 同僚の変更とあなたの変更が別ファイル → 自動マージ成功
   - もし同じファイルの同じ箇所を修正していたら → コンフリクト発生、停止
6. **コンフリクト解決** (発生した場合):
   - Claude Codeが「以下のファイルでコンフリクトが発生しました」とリスト表示
   - 「どちらの変更を優先しますか？」と選択肢が提示される
   - 利用者が選択すると、その内容でマージ完了
7. **/pub実行**: マージ完了後、自動でドキュメント同期・コミット・プッシュが実行される
8. **完了**: リモートに両方の変更が統合された状態で反映される

**効果**: 手動で `git pull`, `git merge`, `git commit`, `git push` を順番に実行する手間が省け、コンフリクトが起きても安全に対処できます。

### (5) Tips/注意点

- **pull前に必ずコミット**: ローカルの未保存の変更がある状態でpullすると、変更が上書きされるリスクがあります。このスキルはそれを防ぐため、先にコミットを促します。
- **コンフリクトは慎重に**: 深刻なコンフリクト（ロジック部分の衝突）は自動解決せず、必ず利用者に確認します。急いでいても確認を飛ばさないでください。
- **force pushは禁止**: このスキルは `git push --force` を使いません。force pushはリモートの他人の変更を破壊する危険があるためです。
- **定期的なpull習慣**: チーム開発では、朝一番や作業開始時に必ずpullする習慣をつけると、大きなコンフリクトを避けられます。

## 4. security-review（セキュリティレビュー）- 17点

### (1) 概要

セキュリティ上の脆弱性（セキュリティホール）をチェックするツールです。認証（ログイン処理）、ユーザー入力の扱い、APIキーの管理など、10項目の観点で「危険なコードがないか」を自動で検査します。弁護士業務では顧問先企業のWebアプリ開発支援や、事務所内ツールの安全性確認に使えます。

### (2) 内部ロジック

このスキルは10項目のチェックリストを持っており、各項目について「良い例」「悪い例」「検証手順」が定義されています:

1. **シークレット管理**: APIキーやパスワードがソースコードに直接書かれていないか
2. **入力バリデーション**: ユーザーが入力したデータを信頼せず、必ず検証しているか
3. **SQLインジェクション対策**: データベースクエリがパラメータ化されているか
4. **認証・認可**: ログイン処理とアクセス権限のチェックが正しいか
5. **XSS対策**: ユーザーが入力したHTMLをそのまま表示していないか
6. **CSRF対策**: なりすまし攻撃への対策があるか
7. **レート制限**: 同じIPから大量のリクエストを受けても耐えられるか
8. **機密データの露出**: エラーメッセージにパスワード等が含まれていないか
9. **ブロックチェーン関連**: ウォレット署名検証が正しいか（該当する場合のみ）
10. **依存パッケージ**: 古いライブラリに脆弱性がないか

**動作方式**:
- コードを読み込み、上記10項目のパターンマッチングを実行
- 問題を発見したら「CRITICAL（緊急）」「HIGH（高）」「MEDIUM（中）」でランク付け
- 修正案を提示

### (3) 使い方

**自動起動**: 認証機能、API連携、ユーザー入力処理を書いたとき、このスキルが自動で起動します。

**手動起動**: 「セキュリティレビューして」と指示するだけです。

**レビュー後の流れ**:
1. 問題リストが表示される
2. CRITICALとHIGHは**必ず修正**
3. MEDIUMは可能な限り修正
4. 修正後、再度レビューを実行して問題が消えたことを確認

### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース1: 顧問先スタートアップのWebアプリ開発支援**

**状況**: 顧問先のスタートアップ企業がユーザー登録機能を持つWebアプリを開発中。弁護士として、サービス利用規約を作成するとともに、個人情報保護の観点から技術面の安全性も確認したい。

**操作**:
1. 開発中のソースコードをClaude Codeで開く
2. 「セキュリティレビューして」と指示
3. **検出された問題**:
   - **CRITICAL**: パスワードがハッシュ化されずに平文でデータベースに保存されている
   - **HIGH**: ログイン失敗時のエラーメッセージに「そのメールアドレスは登録されていません」と表示（メアド存在確認の情報漏洩）
   - **MEDIUM**: レート制限がなく、ブルートフォース攻撃（総当たり攻撃）に脆弱
4. **修正指示**: Claude Codeに「CRITICALとHIGHを修正して」と指示
5. **修正内容**:
   - パスワードをbcryptでハッシュ化する処理を追加
   - エラーメッセージを「メールアドレスまたはパスワードが間違っています」に変更
6. **再レビュー**: 「もう一度セキュリティレビューして」→ 問題なしと確認
7. **顧問先への報告**: 「技術的な安全性を確認しました。個人情報保護法の観点からも問題ありません」と助言

**ケース2: 事務所内の案件管理ツール**

**状況**: 事務所で使っている案件管理ツール（Pythonスクリプト）を改修中。依頼者情報を扱うため、情報漏洩リスクを最小化したい。

**操作**:
1. 「セキュリティレビューして」と指示
2. **検出された問題**:
   - **CRITICAL**: データベース接続情報（パスワード）がソースコードに直接書かれている
3. **修正**: 環境変数に移動し、`.env` ファイルに記載（`.env` は `.gitignore` に追加）
4. **効果**: GitHubに誤ってプッシュしても、パスワードは漏れない

### (5) Tips/注意点

- **CRITICALは即修正**: 緊急度CRITICALの問題は、そのまま運用すると情報漏洩や不正アクセスのリスクがあります。必ず修正してください。
- **デプロイ前に必ず実行**: 本番環境（実際にユーザーがアクセスする環境）にリリースする前に、必ずセキュリティレビューを通してください。
- **依存パッケージの更新**: `npm audit` や `pip-audit` で定期的に古いライブラリをチェックし、脆弱性があれば更新します。
- **過信は禁物**: このスキルは既知の脆弱性パターンをチェックしますが、完璧ではありません。重要なシステムでは専門のセキュリティ診断サービスの利用も検討してください。

## 5. legal-document-namer（法律文書命名規則）- 17点

### (1) 概要

法律事務所の事件フォルダで使うファイル名を、フォルダ種別ごとのルールに従って自動生成するツールです。「02 協議」「03 調停・審判」「04 訴訟」などのフォルダごとに異なる命名パターンがあり、それを手動で覚える必要がなくなります。証拠番号（甲01, 乙02等）の連番生成機能もあります。

### (2) 内部ロジック

このスキルは3つのモジュールで構成されています:

**1. LegalTemplatePreset（テンプレート管理）**:
- フォルダ種別（02協議, 03調停審判, 04訴訟, 05送付書類, 07手持ち資料, 08メール）ごとに命名パターンを保持
- 各パターンは「テンプレート文字列」と「入力フィールド定義」のセット
- 例: `{提出日}_{受信者}へ_{文書名}` というテンプレートに対し、`{"提出日": "20260209", "受信者": "先方代理人", "文書名": "和解案"}` を渡すと `20260209_先方代理人へ_和解案` を生成

**2. DateHelper（日付処理と枝番）**:
- 今日の日付をYYYYMMDD形式で取得
- 日付のバリデーション（2月30日のような不正な日付を検出）
- 同名ファイルがあるときの枝番自動付与（`20260209_準備書面` → `20260209_準備書面_2` → `20260209_準備書面_3`）

**3. EvidenceNumberer（証拠番号生成）**:
- 当事者（甲/乙）と開始番号・生成数を指定すると、連番リストを生成
- 例: `generate_numbers(party="甲", start=1, count=5)` → `["甲01", "甲02", "甲03", "甲04", "甲05"]`
- 証拠ファイル名も生成: `format_evidence_filename(date="20260209", evidence_number="甲01", document_name="契約書")` → `20260209_甲01_契約書`

**データ構造**: frozen dataclass（変更不可能なデータ構造）で命名ルールを管理し、間違って上書きされないように保護しています。

### (3) 使い方

**基本的な使い方**:
1. 「02協議フォルダの発信文書ファイル名を生成して」と指示
2. Claude Codeが「提出日は？」「受信者は？」「文書名は？」と順番に聞いてくる
3. 回答すると、命名ルールに従ったファイル名が生成される

**証拠番号の使い方**:
- 「甲号証を5件分、01から生成して」→ `甲01, 甲02, 甲03, 甲04, 甲05` が生成される
- 「甲05の証拠ファイル名を、契約書という名前で作って」→ `20260209_甲05_契約書` が生成される

**一括リネーム**:
- 複数ファイルに対して「このフォルダのファイルを全部、04訴訟の書証ルールでリネームして」と指示すると、一括で処理されます。

### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース: 民事訴訟で提出する証拠書類の整理**

**状況**: 訴訟案件で証拠書類が50件以上あり、ファイル名がバラバラ（`契約書.pdf`, `領収書スキャン.jpg`, `メール写し.pdf` 等）。裁判所提出前に、統一的な命名規則で整理する必要がある。

**手動での作業**（従来）:
1. 提出日をYYYYMMDD形式で手入力
2. 証拠番号を手動で割り当て（甲01, 甲02...）
3. ファイル名を1件ずつ `20260209_甲01_契約書.pdf` のように変更
4. 途中で同じ番号を2回使ってしまうミスが発生
5. 50件分で1時間以上かかる

**このスキルを使った場合**:
1. 「04訴訟フォルダの書証ファイル名を、甲01から50件分生成して」と指示
2. Claude Codeが証拠リスト（`甲01〜甲50`）を生成
3. 「このリストを使って、フォルダ内のファイルをリネームして」と指示
4. ファイルの種類を確認しながら、自動でリネーム実行:
   - `契約書.pdf` → `20260209_甲01_契約書.pdf`
   - `領収書スキャン.jpg` → `20260209_甲02_領収書.jpg`
   - `メール写し.pdf` → `20260209_甲03_メール.pdf`
5. 所要時間: 5分

**効果**:
- 時間が大幅短縮（1時間 → 5分）
- 命名ミス（番号の重複や日付フォーマットの間違い）がゼロになる
- 証拠説明書との紐付けが容易（番号が一貫しているため）

**追加の使い方: 調停書面の命名**

**ケース**: 家事調停で「主張書面第2回」を提出。ファイル名を正しい形式で作りたい。

**操作**:
1. 「03調停フォルダの申立人の主張書面ファイル名を生成して」と指示
2. Claude Codeが聞いてくる:
   - 「提出日は？」→ 「2月9日」
   - 「文書名は？」→ 「主張書面第2回」
3. 生成結果: `20260209_申立人_1_主張書面第2回`
   - `1` は符号（1=主張書面のルール）

### (5) Tips/注意点

- **日付は必ずYYYYMMDD**: このツールは `20260209` 形式のみ対応。`2026-02-09` や `令和8年2月9日` はエラーになります。
- **「相手方」は使用禁止**: 命名ルールでは「先方本人」「先方代理人」を使い、「相手方」という曖昧な表現は禁止されています。
- **裁判所名は支部まで**: 「東京家裁」ではなく「東京家裁立川支部」のように、支部名まで明記してください。
- **枝番は自動**: 同じ日に同じ名前のファイルを作ろうとすると、自動で `_2`, `_3` が付きます。手動で番号を振る必要はありません。
- **証拠番号は2桁ゼロ埋め**: 甲1ではなく甲01です。これにより、甲1〜甲99までファイル名でソートしたときに正しい順序になります。

## 6. n8n-expression-syntax（n8n式構文）- 16点

### (1) 概要

n8n（ノーコード自動化ツール）で、動的なデータを扱うための「式」の書き方を検証・修正するスキルです。n8nでは `{{$json.email}}` のような形式でデータを参照しますが、カッコの数や記法を間違えると動きません。このスキルは「よくある間違い」を検出し、正しい書き方を提案します。

### (2) 内部ロジック

このスキルは「パターンマッチング」と「構文検証」を組み合わせています:

**1. 基本ルールの検証**:
- 式は必ず `{{` と `}}` で囲まれているか
- ノード名（他のステップの参照）は引用符で囲まれているか（例: `{{$node["HTTP Request"].json}}`）
- フィールド名にスペースがあるときは角括弧記法を使っているか（例: `{{$json['field name']}}`）

**2. よくある間違いのパターン検出**:
- `$json.email` → `{{$json.email}}` に修正（カッコ忘れ）
- `{{$json.name}}` (Webhookノード) → `{{$json.body.name}}` に修正（Webhookは `.body` 配下にデータがある）
- `'={{$json.email}}'` (Codeノード内) → `$json.email` に修正（Codeノードでは `{{}}` 不要）
- `{{$node.HTTP Request}}` → `{{$node["HTTP Request"]}}` に修正（引用符忘れ）

**3. データ構造の確認**:
- Webhookノードの出力構造を確認し、`.body` 配下にユーザーデータがあることを検証
- 他のノードの出力構造と照らし合わせて、パスが正しいかチェック

**4. 修正案の提示**:
- 問題箇所を指摘
- 正しい構文を提示
- 必要に応じて、そのノードの出力例を表示

### (3) 使い方

**自動起動**: n8nのワークフロー設定ファイル（JSON）を開いて編集すると、このスキルが自動で構文チェックを行います。

**手動起動**: 「このn8n式を検証して」とエラーが出ている式を貼り付けると、どこが間違っているかを教えてくれます。

**典型的な流れ**:
1. n8nのワークフローで式を書く
2. エラーが出る
3. Claude Codeに「このエラーを修正して」と指示（エラーメッセージを貼り付け）
4. 正しい式が提示される
5. n8nに貼り付けて動作確認

### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース: 顧問先企業の問い合わせフォームをn8nで自動化**

**状況**: 顧問先企業のWebサイトに問い合わせフォームがあり、送信されると以下の処理を自動化したい:
1. 問い合わせ内容をSlackに通知
2. 問い合わせ者にメールで自動返信
3. Notionデータベースに記録

**n8nワークフロー構成**:
- **Webhookノード**: 問い合わせフォームからのデータを受け取る
- **Slackノード**: Slackの #customer-support チャンネルに通知
- **Emailノード**: 自動返信メールを送信
- **Notionノード**: データベースに記録

**問題発生**:
1. Slackノードのメッセージ欄に `新しい問い合わせ: {{$json.name}}` と書いた
2. テスト実行すると、`name`フィールドが空白で表示される

**原因**（このスキルが検出）:
- Webhookノードの出力構造は以下の通り:
  ```json
  {
    "headers": {...},
    "body": {
      "name": "山田太郎",
      "email": "yamada@example.com",
      "message": "契約書について相談したいです"
    }
  }
  ```
- ユーザーデータは `.body` 配下にあるのに、`{{$json.name}}` と書いている（ルート階層に `name` はない）

**修正**:
- Claude Codeが「Webhookノードのデータは `.body` 配下です。`{{$json.body.name}}` に修正してください」と提案
- 修正後の式:
  ```
  新しい問い合わせ
  名前: {{$json.body.name}}
  メール: {{$json.body.email}}
  内容: {{$json.body.message}}
  ```

**テスト結果**:
- Slackに正しく通知が届く:
  ```
  新しい問い合わせ
  名前: 山田太郎
  メール: yamada@example.com
  内容: 契約書について相談したいです
  ```

**追加の修正**: Emailノードでも同じ間違いをしていたため、一括で修正:
- 件名: `{{$json.body.name}}様、お問い合わせありがとうございます`
- 本文: `{{$json.body.message}}について承りました。`

### (5) Tips/注意点

- **Webhookは必ず `.body`**: n8nのWebhookノードは、ユーザーデータを `.body` の下に格納します。これを忘れると、全てのフィールドが空になります。
- **Codeノードでは `{{}}` 不要**: JavaScriptコードを書くCodeノードでは、`$json.email` のように直接書きます。`{{}}` を付けるとエラーになります。
- **ノード名は大文字小文字を区別**: `{{$node["http request"]}}` と書いても、実際のノード名が "HTTP Request" なら動きません。正確にコピーしてください。
- **エラーメッセージを活用**: n8nはエラーメッセージを表示してくれるので、それをClaude Codeに貼り付けると、素早く解決策が得られます。
- **式エディタでテスト**: n8nの各フィールドには「式エディタ」があり、リアルタイムで結果がプレビューされます。書いたらすぐ確認しましょう。

## 7. iterative-retrieval（段階的コンテキスト取得）- 16点

### (1) 概要

大規模なプロジェクトで、「どのファイルが関連しているか分からない」状況で、段階的に関連ファイルを絞り込んでいくスキルです。最初に広く浅くファイルを探し、関連度の高いものだけを深掘りし、最大3回のサイクルで最適なコンテキスト（AIが参照する情報）を構築します。

### (2) 内部ロジック

4つのフェーズを最大3回繰り返します:

**フェーズ1: DISPATCH（初回検索）**:
- 大まかなキーワードでファイルを検索（例: "認証", "ユーザー", "セッション"）
- 広範囲のパターンマッチング（例: `src/**/*.ts`, `lib/**/*.ts`）
- テストファイルや設定ファイルは除外

**フェーズ2: EVALUATE（関連度評価）**:
- 取得したファイルそれぞれに関連度スコア（0〜1）を付ける:
  - **0.8〜1.0（高）**: 目的の機能を直接実装している
  - **0.5〜0.7（中）**: 関連パターンや型定義を含む
  - **0.2〜0.4（低）**: 間接的に関連
  - **0〜0.2（なし）**: 無関係、除外
- 「何が足りないか」も記録（例: データベース接続部分が見つかっていない）

**フェーズ3: REFINE（検索条件の精緻化）**:
- 高関連度のファイルで使われているキーワードを抽出し、次の検索に追加
- 低関連度のファイルは除外リストに追加
- 「足りない部分」をターゲットにした検索条件を追加

**フェーズ4: LOOP（繰り返し判定）**:
- 高関連度のファイルが3件以上あり、重要な欠落がなければ終了
- そうでなければ、精緻化した条件で再度DISPATCH（最大3サイクル）

**終了条件**:
- 3サイクル完了、または
- 関連度0.7以上のファイルが3件以上見つかり、重要な欠落がない

### (3) 使い方

このスキルは**自動で動作**します。利用者が特別に呼び出す必要はありません。

**自動起動のタイミング**:
- 「この機能を実装して」と指示したとき、Claude Codeが関連ファイルを探すために内部で使用
- 大規模プロジェクト（100ファイル以上）で、どのファイルを読めばいいか不明な場合

**動作の見え方**:
1. 「関連ファイルを検索しています...」と表示
2. 数秒後、「3件の関連ファイルを見つけました」と報告
3. ファイルリストと、それぞれの関連理由が表示される

**手動でコントロールする場合**:
- 「認証機能に関連するファイルを段階的に検索して」と明示的に指示すると、検索プロセスが可視化されます。

### (4) 実践例（弁護士業務での具体的ユースケース）

**ケース: 事務所の契約書管理システムにバグ修正を依頼**

**状況**: 事務所で使っている契約書管理システム（100個以上のTypeScriptファイルで構成）で、「トークンの有効期限が切れてもログアウトされない」というバグが発生。どのファイルを修正すべきか不明。

**手動での作業**（従来）:
1. `src/` フォルダを開いて、全ファイルをざっと眺める
2. `auth.ts` を発見、読む → トークン検証ロジックはここにない
3. `session.ts` を発見、読む → セッション管理はあるが、有効期限チェックがない
4. `middleware/` フォルダを探す → 認証ミドルウェアを発見、ここにトークン検証があった
5. 所要時間: 30分〜1時間

**このスキルを使った場合**:

**サイクル1（DISPATCH）**:
1. 「トークンの有効期限チェックのバグを修正して」と指示
2. Claude Codeが初回検索: キーワード "token", "auth", "expiry" でファイル検索
3. 候補: `auth.ts`, `session.ts`, `user.ts`, `config.ts`

**サイクル1（EVALUATE）**:
- `auth.ts`: 関連度 0.9（認証ロジック実装）
- `session.ts`: 関連度 0.8（セッション管理）
- `user.ts`: 関連度 0.3（ユーザー情報だけ、トークン処理なし）
- `config.ts`: 関連度 0.2（設定ファイル）
- **足りないもの**: トークン検証ミドルウェアが見つかっていない

**サイクル1（REFINE）**:
- `user.ts` と `config.ts` を除外
- 新しいキーワード追加: "middleware", "jwt", "verify"

**サイクル2（DISPATCH）**:
- 精緻化した条件で再検索
- 新候補: `middleware/auth-middleware.ts`, `utils/jwt-utils.ts`

**サイクル2（EVALUATE）**:
- `middleware/auth-middleware.ts`: 関連度 0.95（ここにトークン検証ロジックがあった！）
- `utils/jwt-utils.ts`: 関連度 0.85（JWT検証ユーティリティ）

**サイクル2（LOOP）**:
- 高関連度ファイル: `auth.ts`, `session.ts`, `auth-middleware.ts`, `jwt-utils.ts` の4件
- 重要な欠落なし → 終了

**結果**:
- 所要時間: 10秒
- Claude Codeが「これらのファイルを確認しました」と報告し、バグ修正案を提示
- 実際のバグは `auth-middleware.ts` 内の有効期限チェック漏れだった

**効果**: 100ファイルのプロジェクトでも、関連する4ファイルだけを効率的に特定。手動で探す時間（30分〜1時間）が10秒に短縮。

### (5) Tips/注意点

- **大規模プロジェクト向け**: 小規模（10ファイル未満）では効果が薄いです。このスキルは100ファイル以上のプロジェクトで真価を発揮します。
- **初回検索は広めに**: 最初から狭い条件で検索すると、重要なファイルを見逃します。最初は広く、徐々に絞り込む戦略が正解です。
- **プロジェクトの命名規則に依存**: ファイル名やフォルダ名が機能を表していない場合（例: `file1.ts`, `utils2.ts`）、このスキルの精度が下がります。
- **3サイクルで見つからない場合**: それでも見つからないときは、手動でヒントを与えましょう。「認証処理は middleware フォルダにあります」と教えると、次の検索で見つかります。
- **学習効果**: このスキルは過去の検索結果を学習するため、同じプロジェクトで繰り返し使うほど精度が向上します。

**まとめ**

以上、Sランクスキル7件の詳細解説でした。これらのスキルは単独でも強力ですが、組み合わせて使うことでさらに効果的になります。例えば:

- **pull-merge-pub** でリモートの最新版を取得 → **security-review** で安全性確認 → **claude-md-improver** で変更をCLAUDE.mdに記録
- **iterative-retrieval** で関連ファイルを探索 → **strategic-compact** で不要なコンテキストを圧縮 → 効率的に作業

弁護士業務では、技術的な知識がなくても、これらのスキルを活用することで、顧問先企業への技術面での助言や、事務所内の業務効率化が実現できます。
