## 1. 要望一覧と現状分析

### 【要望1】枝番自動付与機能の削除

**現状**: `DateHelper.add_branch_number()` (legal_renamer.py:385-412) が同名ファイル衝突時に `_2`, `_3` を自動付与。`apply_legal_template()` (pdf_merger_tabbed.py:2310) で呼び出し。

**問題**: 法律事務所の実務では、枝番は意図的に付与するもの（甲01-1 等）。自動付与は意図しない番号が付く原因になる。

**変更方針**:
- `DateHelper.add_branch_number()` の呼び出しを `apply_legal_template()` から削除
- 同名衝突時は警告ダイアログで通知し、ユーザーに判断を委ねる
- `DateHelper.add_branch_number()` メソッド自体は残す（file_renamer.py側で利用可能性あり）

**変更箇所**:

| ファイル | 箇所 | 変更内容 |
|---------|------|---------|
| pdf_merger_tabbed.py | `apply_legal_template()` (L2310) | `add_branch_number()` 呼び出し削除、衝突検知→警告に変更 |

**変更量**: 約10行

- **6種対応**: 甲 / 乙 / 資料 / 添付資料 / 疎甲 / 疎乙

- `format_evidence_filename()`: 変更なし（汎用的に動作済み）

### 【要望6】元のファイル名を文書名に自動入力

**現状**: `generate_filename()` の `original_name` パラメータは存在するが未使用（L304: `original_name: str = ""`）。UIでは文書名フィールドは空欄で表示される。

**要望**: 文書名に元のファイル名を入れることが多いので入力を楽にしたい。

**変更方針**: 2つの機能を提供

#### 6-1. 文書名フィールドに元ファイル名を自動セット

テンプレートの動的フィールド生成時（`on_legal_subtype_selected()`）、「文書名」フィールドのデフォルト値に選択中の最初のファイルの元ファイル名をセットする。

ただし、複数ファイル選択時は1つ目のファイル名しかセットできないため、以下の補助機能も提供:

#### 6-2. 「元ファイル名を文書名に挿入」ボタン

テンプレートセクションに「元ファイル名→文書名」ボタンを追加。押下時の動作:

1. 各選択ファイルの `original_name` を取得
2. テンプレートの文書名フィールドに `original_name` を代入してファイル名を生成
3. ファイルごとに異なる文書名が適用される

**実装**: `generate_filename()` 内で `original_name` の活用

```python
def generate_filename(self, folder_type, subtype, fields, original_name=""):
    # ... 既存処理 ...
    # 文書名フィールドが空で original_name がある場合、自動充当
    for fd in st.fields:
        if fd.name == "文書名" and not str(fields.get("文書名", "")).strip():
            if original_name.strip():
                fields = {**fields, "文書名": original_name.strip()}
    # ... パターン展開 ...
```

**UI変更**:

```
┌─ ① テンプレート ──────────────┐
│ 種別: [▼]                     │
│ サブ: [▼]                     │
│ 日付: [20260214]               │
│ [動的フィールド]               │
│ ☑ 文書名に元ファイル名を使用  │  ← NEW: チェックボックス追加
│ [テンプレート適用]             │
└────────────────────────────────┘
```

チェック ON 時、`apply_legal_template()` でファイルごとに `original_name` を `fields["文書名"]` にセットしてから `generate_filename()` を呼ぶ。

**変更箇所**:

| ファイル | 箇所 | 変更内容 |
|---------|------|---------|
| pdf_merger_tabbed.py | テンプレートセクションUI (L1519-1562) | チェックボックス追加 |
| pdf_merger_tabbed.py | `apply_legal_template()` (L2264-2327) | original_name を fields に自動充当 |

### 【要望7】プレフィックス・サフィックスの統合（区切り操作）

**現状**: 一括操作に「プレフィックス追加」「サフィックス追加」「連番」「検索置換」の4ボタン。`_` 区切り固定。操作が分散して使いにくい。

**要望**: 「区切り目を指定して、挿入・置換・削除する」機能に統合。例: 「2区切り目に●●を挿入する」

**変更方針**: 新しい「区切り操作」UIに統合

#### 7-1. 区切り操作モジュール (file_renamer.py に追加)

```python
def segment_operation(
    self,
    selected_indices: list[int],
    separator: str = "_",
    position: int = 0,      # 0-indexed: 何番目の区切りか
    operation: str = "insert",  # insert / replace / delete
    value: str = "",
) -> None:
    """区切り文字で分割したセグメントに対する操作

    例: ファイル名 = "20260214_相手方へ_通知書"（separator="_"）
        segments = ["20260214", "相手方へ", "通知書"]

        position=1, operation="insert", value="甲01号証"
        → ["20260214", "甲01号証", "相手方へ", "通知書"]
        → "20260214_甲01号証_相手方へ_通知書"

        position=2, operation="replace", value="準備書面"
        → ["20260214", "相手方へ", "準備書面"]
        → "20260214_相手方へ_準備書面"

        position=1, operation="delete"
        → ["20260214", "通知書"]
        → "20260214_通知書"
    """
```

#### 7-2. UI変更

既存の4ボタン（プレフィックス/サフィックス/連番/置換）を統合した新UIに置き換え:

```
┌─ 一括操作 ─────────────────────┐
│ 【基本操作】                    │
│ [連番] [置換] [全選択] [全解除] │
│ [元に戻す]                      │
│                                 │
│ 【区切り操作】                  │
│  区切り文字: [_]                │
│  位置: [▼ 先頭/1/2/3/末尾]     │
│  操作: ◉挿入 ◯置換 ◯削除     │
│  値:  [          ]              │
│  [プレビュー] [適用]            │
│                                 │
│  ※ 現在のファイル名を区切り文字│
│    で分割した位置に対して操作   │
└─────────────────────────────────┘
```

**ポイント**:
- 「先頭」= プレフィックス相当、「末尾」= サフィックス相当
- 位置ドロップダウンは選択中ファイルの最大区切り数に応じて動的に更新
- プレビュー機能で適用前に結果を確認可能
- 連番と検索置換は独立したボタンとして残す（頻度が高い操作のため）

| pdf_merger_tabbed.py | 既存メソッド | `add_prefix_dialog()`, `add_suffix_dialog()` 削除 |

## 5. リスクと注意点

### 5.1 後方互換性

| リスク | 影響度 | 対策 |
|--------|-------|------|
| 枝番削除で同名衝突多発 | 中 | 衝突時に明確な警告ダイアログを表示 |
| 書証パターン変更（「号証」追加） | 低 | 「番号のみ」サブ種別で旧パターンに近い出力も可能 |
| プレフィックス/サフィックスボタン廃止 | 中 | 区切り操作の「先頭」「末尾」で同等機能を提供 |

### 5.2 操作感の変化

| 変更点 | ユーザーへの影響 | 緩和策 |
|--------|---------------|--------|
| 甲/乙ラジオ → ドロップダウン | 2クリック必要になる | デフォルト「甲」をセット |
| 4ボタン → 区切り操作パネル | 学習コスト | ToolTipとヘルプテキスト充実 |
| 個別指定モード | 新しい操作フロー | ハイライト表示で直感的に |

### 5.3 テンプレート定義の拡大

| 現状 | 改善後 |
|------|--------|
| 6フォルダ種別 × 14サブ種別 | 6フォルダ種別 × 19サブ種別 |

サブ種別追加は `_FOLDER_TYPES` のデータ追加のみで、ロジック変更なし。保守性への影響は低い。

### 5.4 Phase 3 の注意点

区切り操作（要望7）は既存のプレフィックス/サフィックスボタンを廃止するため:
- ユーザーに変更を事前告知する
- Phase 1, 2 が安定稼働してから着手推奨
- 必要なら旧ボタンを「非推奨」として一定期間残す選択肢もあり
