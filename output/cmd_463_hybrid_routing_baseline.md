## 1. 背景・目的

### 1.1 なぜ実証が必要だったか

shogun フォークではこれまで「ハイブリッド陣」(足軽1-3=Sonnet / 足軽4-5=Opus / 足軽6-7=Codex) を運用してきたが、モデル割当の根拠は経験則に留まっていた。Bloom レベル (L1-L6) によるルーティング (`bloom_routing: auto`) は導入済みだが、**同一 Bloom レベル内でどのモデルに振るべきか**は未確定。

### 1.2 cmd_463 の北極星

> 「ハイブリッド陣のモデル割当基準を実証データから策定し、家老が即日運用に適用できるガイドを永続化する」

本レポートはこの北極星を直接達成するための最終成果物である。

### 1.3 アプローチ

3カテゴリ × 2タイプ × 3モデル = **12 並列タスク** を実施。同一 task description / 同一制約条件下で各モデルに同じ仕事をさせ、出力品質・トークン効率・自己評価精度・運用適合度を横並び評価。全タスクは軍師 QC を経て本番品質を確認済み。

## 5. モデル特性サマリー

### 5.1 Claude Sonnet 4.6 (足軽1/2/3)

**強み**:
- **最小差分主義**: 元コード構造を最大限尊重し、変更箇所を最小化する。レビューしやすい。
- **自己懸念の明示精度が最高**: P1-1 のような実装詳細レベルの懸念を自分で発見し、改善案まで提示する能力が3モデル中最強。
- **改善提案の具体性**: A2 では全5件の改善提案に diff コードを併記。実装直結度が高い。
- **バランスの良さ**: ボリューム/精度/速度がすべて中位。ハズレの少ない選択肢。

**弱み**:
- **エッジケース見落とし**: B1 で macOS 非互換を P1 として宣言したが、Opus/Codex は最初から両対応した。
- **構造的概念化が中位**: Opus のような「設計核を命名する」深度には届かない。
- **トークン効率は中位**: Opus より少なく、Codex より多い。

**ベストフィット**:
- 既存コードへの最小侵襲な改修
- 実装後の精密な自己レビューが必要なタスク
- 短時間で本番投入したい場合
- 改善提案が即実装に繋がるべき場合

---

### 5.2 Claude Opus 4.6 + Extended Thinking (足軽4/5)

**強み**:
- **設計概念化能力**: 「二層配達モデル」のように設計核を命名・概念化する能力が3モデル中最強。
- **網羅性**: 改善提案の数 (a2: 12件)、評価軸数 (a1: 8軸)、テストケース数 (b1: 8+5)、引用数 (a1: 20件) すべてで最多。
- **一次情報の正確性**: 二次情報の誤りを GitHub API 等で訂正する自己検証能力。
- **フェーズ間知見転送**: 過去分析で発見した問題を後続実装で構造的解決する能力 (a2_opus → b2_opus)。
- **防御的設計**: defense-in-depth (複数箇所リセット)、後方互換 (3パターン吸収) など、堅牢性志向。
- **運用 UX 追加価値**: 仕様外の改善 (🔥 ntfy prefix) を自発的に提案。

**弱み**:
- **トークンコストが最も高い**: 設計書ボリュームが Sonnet の 1.9倍、Codex の 5.7倍。
- **過剰設計の傾向**: シンプルな案件でも複数代替案を検討するため、即決系には不向き。
- **応答速度**: Extended Thinking ON のため、Sonnet より遅い。

**ベストフィット**:
- アーキテクチャ設計・大規模リファクタリング
- 既存コード深層解析・依存関係マップ作成
- ライブラリ選定・技術判断
- 複雑な状態遷移を伴う実装
- 長期保守性が最重要なタスク
- フェーズ跨ぎの一貫性が必要な多段プロジェクト

---

### 5.3 Codex 5.3 (gpt-5-codex, 足軽6/7)

**強み**:
- **トークン効率最良**: 出力ボリュームが3モデル中最少だが、結論の質は他2モデルと同等。
- **構造的設計 (フェーズ2で逆転発見)**: 関数分離 (`week_monday_jst`, `send_priority_nudge`)・環境変数による外部調整 (`HIGH_PRIORITY_NUDGE_GAP_SEC`) など、移植性の高い設計を生む。
- **静的解析の正確性**: bash 構文・関数境界の判定が正確。
- **WebSearch なしでの調査能力**: `raw.githubusercontent.com` から一次情報を直接取得する戦術。
- **read-only 制約遵守**: 原本非変更・`output/` 生成のみで比較実験要件に厳密適合。

**弱み**:
- **分析系では深度に劣る**: A2 では主要関数のみ・主要環境変数のみで、Opus/Sonnet の網羅性に届かない。
- **改善提案の具体性が低い**: A2 で diff コードを併記しない。提案は方針止まり。
- **自己懸念の数が少ない**: P0:0/P1:1/P2:1 のように、Opus/Sonnet と比べ自己レビュー件数が少ない。
- **設計概念化が弱い**: Opus のような「設計核を命名する」深度には届かない。

**ベストフィット**:
- 単一ファイルのコード生成 (関数追加・既存パターン適用)
- 構造的責務分離が重要な実装 (関数抽出が自然)
- トークンコストを抑えたい繰り返しタスク
- 一次情報源 URL が明確な調査
- read-only 制約下での修正版生成

**🌟 特性逆転の重要性**: フェーズ1の分析系では「最簡素」と評価された Codex が、フェーズ2の実装系では「最も構造的」と逆転する。これは「分析系=広く浅く読む / 実装系=狭く深く書く」という Codex の訓練特性が現れた可能性が高い。**Codex を分析系で評価して足切りすると、実装系で発揮される構造化能力を見逃す**。

## 7. Decision Tree (モデル選択フロー)

```
タスク受領
  │
  ├─【1】タスクの主目的は?
  │   │
  │   ├── 分析・調査 (read heavy) ──→ 【2A】
  │   ├── 実装・修正  (write heavy) ──→ 【2B】
  │   └── 設計・戦略  (think heavy) ──→ 【2C】
  │
  ├─【2A】分析・調査
  │   │
  │   ├── 既存パターンの当てはめ (L1-L3) ──→ Sonnet または Codex
  │   │   ├── トークン抑制最優先 ──→ Codex
  │   │   └── 改善 diff も欲しい ──→ Sonnet
  │   │
  │   └── 概念化・構造化が必要 (L4-L5) ──→ Opus
  │       └── (1000行超のコード深層解析、技術選定など)
  │
  ├─【2B】実装・修正
  │   │
  │   ├── 単一ファイル
  │   │   │
  │   │   ├── 既存パターン適用 (L3) ──→ Codex (関数分離が自然)
  │   │   ├── 最小差分修正 ──→ Sonnet
  │   │   └── 高互換 + 多テスト ──→ Opus
  │   │
  │   └── 複数ファイル横断
  │       │
  │       ├── シンプルな機能追加 ──→ Codex (関数抽出で責務分離)
  │       ├── 状態遷移・防御層 ──→ Opus (state flag + defense-in-depth)
  │       └── 最小差分主義 ──→ Sonnet
  │
  └─【2C】設計・戦略
      │
      ├── 短い設計メモ (~100行) ──→ Codex
      ├── 標準設計書 (200-300行) ──→ Sonnet
      ├── 深い設計書 (400行+) ──→ Opus
      └── アーキテクチャ判断・代替案 ──→ Opus (4案以上検討)
```

### 7.1 緊急判定の補助フロー

```
時間制約は厳しいか?
  ├── YES → Sonnet または Codex
  │   ├── 既存パターンあり → Codex
  │   └── 新規ロジック必要 → Sonnet
  │
  └── NO → タスク種別で再判定
      └── 深い思考が価値を生むなら Opus
```

### 7.2 トークンコスト感度の補助フロー

```
コスト最重要?
  ├── YES → Codex
  │   └── (出力品質は同等で、行数 1/3-1/5)
  │
  └── NO → タスク種別で再判定
```

## 10. 結論 (Executive Summary)

cmd_463 は12 並列タスクで Sonnet 4.6 / Opus 4.6 / Codex 5.3 を実証比較し、以下の3つの重要な経験則を確立した:

### 10.1 三つの経験則

1. **Opus = 思考の深さの王**: 設計概念化、フェーズ間知見転送、防御的設計、網羅的検証で他2モデルを引き離す。コストと引き換えに、長期保守性を最大化する選択肢。
2. **Sonnet = 自己レビューの王**: 実装直後の自己懸念検出精度が3モデル中最高。最小差分主義と改善 diff 提示で実装直結度が最強。
3. **Codex = 構造化の隠れた王**: 分析系では簡素だが、**実装系では関数分離・環境変数・責務分離で最も移植性の高い設計を生む**。トークン効率も最良。**特性逆転を見逃さないことが重要**。

### 10.2 即日適用可能なルール

家老は以下のデフォルトをすぐに使える:

- 単発バグ修正/L1-L3 → **Sonnet**
- 既存パターン適用/関数追加 → **Codex**
- アーキテクチャ・大規模解析・ライブラリ選定 → **Opus**
- フェーズ跨ぎの一貫性が必要なプロジェクト → **Opus 単独**
- 高頻度の小タスク → **Codex 単独**

### 10.3 永続化

本レポートは `output/cmd_463_hybrid_routing_baseline.md` (本ファイル) と `docs/agent-routing-baseline.md` (家老の常時参照用) の2箇所に保存される。家老は cmd を書く前に後者を参照し、Routing Matrix・Decision Tree から最適モデルを選択せよ。
