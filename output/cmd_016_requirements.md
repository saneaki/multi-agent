# Gmail自動管理システム 要件定義書

## 概要

法律事務所がクライアントからのGmail受信メールを自動処理し、以下を実現するシステムの要件を定義する。

- クライアント別スプレッドシートに受信メール情報を記録
- メールをPDF化してGoogle Driveの指定フォルダに保存
- Gemini APIによるメール概要の自動生成
- マスタースプレッドシートによる全クライアント一元管理

**技術基盤**: Google Apps Script (GAS) + Google Workspace API群

## 2. データフロー

```
[Gmail受信]
    │
    ▼
(1) メール検索・取得
    │  入力: Gmail検索クエリ（ラベル/送信元フィルタ）
    │  処理: GmailApp.search() で未処理メール取得
    │  出力: GmailMessage[] オブジェクト配列
    │
    ▼
(2) クライアント判定
    │  入力: メール送信者アドレス/ドメイン
    │  処理: マスタースプレッドシートの送信元マッピングと照合
    │  出力: クライアント情報（名前、フォルダID、シートID等）
    │  例外: マッチしない場合 → 「未分類」フォルダに保存 + 警告ログ
    │
    ▼
(3) PDF変換
    │  入力: GmailMessage（HTML本文 + 添付ファイル）
    │  処理: HTML→Blob→PDF変換（添付ファイルは別途保存）
    │  出力: PDF Blob オブジェクト
    │  例外: 変換失敗 → エラーログ記録、次のメールへスキップ
    │
    ▼
(4) Google Drive保存
    │  入力: PDF Blob + クライアント別フォルダID
    │  処理: DriveApp.getFolderById().createFile(blob)
    │  出力: DriveファイルURL
    │  例外: フォルダ未存在 → 自動作成 or エラー報告
    │
    ▼
(5) Gemini要約生成
    │  入力: メール本文テキスト
    │  処理: Gemini API呼び出し（プロンプト付き）
    │  出力: 要約テキスト（200字以内目安）
    │  例外: API制限/エラー → 「要約生成失敗」と記録
    │
    ▼
(6) スプレッドシート記録
    │  入力: メールメタ情報 + PDFリンク + 要約
    │  処理: クライアント別シートに行追加 + マスター集計更新
    │  出力: スプレッドシートの新規行
    │
    ▼
(7) 処理済みマーク
    │  処理: メールIDをPropertiesServiceに保存（重複防止）
    │  補助: 処理済みメールにGmailラベル付与（視覚的確認用）
```

## 3. マスタースプレッドシート設計

マスタースプレッドシートはシステム全体の設定管理を担う。以下の2シートで構成する。

| カラム名 | データ型 | 制約 | 説明 |
|---------|---------|------|------|
| client_id | String | PK, 必須, ユニーク | クライアント識別子（例: `CL001`） |
| client_name | String | 必須 | クライアント名（日本語） |
| email_pattern | String | 必須 | メールアドレスまたはドメイン（例: `*@example.co.jp`） |
| spreadsheet_id | String | 必須 | クライアント別スプレッドシートのID |
| sheet_name | String | デフォルト: `メール一覧` | 対応シート名 |
| drive_folder_id | String | 必須 | PDF保存先Google DriveフォルダID |
| is_active | Boolean | デフォルト: TRUE | 有効/無効フラグ |
| created_at | Date | 自動設定 | 登録日 |
| notes | String | 任意 | 備考 |

### 3.2 実行ログシート (`execution_log`)

| カラム名 | データ型 | 説明 |
|---------|---------|------|
| execution_id | String | 実行ID（タイムスタンプベース） |
| started_at | DateTime | 実行開始日時 |
| finished_at | DateTime | 実行終了日時 |
| emails_processed | Number | 処理メール数 |
| emails_failed | Number | 失敗メール数 |
| status | String | `success` / `partial` / `error` |
| error_details | String | エラー詳細（ある場合） |

各クライアントに1つのスプレッドシートを割り当てる。

### メール一覧シート

| カラム | データ型 | フォーマット | 説明 |
|-------|---------|------------|------|

| 送信者 | String | — | 送信者名 + メールアドレス |

| PDFリンク | String (URL) | ハイパーリンク | Google Drive上のPDFファイルへのリンク |

| 添付ファイル | String | — | 添付ファイル名（複数はカンマ区切り） |

## 5. Gmail→PDF変換方式の調査・比較

### 方式比較表

| 方式 | 概要 | メリット | デメリット | 追加OAuth不要 |
|------|------|---------|-----------|-------------|
| **A. Blob直接変換** | `Utilities.newBlob(html,'text/html').getAs('application/pdf')` | Drive APIスコープ不要、シンプル | インライン画像が欠落する場合あり | ○ |
| **B. Google Docs中間変換** | HTML→一時Googleドキュメント→PDF export | レイアウト保持が比較的良好 | Drive API必要、一時ファイルのクリーンアップ必要 | × |
| **C. pixelcog/gmail-to-pdf ライブラリ** | OSSライブラリによる変換 | インライン画像・添付ファイル対応、実績あり | 外部依存、メンテナンス状況の確認要 | △（ライブラリ依存） |
| **D. 外部APIサービス** | HTML→PDF変換の外部API（例: html2pdf.app） | 高品質なレンダリング | 外部依存、コスト発生、データが外部に出る | — |
| **E. Cloud Functions + Python** | GASからCloud Functions（Python）を呼び出し、WeasyPrint/reportlab等で高品質PDF生成 | 高品質PDF（CSS完全対応、画像保持）、柔軟なレイアウト制御、GAS実行時間制限の回避 | GCPプロジェクト必要、追加コスト（Cloud Functions無料枠あり）、構成複雑化 | × |

### 推奨方式: **A. Blob直接変換 + 補完処理**

**選定理由**:

1. **最小権限の原則**: 追加のOAuthスコープが不要（法律事務所のセキュリティ要件に適合）
2. **シンプルさ**: 外部依存なし、GAS標準機能のみで完結
3. **コスト**: 追加費用なし

**実装方針**:

```
// 基本変換フロー
1. message.getBody() → HTML文字列取得
2. インライン画像のCID参照をBase64データURIに置換（補完処理）
3. Utilities.newBlob(processedHtml, 'text/html', fileName + '.html')
4. blob.getAs('application/pdf')
5. DriveApp.getFolderById(folderId).createFile(pdfBlob)
```

**インライン画像の補完処理**:
- メール内の `cid:` 参照画像を `message.getAttachments()` から取得
- Base64エンコードしてHTML内の `<img src="cid:...">` を `<img src="data:image/...;base64,...">` に置換
- これによりBlob変換時に画像が保持される

**フォールバック**: 変換失敗時はHTML形式のまま保存し、エラーログに記録

### 段階的拡張

Phase 1ではGAS Blob変換（方式B推奨）を使用し、PDF品質に問題がある場合はPhase 2以降でCloud Functions + Python（方式E）に切り替える。これにより、初期段階は最小構成で迅速に導入し、必要に応じて高品質化する柔軟な拡張戦略を採る。

## 7. Google Driveフォルダ構成設計

### フォルダ階層

```
Gmail自動管理/                          ← ルートフォルダ
├── クライアントA（株式会社○○）/
│   ├── 2026/
│   │   ├── 01/
│   │   │   ├── 20260115_田中太郎_契約書送付の件.pdf
│   │   │   └── 20260120_田中太郎_追加資料について.pdf
│   │   └── 02/
│   │       └── 20260203_鈴木花子_打ち合わせ日程.pdf
│   └── 添付ファイル/
│       └── 2026/
│           └── 01/
│               └── 20260115_契約書_draft_v2.docx
├── クライアントB（○○法人）/
│   └── ...
└── _未分類/                            ← マッチしないメールの保存先
    └── 2026/
        └── ...
```

### 命名規則

| 対象 | 命名規則 | 例 |
|------|---------|-----|
| クライアントフォルダ | `{client_name}` | `株式会社ABC` |
| 年フォルダ | `yyyy` | `2026` |
| 月フォルダ | `MM` | `01`, `02`, ..., `12` |
| PDFファイル | `yyyyMMdd_{送信者名}_{件名}.pdf` | `20260115_田中太郎_契約書送付の件.pdf` |
| 添付ファイル | `yyyyMMdd_{元ファイル名}` | `20260115_契約書_draft_v2.docx` |

**命名規則の補足**:
- 件名が長い場合は50文字で切り捨て
- ファイル名に使用不可文字（`/ \ : * ? " < > |`）はアンダースコアに置換
- 同名ファイルが存在する場合は末尾に `_2`, `_3` を付与

### 年月サブフォルダの採用理由

- **検索性**: 月単位での絞り込みが容易
- **パフォーマンス**: 1フォルダ内のファイル数を抑制（Drive APIの一覧取得速度に影響）
- **運用**: 月次での棚卸し・アーカイブに対応

## 8. トリガー方式の検討

### 方式比較

| 方式 | 仕組み | メリット | デメリット |
|------|--------|---------|-----------|
| **時間駆動（定期実行）** | GAS時間トリガーで定期的にメール検索 | 設定シンプル、GAS内で完結 | リアルタイムではない、無駄な実行あり |
| **イベント駆動（Push通知）** | Gmail API + Cloud Pub/Sub | ほぼリアルタイム | GCP設定必要、watch()の7日更新必須、複雑 |

### 推奨方式: **時間駆動（5分間隔）**

**選定理由**:

1. **シンプルさ**: GASのトリガー設定のみで完結。GCPプロジェクトやPub/Subの設定不要
2. **十分な即時性**: 法律事務所のメール処理は5分遅延で問題ない（リアルタイム要件なし）
3. **安定性**: Push通知はwatch()の7日更新が必要。更新漏れで通知が停止するリスク
4. **コスト**: 追加のGCPリソース不要
5. **保守容易性**: GASスクリプトのみで完結するため、弁護士事務所のIT担当者でも運用可能

**実行頻度の設計**:

| 時間帯 | 頻度 | 理由 |
|--------|------|------|
| 営業時間（8:00-20:00） | 5分間隔 | メール処理の即時性確保 |
| 夜間（20:00-8:00） | 30分間隔 or 停止 | API クォータ節約 |

**トリガー設定**:
- メイントリガー: 5分間隔の時間駆動トリガー
- 日次メンテナンストリガー: 毎日深夜にクリーンアップ処理（ログ整理、プロパティ整理）

**GAS トリガー制約への対応**:
- 1ユーザー1スクリプトあたり最大20トリガー → メイン + メンテナンスの2つで十分
- 1日のトリガー総実行時間90分 → 1回あたり30秒以内を目標とすれば、5分間隔×12時間=144回×30秒=72分で制約内

## 9. エラーハンドリング設計

### 9.1 API制限への対処

| 制限 | 値（Consumer） | 値（Workspace） | 対処策 |
|------|--------------|----------------|--------|
| スクリプト実行時間 | 6分/回 | 6分/回 | バッチ処理（1回あたり処理件数の上限設定） |
| URL Fetch日次上限 | 20,000/日 | 100,000/日 | カウンター管理 + 上限接近時の警告 |
| メール読み書き日次上限 | 20,000/日 | 50,000/日 | 十分な余裕あり（通常利用では到達しない） |
| Gemini API無料枠 | 1,000 RPD | 同左 | 日次処理数のカウンター管理 |
| トリガー総実行時間 | 90分/日 | 同左 | 1回あたりの処理時間を監視 |

### 9.2 PDF変換失敗時の処理

```
PDF変換試行
    │
    ├── 成功 → Drive保存 → 通常フローへ
    │
    └── 失敗
        ├── リトライ（最大2回、指数バックオフ）
        │   ├── 成功 → 通常フローへ
        │   └── 最終失敗
        │       ├── HTML形式のまま保存（フォールバック）
        │       ├── スプレッドシートの「PDFリンク」に「(HTML)」注記
        │       ├── エラーログ記録
        │       └── 次のメール処理へ続行（全体停止しない）
```

### 9.3 重複処理防止

**方式**: PropertiesService + Gmailラベル の二重管理

1. **PropertiesService**: 処理済みメールIDを `processed_{messageId}` キーで保存
2. **Gmailラベル**: 処理済みメールに `_auto/processed` ラベルを付与
3. **検索時除外**: Gmail検索クエリに `-label:_auto/processed` を含める

**PropertiesServiceの容量対策**:
- プロパティストア上限: 500KB（約9,000プロパティ）
- 対策: 90日以上経過した処理済みIDを日次メンテナンスで削除
- ラベルは永続的に残るため、PropertiesService削除後もラベルで重複を検知可能

### 9.4 リトライ戦略

| エラー種別 | リトライ回数 | 間隔 | 備考 |
|-----------|------------|------|------|
| API一時エラー (5xx) | 3回 | 指数バックオフ（1s, 2s, 4s） | |
| レート制限 (429) | 3回 | 指数バックオフ（5s, 15s, 45s） | |
| PDF変換失敗 | 2回 | 1s, 3s | 最終失敗→HTMLフォールバック |
| Gemini API失敗 | 2回 | 1s, 3s | 最終失敗→「要約生成失敗」記録 |
| 認証エラー (401/403) | 0回 | — | 即座にエラーログ、手動対応必要 |

## 10. セキュリティ考慮

### 10.1 OAuthスコープ（最小権限の原則）

| スコープ | 用途 | 分類 |
|---------|------|------|
| `https://www.googleapis.com/auth/gmail.readonly` | メール本文・添付ファイルの読み取り | Restricted |
| `https://www.googleapis.com/auth/gmail.labels` | 処理済みラベルの付与 | Non-sensitive |
| `https://www.googleapis.com/auth/drive.file` | スクリプトが作成したファイルのみアクセス | Sensitive |
| `https://www.googleapis.com/auth/spreadsheets` | スプレッドシートの読み書き | Sensitive |
| `https://www.googleapis.com/auth/script.external_request` | Gemini API呼び出し（UrlFetchApp） | Non-sensitive |

**注意**:
- `gmail.readonly` はRestricted Scopeに分類される。社内利用（同一ドメイン内）であればGoogle Workspace管理者が承認可能
- `drive` ではなく `drive.file` を使用し、スクリプトが作成したファイルのみにアクセス範囲を限定

### 10.2 法律事務所のデータ機密性への対応

| 対策 | 説明 |
|------|------|
| **データ非外部送信** | メール本文はGemini API（Google AI Studio）にのみ送信。Google以外の外部サービスには一切送信しない |
| **Gemini APIのデータ利用ポリシー** | 有料枠: 入力データはモデルトレーニングに使用されない。無料枠: 要確認（利用規約で確認必要） |
| **アクセス制限** | スクリプトの実行権限は事務所管理者アカウントのみに付与 |
| **共有設定** | スプレッドシート・Driveフォルダの共有は事務所ドメイン内に限定 |
| **ログ管理** | 処理ログにメール本文を含めない（メタ情報のみ記録） |

### 10.3 データ保護

- **転送暗号化**: Google API通信はすべてTLS 1.2+（Google基盤による保証）
- **保存暗号化**: Google Drive/Sheetsはデフォルトでサーバー側暗号化（AES-256）
- **APIキー管理**: Gemini APIキーはPropertiesService（Script Properties）に保存。コードにハードコードしない

### 10.4 監査証跡

- **実行ログシート**: 全実行の日時・処理件数・エラーを記録（§3.2参照）
- **GAS実行ログ**: Apps Scriptダッシュボードで実行履歴を閲覧可能（30日間保持）
- **Driveアクティビティ**: Google Drive監査ログで全ファイル操作を追跡可能

### 10.5 個人情報保護への配慮

- 処理対象はクライアントから事務所宛のメールに限定（マスタースプレッドシートのフィルタ設定）
- Gemini APIへの送信時、不要な個人情報（署名ブロック等）を除去するフィルタの実装を推奨（Phase 3以降で検討）
- 要約結果に具体的な個人情報が含まれる場合のマスキング処理も将来検討

## 11. GASの技術的制約と対処法

### 制約一覧と対策

| 制約 | 値 | 影響 | 対処策 |
|------|-----|------|--------|
| **実行時間** | 6分/回 | 大量メール処理時にタイムアウト | バッチ処理（1回あたり上限20通）+ 未処理分は次回トリガーで継続 |
| **日次トリガー総実行時間** | 90分/日 | 頻繁なトリガーで上限到達 | 1回30秒以内を目標。夜間はトリガー停止/低頻度化 |
| **URL Fetch日次上限** | 20,000/日（Consumer）、100,000/日（Workspace） | Gemini API呼び出し数に影響 | 日次カウンター管理。Consumer上限では200通/日×(Gmail API+Gemini+Drive)で十分 |
| **PropertiesService容量** | 500KB | 処理済みID蓄積で容量圧迫 | 90日超過IDの定期削除（日次メンテナンス） |
| **トリガー数** | 20個/ユーザー/スクリプト | — | メイン+メンテナンスの2つで十分 |
| **スプレッドシート作成** | 250/日（Consumer） | クライアント追加時のみ | 実運用上問題なし |
| **Blob変換** | 50MB/回 | 巨大HTMLメールの変換失敗 | HTML本文のサイズチェック（5MB上限）→ 超過時は分割 or テキスト抽出 |
| **V8ランタイム** | Rhinoは2026年1月廃止済み | — | V8ランタイムでの開発必須 |

### バッチ処理の実装パターン

```
function processEmails() {
  // 1. 実行開始時刻を記録
  const startTime = Date.now();
  const MAX_EXECUTION_MS = 5 * 60 * 1000; // 5分（1分のマージン確保）
  const BATCH_SIZE = 20;

  // 2. 未処理メール取得（最大BATCH_SIZE件）
  // 3. 各メールを処理（ループ内で経過時間チェック）
  //    - 残り60秒未満 → ループ中断、次回に繰り越し
  // 4. 処理結果をログに記録
}
```

### トリガーチェイニング（大量処理時）

通常の5分間隔トリガーでは処理しきれない場合:
1. 1回の実行で処理しきれなかった場合、残件数をPropertiesServiceに保存
2. 次回トリガーで残件から処理を再開
3. 残件が0になるまで継続

## 12. 段階的実装ロードマップ

### Phase 1: 基本機能（メール取得→スプレッドシート記録）

**目標**: メール情報をスプレッドシートに自動記録する基盤構築

**成果物**:
- マスタースプレッドシート（クライアント設定+実行ログ）
- クライアント別スプレッドシートのテンプレート
- メール取得・クライアント判定・シート記録スクリプト
- 重複処理防止（PropertiesService + ラベル）
- 時間駆動トリガー（5分間隔）
- 基本エラーハンドリング

**前提条件**: なし

**想定工数**: 中（GASスクリプト + スプレッドシート設計）

### Phase 2: PDF変換→Drive保存

**目標**: メールをPDF化してGoogle Driveに自動保存

**成果物**:
- HTML→PDF Blob変換モジュール（インライン画像補完処理含む）
- Drive フォルダ自動作成（年/月サブフォルダ）
- ファイル命名規則の適用
- 添付ファイルの別途保存
- PDFリンクのスプレッドシートへの記録
- HTMLフォールバック処理
- PDF変換品質に問題がある場合、Cloud Functions + Pythonへの切り替えを検討

**前提条件**: Phase 1完了（クライアント判定・フォルダID取得の基盤）

**想定工数**: 中〜大（PDF変換の品質検証に時間を要する可能性）

### Phase 3: Gemini要約生成

**目標**: 各メールの自動要約を生成

**成果物**:
- Gemini API呼び出しモジュール（Google AI Studio API）
- 要約プロンプトの最適化
- 要約結果のスプレッドシート記録
- APIキー管理（Script Properties）
- エラーハンドリング（レート制限、API障害）

**前提条件**: Phase 1完了（スプレッドシート記録基盤）。Phase 2とは独立して並行開発可能

**想定工数**: 小〜中（API呼び出し自体はシンプル。プロンプト調整に時間が必要）

### Phase 4: 管理UI・運用改善

**目標**: 運用・保守を容易にする管理インターフェース

**成果物**:
- マスタースプレッドシートにカスタムメニュー追加（クライアント追加/編集/無効化）
- ダッシュボードシート（処理統計、エラー率、API使用量）
- 手動再処理機能（特定メールの再処理ボタン）
- アラート通知（エラー発生時にメール通知）
- ドキュメント整備（運用マニュアル）

**前提条件**: Phase 1〜3完了

**想定工数**: 中

### Phase間の依存関係

```
Phase 1 ──→ Phase 2 ──→ Phase 4
   │                        ↑
   └──→ Phase 3 ────────────┘
```

Phase 2とPhase 3は独立して並行開発可能。Phase 4は全Phase完了後。
