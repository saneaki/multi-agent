## 1. エグゼクティブサマリー

### 結論

Notion + Google Drive + n8n による法律事務所の資料一元化は **実現可能** である。Gmail・Motionとの自動連携もn8n標準機能で構築でき、弁護士が自力で導入・運用できる現実的な設計が可能。

### 推奨アプローチ

- **Notion**: 案件DB（5テーブル構成）で案件情報・書面・期日・通信履歴・当事者を統合管理
- **Google Drive**: 実ファイルの保管先。案件フォルダ構成と命名規則で整理
- **n8n（セルフホスト）**: Gmail→Notion自動連携、Drive→Notion書面登録を自動化
- **LINE**: 個人LINEはAPI非対応のため手動運用。LINE公式アカウントは将来オプション

### 全体コスト概算

| 構成 | 初期費用 | 月額ランニング |
|------|---------|---------------|
| **最小構成**（Notion+Drive+n8nセルフホスト） | 0円（既存環境活用） | 約3,500〜5,500円/月 |
| **フル構成**（上記+LINE公式+Motion Team） | 0円 | 約15,000〜25,000円/月 |

### 主要リスクと対策

| リスク | 深刻度 | 対策 |
|-------|--------|------|
| Notionデータが米国保管 | 中 | 暗号化+アクセス制限で守秘義務と整合。委任契約書にクラウド利用を明記 |
| 案件紐付け精度の低さ | 中 | 件名ルール策定+送信元マッチング併用+週次レビューで未分類を処理 |
| n8nセルフホストの運用負荷 | 低〜中 | VPS自動バックアップ+暗号化キー厳重管理。困難ならn8n Cloudに移行 |

### データフロー

```
Gmail受信 ──→ n8n Gmail Trigger ──→ 案件紐付け判定 ──→ Notion通信記録DB登録
                                  ├→ 添付ファイル ──→ Google Drive保存 ──→ Notion書面DB登録
                                  └→ 紐付け失敗 ──→ 未分類DB ──→ 週次レビュー

Motion完了 ──→ n8n Schedule Trigger ──→ Motion API取得 ──→ Notion案件マスタ更新

Drive書面 ──→ n8n Drive Trigger ──→ ファイル名パース ──→ Notion書面DB自動登録

LINE受信 ──→ （手動）トーク履歴コピー or メール転送依頼 ──→ Notion通信記録DB手動登録
```

### 各コンポーネントの役割

| コンポーネント | 役割 | データの流れ |
|--------------|------|-------------|
| **Notion** | 情報ハブ。案件メタデータの統合管理・検索・ビュー表示 | 全データの閲覧・検索先 |
| **Google Drive** | 実ファイル保管。書面PDF・証拠資料の原本管理 | Notion書面DBからURLリンクで参照 |
| **n8n** | 自動化基盤。Gmail/Drive/Motionからのデータ取得→Notion登録を自動化 | データソース→Notionへの橋渡し |
| **Gmail** | メール受信。依頼者・相手方との通信の入口 | n8n経由でNotionに自動登録 |
| **Motion** | タスク管理。AIスケジューリングによる期日・作業管理 | n8n経由でNotionに一方向同期 |
| **LINE** | 依頼者連絡。個人LINEは手動運用、公式アカウントは将来導入 | 手動でNotionに記録 |

## 3. Notion案件DB設計

### 3.1 テーブル構成（5テーブル）

#### DB1: 案件マスタ（Cases）

全案件の中核DB。他の全DBがここを参照する。

| プロパティ名 | タイプ | 説明 | 設定値・制約 |
|-------------|--------|------|-------------|
| 案件ID | Title | 一意識別子 | `2026_001` 形式（年_連番） |
| 案件名 | Text | 短い案件名 | 例: `山田太郎_離婚調停` |
| 受任日 | Date | 受任日 | - |
| 終了日 | Date | 終結日 | - |
| 案件種別 | Select | 案件分類 | 離婚・相続・交通事故・債務整理・刑事・その他 |
| ステータス | Status | 進行状況 | 相談中・受任・進行中・終結・アーカイブ |
| 担当弁護士 | Person | 主担当者 | - |
| 書面数 | Rollup | 紐づく書面数 | 書面DB → Count all |
| 次回期日 | Rollup | 最も近い期日 | 期日DB → 日時 → Earliest |
| 最新通信日 | Rollup | 最終通信日 | 通信記録DB → 日時 → Latest |
| 備考 | Text | 補足情報 | 長文可 |
| Drive案件フォルダ | URL | フォルダリンク | Google DriveのURL |

**ビュー**:

- **案件一覧（テーブル）**: ステータス別フィルター、受任日降順ソート
- **進行中案件（ボード）**: ステータスでグルーピング（カンバン方式）
- **今月終結予定（フィルタード）**: 終了日が今月の案件のみ

#### DB2: 当事者（Parties）

依頼者・相手方・相手方代理人など案件関係者を一元管理。

| プロパティ名 | タイプ | 説明 | 設定値・制約 |
|-------------|--------|------|-------------|
| 当事者名 | Title | フルネーム | - |
| 案件 | Relation | 紐づく案件 | → 案件マスタDB |
| 続柄/立場 | Select | 関係性 | 依頼者・相手方・相手方代理人・証人・その他 |
| 電話番号 | Phone | 連絡先 | - |
| メールアドレス | Email | 連絡先 | - |
| 住所 | Text | 郵送先 | - |
| 備考 | Text | 補足情報 | 家族構成・注意事項等 |

**ビュー**: 依頼者一覧、相手方代理人一覧（続柄でフィルター）

#### DB3: 書面（Documents）

訴状・準備書面・証拠（甲乙号証）など、すべての書面を管理。Google Driveと連携。

| プロパティ名 | タイプ | 説明 | 設定値・制約 |
|-------------|--------|------|-------------|
| 書面名 | Title | 書面の名称 | 例: `原告第一準備書面` |
| 案件 | Relation | 紐づく案件 | → 案件マスタDB |
| 種別 | Select | 書面分類 | 訴状・答弁書・準備書面・甲号証・乙号証・陳述書・裁判所文書・その他 |
| 証拠番号 | Text | 証拠番号 | 例: `甲1`, `乙3-2` |
| 提出日 | Date | 提出日 | 空欄可 |
| 受領日 | Date | 受領日 | 空欄可 |
| Driveリンク | URL | ファイル直リンク | Google DriveのURL |
| 備考 | Text | 補足情報 | - |

**ビュー**: 書面一覧（提出日降順）、案件別書面（グループ）、証拠一覧（甲/乙フィルター+番号ソート）、タイムライン

#### DB4: 期日（Hearings）

口頭弁論・調停・打合せなど、すべての期日をカレンダー管理。

| プロパティ名 | タイプ | 説明 | 設定値・制約 |
|-------------|--------|------|-------------|
| 期日名 | Title | 期日の名称 | 例: `第3回口頭弁論` |
| 案件 | Relation | 紐づく案件 | → 案件マスタDB |
| 期日種別 | Select | 期日分類 | 口頭弁論・弁論準備・調停・示談交渉・打合せ・その他 |
| 日時 | Date | 期日日時 | 時刻含む |
| 場所 | Text | 裁判所名等 | 例: `東京地裁401号法廷` |
| 備考 | Text | 補足情報 | 持参書類・注意事項 |
| 完了 | Checkbox | 終了フラグ | - |

**ビュー**: 期日カレンダー、今月の期日（未完了フィルター）、案件別期日（グループ）

#### DB5: 通信記録（Communications）

メール・電話・LINEなど、すべての対外通信履歴を記録。

| プロパティ名 | タイプ | 説明 | 設定値・制約 |
|-------------|--------|------|-------------|
| 通信タイトル | Title | 通信の件名 | 例: `山田様からの電話相談` |
| 案件 | Relation | 紐づく案件 | → 案件マスタDB |
| 相手方 | Relation | 通信相手 | → 当事者DB |
| 通信種別 | Select | 通信手段 | メール・電話・LINE・FAX・書面受領・その他 |
| 日時 | Date | 通信日時 | - |
| 要旨 | Text | 概要メモ | 数行で要点をまとめる |
| 原文リンク | URL | 元データへのリンク | Notionページ/Drive/GmailスレッドURL |
| 要返信 | Checkbox | 返信必要フラグ | 未対応タスクの可視化 |

**ビュー**: 通信履歴（日時降順）、未処理通信（要返信フィルター）、案件別通信（グループ）

### 3.3 テンプレート

#### 新規案件登録テンプレート

受任時にNotionのテンプレートボタンで以下を一括作成:

1. 案件マスタ: 案件ID（年_連番）、案件名、受任日、案件種別、担当弁護士
2. 当事者DB: 依頼者の基本情報（氏名・連絡先）
3. 期日DB: 初回打合せ日時
4. 通信記録DB: 相談受付時の記録

### 3.4 Notionでの作成手順

1. 新規ページ「案件管理システム」を作成
2. `/database` で5つのテーブル型DBを作成（案件マスタ、当事者、書面、期日、通信記録）
3. 各DBのプロパティを上記テーブル定義に従って設定
4. リレーション設定: 当事者→案件、書面→案件、期日→案件、通信記録→案件+当事者
5. ロールアップ設定: 案件マスタに書面数・次回期日・最新通信日
6. ビュー作成: フィルター・ソート・グルーピングを設定

## 4. Google Drive書面管理

### 4.1 フォルダ構成

```
法律事務所/
├── 案件/
│   ├── 2026_001_山田太郎_離婚/
│   │   ├── 01_甲号証/
│   │   ├── 02_乙号証/
│   │   ├── 03_準備書面/
│   │   ├── 04_訴状_答弁書/
│   │   ├── 05_陳述書/
│   │   ├── 06_裁判所文書/
│   │   ├── 07_通信記録/
│   │   └── 08_その他/
│   ├── 2026_002_佐藤花子_相続/
│   │   └── ... (同様の8フォルダ構成)
│   └── ...
├── テンプレート/
│   ├── 委任契約書_テンプレート.docx
│   ├── 新規案件チェックリスト.md
│   ├── 準備書面_ひな形.docx
│   └── 証拠説明書_ひな形.xlsx
└── 共通/
    ├── 事務所パンフレット.pdf
    ├── 弁護士会規則_参照用/
    └── 裁判所書式集/
```

**設計ポイント**:

- フォルダ名は Notion案件ID と完全一致（`2026_001_山田太郎_離婚`）
- サブフォルダは番号付き（01〜08）で並び順を固定
- テンプレートフォルダで案件フォルダの一括作成が可能

### 4.2 ファイル命名規則

**基本形式**: `YYYYMMDD_種別番号_書面名.拡張子`

| 書面種別 | ファイル名例 |
|---------|-------------|
| 甲号証 | `20260212_甲1_売買契約書.pdf` |
| 甲号証（枝番） | `20260212_甲1-2_契約書別紙.pdf` |
| 乙号証 | `20260215_乙5_診断書.pdf` |
| 準備書面 | `20260220_準備書面1_原告第一準備書面.pdf` |
| 訴状 | `20260210_訴状_損害賠償請求.pdf` |
| 答弁書 | `20260218_答弁書.pdf` |
| 陳述書 | `20260225_陳述書_依頼者.pdf` |
| 裁判所文書 | `20260301_第2回期日調書.pdf` |

**ルール**:

- 日付プレフィックスで時系列ソートが容易
- 証拠番号はハイフンで枝番を表現（`甲1-2`）
- ファイル形式はPDF原則（Word`.docx`、Excel`.xlsx`も可）

### 4.3 Notion ↔ Drive 連携

#### 手動連携（Phase 1で運用）

1. Google Driveの案件フォルダに書面をアップロード
2. ファイルを右クリック →「リンクを取得」→ URLコピー
3. Notion書面DBに新規ページ作成 → Driveリンク/書面名/種別/証拠番号を入力

#### 自動連携（Phase 2でn8n構築）

```
[Google Drive Trigger: File Created]
    ↓
[Function Node: ファイル名パース]
    ↓ YYYYMMDD_種別番号_書面名 → 種別・証拠番号を抽出
[Notion Node: 書面DB作成]
    ↓ 書面名・種別・証拠番号・Driveリンクを登録
[通知] (optional)
```

### 4.4 運用ルール

| 項目 | 運用 |
|------|------|
| **フォルダ権限** | 事務所メンバーに編集権限。案件フォルダ配下は外部共有を無効化 |
| **バックアップ** | Google Takeoutで月1回。重要案件は終結時にローカルHDDにも保存 |
| **アーカイブ** | 終結から3年経過 → `案件_アーカイブ/` に移動、Notionステータスも「アーカイブ」に変更 |

## 5. n8n自動化ワークフロー設計

### 5.1 Gmail → Notion 連携ワークフロー

**最優先で構築するワークフロー。**

#### ワークフロー構成

```
[Gmail Trigger: ポーリング5分間隔、ラベルフィルタ]
    ↓
[IF: 添付ファイル有無判定]
    ├─ YES → [Google Drive: Upload File] → [DriveリンクURL取得]
    └─ NO → [空文字設定]
    ↓
[IF: 件名パターンマッチ（【案件名】プレフィックス）]
    ├─ Match → [案件名抽出]
    └─ No Match → [案件名 = "未分類"]
    ↓
[Notion: Database Item Create]
    ↓
[Stop]
```

#### Gmail Trigger 設定

- ポーリング間隔: 5分
- ラベルフィルタ: `law-firm/inbox`（任意）
- 添付ファイル自動ダウンロード: 有効

#### 案件紐付けロジック（優先順位）

1. **件名パターンマッチ**: `【案件名】` プレフィックスを正規表現で抽出
2. **送信元アドレス紐付け**: Notion当事者マスタのEmailで検索 → 案件ID取得
3. **Gmailラベル活用**: `law-firm/case-A` → スラッシュ以降を案件名として抽出
4. **紐付け失敗**: 未分類として登録 → 週次レビューで手動振り分け

```javascript
// 件名パターンマッチの実装例（n8n Code Node）
const subject = $json.subject || '';
const match = subject.match(/^【(.+?)】/);
return { caseName: match ? match[1] : '未分類' };
```

#### 添付ファイル処理

```
[Gmail Trigger: 添付ファイルをバイナリデータで取得]
    ↓
[Loop: 各添付ファイル（Split In Batches）]
    ↓
[Code Node: ファイル名にタイムスタンプ追加]
    例: "契約書.pdf" → "契約書_20260212_1030.pdf"
    ↓
[Google Drive: Upload File]
    - Parents: {{$env.GDRIVE_LAW_FOLDER_ID}}
    ↓
[Notion: Create Database Item]
    - 添付ファイルリンク: DriveのURL
```

**ファイルサイズ別対応**:

| サイズ | 処理方式 |
|-------|---------|
| < 10MB | デフォルト（メモリ内処理） |
| 10-50MB | Filesystemモード |
| 50-200MB | タイムアウト延長（600秒） |
| > 200MB | S3モード推奨（法律事務所では稀） |

#### 認証設定

**Gmail OAuth2**:

1. Google Cloud Console → プロジェクト作成 → Gmail API有効化
2. OAuth 2.0クライアントID作成（Webアプリ）
3. リダイレクトURI: `https://<n8n-instance>/rest/oauth2-credential/callback`
4. Scope: `https://www.googleapis.com/auth/gmail.readonly`（読取専用推奨）

**Notion Internal Integration**:

1. [Notion Integrations](https://www.notion.com/integrations) → New integration
2. Capabilities: Read/Update/Insert content を有効化
3. Integration Secret を即座にコピー（一度のみ表示）
4. 対象DB各ページで「共有」→ 作成したIntegrationを招待

**重要**: n8nのNotionノードはInternal Integration Tokenのみサポート（OAuth2非対応）。

#### エラーハンドリング

```
[Error Trigger]
    ↓
[IF: エラータイプ判定]
    ├─ "紐付け失敗" → [Notion: 未分類DBに追加] → [通知]
    ├─ "API Rate Limit" → [Wait] → [リトライ]
    └─ "その他" → [通知]
```

- Retry on Fail: ON / Max Tries: 3 / Exponential Backoff: ON
- Notion API Rate Limit: 3req/sec → Wait Node（300ms）挿入で制御

### 5.2 Motion → Notion 同期ワークフロー

**Motion APIが正式提供されているため、直接連携が可能。**

#### 推奨: 一方向同期（Motion → Notion）

双方向同期は競合リスク・複雑性が高いため、一方向を強く推奨。

- Motion側: AIスケジューリングでタスク管理を完結
- Notion側: 案件管理・資料紐付けを完結
- 同期: Motionの完了タスクや更新をNotionに反映

#### ワークフロー構成

```
[Schedule Trigger: 5分ごと]
    ↓
[HTTP Request: GET https://api.usemotion.com/v1/tasks]
    - Header: X-API-Key = {{$env.MOTION_API_KEY}}
    - Query: status=COMPLETED&modifiedAfter={{lastSyncTime}}
    ↓
[Loop: 各タスクを処理]
    ↓
[Notion: Query Database (Motion Task IDで検索)]
    ↓
[IF: 既存レコードあり？]
    ├─ YES → [Notion: Update Database Item]
    └─ NO → [Notion: Create Database Item]
    ↓
[Code Node: lastSyncTime更新]
```

#### データマッピング

| Motion フィールド | Notion プロパティ | 変換 |
|------------------|-----------------|------|
| `name` | タイトル（Title） | そのまま |
| `dueDate` | 期日（Date） | ISO 8601形式 |
| `status` | ステータス（Select） | "COMPLETED" → "完了" |
| `assignees[0].name` | 担当者（Person） | ユーザーマッピング |
| `project.name` | プロジェクト（Relation） | 案件DBと紐付け |
| `id` | Motion Task ID（Text） | 同期管理用 |

#### Motion APIの制約

| プラン | レート制限 | 推奨 |
|--------|----------|------|
| Individual | 12 req/min | 同期間隔10分以上に設定 |
| **Team（推奨）** | 120 req/min | 5分間隔で安定運用可能 |

### 5.3 Drive → Notion 書面自動登録ワークフロー

```
[Google Drive Trigger: File Created in 案件フォルダ]
    ↓
[Function Node: ファイル名パース]
    - YYYYMMDD_種別番号_書面名 を正規表現で分解
```

## 7. セキュリティ・コンプライアンス

### 7.1 弁護士情報セキュリティ規程（2024年6月施行）の要件

旧「弁護士情報セキュリティガイドライン」は廃止され、**規程**（拘束力あり）に格上げ。

| 要件 | 本設計での充足方法 |
|------|-------------------|
| 適切なセキュリティ措置 | 全サービスで暗号化（TLS+AES-256）、2FA有効化 |

| データ保管場所の考慮 | n8nは日本VPSでセルフホスト。Notionは米国保管だが暗号化で対応 |

### 7.2 守秘義務（弁護士法23条）との整合性

- **結論**: 暗号化・アクセス制限・2FA等の措置を講じれば、クラウド利用は守秘義務に違反しない
- **対応策**: 委任契約書に「セキュリティ対策を講じたクラウドサービスを利用する旨」を明記し、依頼者同意を取得
- **例文**: 「当事務所は、セキュリティ対策を講じたクラウドサービス（Notion、Google Workspace等）を利用して事件情報を管理します。」

### 7.3 各サービスのセキュリティ設定チェックリスト

#### Notion

- [ ] 2要素認証（2FA）を有効化
- [ ] ゲストアクセスを制限（内部メンバーのみ編集権限）
- [ ] Integration（API）の権限を最小限に設定（Read/Update/Insertのみ）
- [ ] ビジネスプラン以上で監査ログ（90日間）を有効化
- [ ] 転送時暗号化: TLS 1.2以降（自動）
- [ ] 保存時暗号化: AES-256（AWS S3 SSE-S3、自動）
- [ ] **注意**: データ保管は米国（AWS US East）。日本リージョン未対応

#### Google Drive (Workspace)

- [ ] 外部共有を「組織内のみ」に制限
- [ ] リンク共有を「組織内のユーザーのみ閲覧可能」に設定
- [ ] 案件フォルダの共有ドライブ化（個人ドライブより安全）

### 7.4 リスク軽減策一覧

| 対策 | 内容 |
|------|------|
| **暗号化** | 転送時（TLS）と保存時（AES-256）の暗号化を全サービスで確認 |
