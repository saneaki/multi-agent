## 13. /learn — セッションから再利用パターンを抽出

### (1) 概要

Claude Codeで作業した内容の中から、将来また使える「知恵」を抽出して保存するツールです。作業中に解決した問題、工夫した方法、発見したコツ等を「スキル」として記録し、次回同じ状況になったときにすぐ使えるようにします。

### (2) 内部ロジック

1. **セッションレビュー**: 現在のClaude Code作業履歴を分析します
2. **抽出対象の判別**: 以下の4種類を探します
    - **エラー解決パターン**: どんなエラーが出て、何が原因で、どう直したか
    - **デバッグ技法**: 普通は気づかないトラブル調査の手順やツールの組み合わせ
    - **回避策**: ライブラリ（外部ソフトウェア部品）の制限やバージョン固有の問題への対処法
    - **プロジェクト固有のパターン**: このプロジェクト特有の書き方や設計判断
3. **スキルファイル作成**: `~/.claude/skills/learned/[パターン名].md` という形式でファイルに保存します
4. **ユーザー確認**: 保存する前にあなたに内容を確認してもらいます

スキルファイルには「問題」「解決策」「例」「いつ使うか」の4項目が記録されます。

### (3) 使い方

チャットで `/learn` と入力するだけです。引数（追加の指定）は不要です。Claude Codeが自動的に現在のセッションを分析し、保存すべきパターンを提案します。

### (4) 実践例：弁護士業務での活用

**シーン**: 法律文書PDFをOCR（文字認識）で処理するとき、特定の裁判所フォーマットで毎回同じエラーが出る

**使い方**:
1. エラーを解決した後、チャットで `/learn` を実行
2. Claude Codeが「〇〇地裁の判決書フォーマットでOCR失敗時の回避策」を抽出提案
3. 確認してOKなら保存
4. 次回同じ裁判所の書面を処理するとき、Claude Codeが自動でこのスキルを参照し、エラーを避ける方法をすぐに適用

別の例:
- **契約書テンプレート自動生成**の際、Word差し込み印刷での特定条項の扱い方
- **案件管理スプレッドシート**のマクロで遭遇したエラーと解決法
- **クライアント情報検索システム**で見つけた効率的なデータ抽出パターン

### (5) Tips/注意点

- **些細な修正は対象外**: 単なる誤字脱字の修正や簡単な構文エラーは抽出しません。将来の時間節約になるパターンのみが対象です。
- **一度きりの問題は対象外**: たまたまその日だけサーバーが落ちていた、という類の問題は保存しません。再現性のあるパターンのみです。
- **1スキル1パターン**: 1つのスキルファイルには1つの明確なパターンだけを記録します。複数の問題を混ぜると後で使いにくくなります。
- **技術的な内容が苦手でも安心**: スキル抽出はClaude Codeが自動で行います。あなたは「この知恵は保存しておきたい」と思ったタイミングで `/learn` を実行するだけです。

## 14. /orchestrate — 複数エージェントの連携ワークフロー

### (1) 概要

複雑なタスク（機能追加、バグ修正、設計変更等）を遂行するとき、複数の専門エージェント（AIアシスタント）を順番に呼び出して作業させるツールです。「設計担当→テスト担当→品質確認担当→セキュリティ確認担当」のように、プロジェクトチームの流れ作業を自動化します。

### (2) 内部ロジック

1. **ワークフロータイプの判別**: 以下4種類の定義済みワークフローから選択、またはカスタム（自由に組み合わせ）も可能
    - **feature（機能追加）**: planner（設計） → tdd-guide（テスト駆動開発） → code-reviewer（品質確認） → security-reviewer（セキュリティ確認）
    - **bugfix（バグ修正）**: explorer（調査） → tdd-guide（テスト＋修正） → code-reviewer（確認）
    - **refactor（設計改善）**: architect（アーキテクト、設計専門家） → code-reviewer（確認） → tdd-guide（テスト）
    - **security（セキュリティ重視）**: security-reviewer → code-reviewer → architect
2. **順次実行**: 各エージェントを順番に起動します
3. **引き継ぎ文書作成**: あるエージェントが終わったら「何をしたか」「発見事項」「修正ファイル」「次のエージェントへの質問」を文書化します
4. **次のエージェントへ渡す**: 前のエージェントの成果物を受け取り、次の作業を開始します
5. **最終レポート集約**: 全員の作業結果を統合したレポートを作成します

並列実行（同時に複数エージェントを動かす）もサポートしています。品質確認・セキュリティ確認・設計レビューは独立しているため、同時実行して時間を節約できます。

### (3) 使い方

```
/orchestrate [workflow-type] [タスク説明]
```

例:
- `/orchestrate feature "ユーザー認証機能を追加"`
- `/orchestrate bugfix "ログイン画面で日本語入力が効かない"`
- `/orchestrate refactor "データベース接続部分を整理"`
- `/orchestrate security "個人情報保護の監査"`
- `/orchestrate custom "architect,tdd-guide,code-reviewer" "キャッシュ層の再設計"`

### (4) 実践例：弁護士業務での活用

**シーン**: 契約書テンプレート自動生成システムに「条項カスタマイズ機能」を追加する

**使い方**:
```
/orchestrate feature "契約書テンプレートに条項カスタマイズ機能を追加"
```

**実行の流れ**:
1. **Planner（設計担当）**: 条項カスタマイズ機能の設計を立てる。どのファイルをどう変更するか、依存関係は何かを整理
2. **TDD Guide（テスト駆動開発担当）**: 先にテストコード（動作確認プログラム）を書き、それを満たす実装を追加
3. **Code Reviewer（品質確認担当）**: コードの可読性、命名、構造をレビュー
4. **Security Reviewer（セキュリティ担当）**: 契約書データに機密情報が含まれるため、漏洩リスク・アクセス権限・暗号化の確認

**最終レポート例**:
```
ワークフロー: feature
タスク: 契約書テンプレートに条項カスタマイズ機能を追加
エージェント: planner → tdd-guide → code-reviewer → security-reviewer

概要: 条項カスタマイズ機能を追加。条項マスタテーブル新設、テンプレート生成エンジン改修。

変更ファイル: template_engine.py, clause_master.py, test_clause.py
テスト結果: 全テスト通過（28/28）
セキュリティ: 機密情報の暗号化確認済み、アクセスログ記録追加

推奨: SHIP（リリース可能）
```

### (5) Tips/注意点

- **featureワークフローは必ずplannerから開始**: 複雑な機能追加では、設計をスキップすると後で手戻りが発生します
- **常にcode-reviewerを含める**: マージ（本番反映）前の品質チェックは必須です
- **認証・決済・個人情報を扱う場合は必ずsecurity-reviewerを含める**: 弁護士業務では依頼者情報、事件記録、契約書等の機密情報を扱うため、セキュリティ確認は省略厳禁です
- **引き継ぎ文書は簡潔に**: 次のエージェントが必要とする情報だけを渡します。長すぎると逆に読まれません
- **並列実行で時間短縮**: 独立した確認作業（品質・セキュリティ・設計）は同時実行できます

## 15. /sessions — セッション履歴の管理

### (1) 概要

Claude Codeで過去に行った作業セッション（会話履歴）を一覧表示したり、読み込んだり、覚えやすい別名（エイリアス）を付けたりするツールです。「あの時どうやって解決したっけ？」を素早く振り返ることができます。

### (2) 内部ロジック

セッション履歴は `~/.claude/sessions/` というフォルダに1セッション1ファイル（Markdown形式）で保存されています。このツールはNode.js（JavaScriptの実行環境）のスクリプトを使って以下の操作を行います：

1. **list（一覧表示）**: 全セッションファイルを日付順で表示。フィルタ（日付指定、キーワード検索）や件数制限も可能
2. **load（読み込み）**: 特定のセッションIDまたは別名を指定して、その内容を表示
3. **alias（別名付け）**: セッションに覚えやすい名前を付ける。例: `2026-02-01-abc123def456` → `today-work`
4. **info（詳細情報）**: セッションの統計情報（行数、タスク数、完了数、ファイルサイズ等）を表示
5. **aliases（別名一覧）**: 登録済みの別名を全て表示

別名は `~/.claude/session-aliases.json` というファイルに保存されます。

### (3) 使い方

```
/sessions [サブコマンド] [オプション]
```

**一覧表示**:
```
/sessions                           # 全セッション表示（デフォルト）
/sessions list                      # 同上
/sessions list --limit 10           # 最新10件のみ
/sessions list --date 2026-02-01    # 2026年2月1日のセッションのみ
/sessions list --search abc         # IDに "abc" を含むセッションのみ
```

**セッション読み込み**:
```
/sessions load 2026-02-01           # 日付で読み込み
/sessions load a1b2c3d4             # セッションIDで読み込み（短縮形可）
/sessions load my-alias             # 別名で読み込み
```

**別名作成**:
```
/sessions alias 2026-02-01 today-work   # "today-work" という別名を付ける
```

**別名削除**:
```
/sessions alias --remove today-work
/sessions unalias today-work           # 同上
```

**セッション情報**:
```
/sessions info today-work           # 詳細情報表示
```

**別名一覧**:
```
/sessions aliases                   # 登録済み別名を全て表示
```

### (4) 実践例：弁護士業務での活用

**シーン**: 過去に類似案件で作成した「契約書雛形改訂」作業を参照したい

**使い方**:
1. **別名を付ける**: 契約書雛形改訂作業をしたときに、セッション終了前に別名を付けておく
   ```
   /sessions alias 2026-01-15 contract-template-revision
   ```
2. **後日、類似案件が来たとき**: 別名で過去のセッションをすぐ読み込む
   ```
   /sessions load contract-template-revision
   ```
3. **過去の判断を確認**: どの条項をどう修正したか、どんな問題があったか、どう解決したかを即座に振り返る

別の例:
- **顧問先ごとにセッションを分類**: `client-A-consultation`, `client-B-contract-review` 等の別名を付ける
- **よく使うツールの設定手順を記録**: `pdf-merger-setup`, `ocr-tool-config` 等の別名で、セットアップ手順をいつでも参照できる
- **月次定型業務**: `monthly-report-jan`, `monthly-report-feb` のように月ごとに別名を付け、前月の作業を参照しながら処理

### (5) Tips/注意点

- **セッションIDは短縮可能**: 完全なIDは長いですが、最初の4〜8文字だけで十分識別できます（例: `a1b2c3d4` で `a1b2c3d4ef5678ab90cd1234...` を指定）
- **よく参照するセッションには必ず別名を付ける**: 長いIDを毎回入力するのは非効率です
- **別名は意味が分かる名前に**: `session1`, `work2` のような名前ではなく、`contract-review-202601`, `template-automation` のように内容が分かる名前にしましょう
- **別名ファイルはバックアップ推奨**: `~/.claude/session-aliases.json` を定期的にバックアップすると、PCを変更しても別名を引き継げます

## 16. /checkpoint — 作業の節目で状態を記録

### (1) 概要

プロジェクトの途中経過を「チェックポイント」として記録し、後で「あの時と比べてどう変わったか」を検証できるツールです。機能追加やリファクタリング（設計改善）の前後で、変更範囲やテスト結果の差分を確認できます。

### (2) 内部ロジック

**チェックポイント作成（create）**:
1. **現在の状態を検証**: `/verify quick` コマンドを実行し、ビルドエラーやテスト失敗がないことを確認
2. **Gitでスナップショット作成**: git stash（一時退避）またはコミット（変更記録）で現在の状態を保存
3. **ログに記録**: `.claude/checkpoints.log` というファイルに以下を記録
   ```
   2026-02-13-21:00 | feature-start | a1b2c3d
   ```
   （日時 | チェックポイント名 | Gitコミットハッシュ（バージョン識別子））
4. **完了報告**: チェックポイント作成完了を通知

**チェックポイント検証（verify）**:
1. **ログからチェックポイントを読み込む**: 指定した名前のチェックポイントを探す
2. **現在との差分を比較**:
    - 追加されたファイル数
    - 変更されたファイル数
    - テストの成功・失敗数の変化
    - カバレッジ（テストがコードの何%をカバーしているか）の変化
3. **比較レポート作成**:
   ```
   チェックポイント比較: feature-start
   ============================
   変更ファイル: 12
   テスト: +5 成功 / -2 失敗
   カバレッジ: +3.5%
   ビルド: [PASS]
   ```

**チェックポイント一覧（list）**:
- 全チェックポイントを名前・日時・Gitバージョン・状態（現在地/過去/未来）で表示

### (3) 使い方

```
/checkpoint [create|verify|list] [名前]
```

**作成**:
```
/checkpoint create "feature-start"      # 機能追加開始時点を記録
/checkpoint create "core-done"          # コア機能完成時点を記録
```

**検証**:
```
/checkpoint verify "feature-start"      # 機能追加開始時点との差分を確認
/checkpoint verify "core-done"          # コア完成時点との差分を確認
```

**一覧**:
```
/checkpoint list                        # 全チェックポイントを表示
```

**古いチェックポイント削除**:
```
/checkpoint clear                       # 最新5個を残して古いものを削除
```

### (4) 実践例：弁護士業務での活用

**シーン**: 法律文書管理システムの大規模改修

**使い方**:
1. **改修開始前**: `/checkpoint create "before-refactor"` で現状を記録
2. **コア機能完成後**: `/checkpoint create "core-done"` で中間地点を記録
3. **リファクタリング完了後**: `/checkpoint create "refactor-done"` で完成時点を記録
4. **PR（プルリクエスト、レビュー依頼）前**: `/checkpoint verify "before-refactor"` で全体の変更量を確認
   ```
   チェックポイント比較: before-refactor
   ============================
   変更ファイル: 28
   テスト: +15 成功 / 0 失敗
   カバレッジ: +8.2%
   ビルド: PASS
   ```
5. **安心してリリース判断**: 開始時点と比べてテストが増え、カバレッジが上がり、ビルドも通っているので品質が担保されている

別の例:
- **契約書テンプレート自動生成機能追加**: `template-before`, `template-core`, `template-complete` と3段階でチェックポイントを作り、各段階での変更範囲を把握
- **データベース移行**: `db-before-migration`, `db-after-migration` でデータ移行前後の差分を記録

### (5) Tips/注意点

- **feature開始時には必ずcreate**: 後で「最初はどうだったっけ？」となるのを防ぎます
- **PR前には必ずverify**: 全体でどれだけ変わったかを把握し、レビュアー（確認者）に説明する材料にします
- **名前は段階が分かるように**: `checkpoint1`, `checkpoint2` ではなく、`feature-start`, `core-done`, `refactor-done` のように作業段階が分かる名前を付けます
- **古いチェックポイントは定期的に削除**: `/checkpoint clear` で最新5個だけ残し、古いものは削除できます

## 17. /pull-build — リモートから更新を取得して自動ビルド

### (1) 概要

Gitリポジトリ（プロジェクトファイルの保管庫）のリモート（他のPCや共有サーバー）から最新の変更を取得（pull）し、変更があった場合に自動でビルド（実行可能ファイルの生成）を行うツールです。チーム開発や複数PCで作業する場合に便利です。

### (2) 内部ロジック

1. **現在の状態確認**: `git status` でローカル（手元のPC）に未コミット（保存していない）の変更があるかチェック
    - 変更がある場合: あなたに確認（退避するか、コミットするか、破棄するか）
2. **プル実行**: `git pull` でリモートから変更を取得
    - 出力メッセージを解析:
        - "Already up to date" → 変更なし（ビルドスキップ）
        - "Fast-forward" またはファイル変更あり → 更新あり（ビルド対象を判定）
3. **変更ファイルの判定**: 以下のファイルが変更されていればビルド実行
    - `.py` ファイル（Pythonソースコード）
    - `.spec` ファイル（PyInstallerビルド設定）
    - `requirements.txt`（依存ライブラリリスト）
4. **ビルド実行**（更新があった場合のみ）: `python -m PyInstaller <project>.spec` を実行
    - specファイルはプロジェクトルートから自動検出
5. **結果サマリー**:
    - プル結果（更新あり/なし）
    - ビルド結果（成功/失敗/スキップ）
    - 出力ファイルのパス

**ビルドをスキップする変更**:
- ドキュメントのみ（`.md`, `.txt`）
- 設定ファイルのみ（`.json`, `.yaml`）

### (3) 使い方

```
/pull-build
```

引数は不要です。このコマンド1つで「プル→変更判定→必要ならビルド」の流れが全自動で実行されます。

### (4) 実践例：弁護士業務での活用

**シーン**: PDF結合ツールを事務所の複数PCで共有開発

**状況**:
- リポジトリはGitHub等のリモートサーバーに置いている
- PC-A（自宅PC）とPC-B（事務所PC）の両方で開発
- PC-Aで修正した内容をPC-Bでもすぐ使いたい

**使い方**:
1. **PC-Aで修正**:
   - PDF結合ツールのバグを修正
   - `git commit` → `git push` でリモートに反映
2. **PC-Bで自動取得＋ビルド**:
   ```
   /pull-build
   ```
   - リモートから変更を自動取得
   - .pyファイルが変更されているのでビルド実行
   - 最新版の実行ファイル（.exe）が生成される
3. **すぐに使える**: 事務所PCでも修正済みバージョンがすぐに利用可能

別の例:
- **法律文書OCRツール**: チームメンバーが改良した機能を自動取得＋ビルドして全員が最新版を使用
- **契約書テンプレート生成ツール**: テンプレート追加や修正をリモートから取得し、自動でビルドして配布用exeファイルを更新

### (5) Tips/注意点

- **ドキュメントのみ変更ならビルドスキップ**: READMEやコメントの修正だけならビルド不要なので、自動でスキップされます
- **ソースコード変更時のみビルド実行**: `.py`, `.spec`, `requirements.txt` 等の実行に影響するファイルが変更されたときだけビルドします
- **未コミット変更がある場合は確認**: 手元で作業中の内容がある場合、プル前にあなたに確認が入ります。退避（stash）するか、コミットするか、破棄するかを選択してください
- **ビルドエラーが出たら**: エラーメッセージを確認し、依存ライブラリが不足している場合は `pip install -r requirements.txt` を実行してください
- **PyInstaller専用**: 現在はPyInstallerでのPythonプロジェクトビルドに対応しています。他の言語・ビルドツールには未対応です

## まとめ

この5つのコマンドは、Claude Codeの「記録」「連携」「履歴管理」「状態保存」「自動化」機能を担当しています。

| コマンド | 主な用途 | 弁護士業務での活用場面 |
|---------|---------|----------------------|
| /learn | 知恵の蓄積 | 法律文書処理のノウハウ、エラー対処法の記録 |
| /orchestrate | 複数エージェント連携 | 機能追加時の設計→実装→品質→セキュリティの流れを自動化 |
| /sessions | 過去作業の参照 | 類似案件の過去処理方法を即座に振り返り |
| /checkpoint | 作業途中経過の記録 | 大規模改修の段階ごとの変更量・品質を記録 |
| /pull-build | リモート同期＋自動ビルド | チーム開発での最新版自動取得・配布用exe自動生成 |

これらのコマンドを使いこなすことで、Claude Codeを単なるコーディング支援ツールから「業務ナレッジ（知識）管理システム」「チーム開発自動化ツール」へと進化させることができます。

技術的な詳細が分からなくても、**「いつ使うか」**を覚えておけば、Claude Codeが適切なタイミングで提案してくれます。
