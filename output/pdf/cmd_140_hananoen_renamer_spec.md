# hananoen PDFリネーマー 現状仕様レポート

**作成日**: 2026-02-14
**cmd_140**: hananoenプロジェクト PDFリネーマー機能 現状仕様調査
**目的**: テンプレート改善方針の判断材料を殿に提供する

---

## 1. エグゼクティブサマリー

hananoenプロジェクトのPDFリネーマーは、**汎用リネーマー**（file_renamer.py）と**法律事務所向けリネーマー**（legal_renamer.py）の2層構造で構成される。GUIはpdf_merger_tabbed.pyの「ファイル名変更」タブ（Tab 4）として統合されている。

### 全体像

| 構成要素 | 役割 | クラス数 | テスト数 |
|---------|------|---------|---------|
| file_renamer.py | 汎用リネーム操作（プレフィックス、サフィックス、連番、検索置換、Undo） | 1クラス（14メソッド） | 12ケース |
| legal_renamer.py | 法律事務所向けテンプレート（6フォルダ種別・14サブ種別）、日付処理、証拠番号 | 3クラス | 43ケース |
| pdf_merger_tabbed.py | GUI統合（2カラムレイアウト、動的フィールド生成） | メインアプリの一部 | — |
| help_system.py | ヘルプテキスト・ToolTip管理 | ヘルプシステムの一部 | — |

### 主要特徴

- **テンプレート変数**: file_renamer.pyには**未実装**。legal_renamer.pyでは `{提出日}_{受信者}へ_{文書名}` 形式のプレースホルダーを使用
- **日付処理**: YYYYMMDD形式（例: 20260214）、今日の日付を自動入力、バリデーション（閏年対応）
- **枝番自動付与**: 同名ファイル衝突時に `_2`, `_3` を自動付与（欠番も埋める）
- **証拠番号**: 甲/乙の2桁ゼロ埋め連番（甲01, 甲02...）、枝番は手動運用
- **Undo**: 直前1回分のリネームを取り消し可能（JSON形式のバックアップ、最新5件保持）

### 発見された課題

1. **file_renamer.pyにテンプレート変数機能がない** — プレースホルダー展開は legal_renamer.py のみ
2. **バックアップディレクトリ名が `pdf_merger_tool`** — プロジェクト名と不一致
3. **証拠番号の枝番が手動** — 甲01-1 等の枝番は自動採番できない
4. **テンプレート定義がハードコード** — 6フォルダ種別が Python コードに直接記述
5. **file_renamer.pyのテストカバレッジ不足** — rename_files(), Undo, エラーハンドリングのテストなし

## 2. file_renamer.py 仕様

### 2.1 FileRenamerクラス メソッド一覧

| カテゴリ | メソッド | 役割 |
|---------|---------|------|
| 初期化 | `__init__()` | folder_path, files, rename_mapを空で初期化 |
| ファイル取得 | `get_files_in_folder(folder_path)` | フォルダ直下のファイル一覧取得（サブフォルダ・隠しファイル・システムファイル除外） |
| 検証 | `validate_filename(filename)` | ファイル名の妥当性チェック（無効文字、予約語、長さ等） |
| 検証 | `check_duplicate(new_name, extension, exclude_original)` | 同名ファイルの存在チェック |
| リネーム実行 | `rename_files(rename_list, progress_callback)` | ファイル名の一括変更（RenameResult返却） |
| バックアップ | `_get_undo_backup_dir()` | Undo用バックアップディレクトリ取得 |
| バックアップ | `_cleanup_old_backups(backup_dir, keep=5)` | 古いバックアップの自動クリーンアップ（最新5件保持） |
| バックアップ | `create_undo_backup(renamed_files)` | リネーム結果のJSON形式バックアップ作成 |
| バックアップ | `undo_rename(backup_file)` | バックアップからリネームを元に戻す |
| 一括操作 | `add_prefix(prefix, selected_indices)` | プレフィックス追加 |
| 一括操作 | `add_suffix(suffix, selected_indices)` | サフィックス追加 |
| 一括操作 | `add_sequential_numbers(selected_indices, start, digits, position)` | 連番付与（prefix/suffix選択可） |
| 一括操作 | `find_replace(find, replace, selected_indices, case_sensitive)` | 検索置換 |
| 一括操作 | `copy_original_to_new(unselected_indices)` | 非選択ファイルの名前を元に戻す |

### 2.2 リネーム操作の5種類

1. **プレフィックス追加**: ファイル名先頭に文字列追加
2. **サフィックス追加**: ファイル名末尾に文字列追加
3. **連番付与**: ゼロ埋め連番（prefix: `{連番}_{名前}` / suffix: `{名前}_{連番}`）
4. **検索置換**: 大文字小文字区別あり/なし
5. **元の名前コピー**: 非選択ファイルに元のファイル名を復元

### 2.3 ファイル名バリデーション

| チェック項目 | ルール |
|-------------|-------|
| 空文字 | 空文字・空白のみは不可 |
| 先頭/末尾スペース | 不可 |
| 無効文字 | `<>:"/\|?*` を含む場合不可 |
| Windows予約語 | CON, PRN, AUX, NUL, COM1-9, LPT1-9（大文字小文字不問） |
| 長さ | 255文字以内 |
| 末尾ピリオド/スペース | 不可（Windows制限） |

### 2.4 Undo/バックアップ機能

- **バックアップ先**: Windows: `%APPDATA%/pdf_merger_tool/undo` / その他: `~/.pdf_merger_tool/undo`
- **ファイル形式**: `rename_backup_{YYYYMMDD_HHMMSS}.json`
- **JSON構造**: timestamp, folder（絶対パス）, changes（old/new のペアリスト）
- **保持数**: 最新5件（古いものは自動削除）
- **Undo動作**: changesリストを逆順で処理し、rename_files()で元に戻す

### 2.5 衝突回避

- **事前チェック**: `check_duplicate()` で同名ファイル検出
- **実行時**: 衝突ファイルはエラーリストに追加してスキップ
- **自動連番**: 未実装（"file (1).txt" のような自動回避なし）

### 2.6 除外対象

| 種類 | 対象 |
|------|------|
| サブディレクトリ | 全て除外（フォルダ直下のみ） |
| 隠しファイル | 先頭が `.` のファイル |
| システムファイル | Thumbs.db, desktop.ini, .DS_Store |

## 3. legal_renamer.py 仕様

### 3.1 構成クラス

| クラス | 役割 | 設計特徴 |
|--------|------|---------|
| LegalTemplatePreset | テンプレートプリセット管理（6フォルダ種別） | frozen dataclasses, tuple使用で不変性保証 |
| DateHelper | 日付処理（YYYYMMDD生成・検証・枝番付与） | 静的メソッド中心 |
| EvidenceNumberer | 証拠番号自動採番（甲/乙 + 2桁連番） | VALID_PARTIES定数で制約 |

### 3.2 LegalTemplatePreset — テンプレートプリセット管理

#### メソッド一覧

| メソッド | 入力 | 出力 | 用途 |
|---------|------|------|------|
| `get_folder_types()` | なし | `[{key, label}, ...]` | フォルダ種別一覧取得 |
| `get_subtypes(folder_type)` | フォルダ種別キー | `[{key, label}, ...]` | サブ種別一覧取得 |
| `get_fields(folder_type, subtype)` | フォルダ種別 + サブ種別 | `[{name, label, default, required}, ...]` | フィールド定義取得 |
| `generate_filename(folder_type, subtype, fields, original_name)` | 全パラメータ | 生成されたファイル名 | パターン展開でファイル名生成 |

#### ファイル名生成フロー

1. 必須フィールドの空文字/空白チェック → ValueError
2. パターン内の `{フィールド名}` をフィールド値で置換
3. 値のトリム処理（前後空白削除）
4. 未置換プレースホルダーが残っていたら → ValueError

### 3.3 DateHelper — 日付処理

| メソッド | 機能 | 詳細 |
|---------|------|------|
| `get_today_yyyymmdd()` | 今日の日付取得 | `datetime.now().strftime("%Y%m%d")` |
| `validate_date(date_str)` | 日付検証 | 8桁数字 + 実在日付チェック（閏年対応） |
| `add_branch_number(base_name, existing_names)` | 枝番自動付与 | 衝突時 `_2`, `_3` を付与、欠番も埋める |

#### 枝番付与の動作例

| 既存ファイル | 新規ファイル名 | 結果 |
|-------------|--------------|------|
| なし | `20260214_契約書` | `20260214_契約書`（そのまま） |
| `20260214_契約書` | `20260214_契約書` | `20260214_契約書_2` |
| `file1`, `file1_2`, `file1_3` | `file1` | `file1_4` |
| `file1`, `file1_2`, `file1_4` | `file1` | `file1_3`（欠番を埋める） |

### 3.4 EvidenceNumberer — 証拠番号採番

| メソッド | 機能 | 出力例 |
|---------|------|--------|
| `generate_numbers(party, start, count)` | 証拠番号連番生成 | `["甲01", "甲02", "甲03"]` |
| `format_evidence_filename(date, evidence_number, document_name)` | 証拠ファイル名生成 | `"20260209_甲01_準備書面"` |

- **当事者区分**: 甲（原告）/ 乙（被告）の2種のみ
- **番号形式**: 2桁ゼロ埋め（`str(i).zfill(2)`）。01〜99は2桁、100以上は3桁以上
- **枝番**: EvidenceNumberer自体には枝番機能なし。`甲01-1` 等は手動入力で運用

### 4.3 操作フロー

#### 基本リネーム操作

1. 📁ボタンまたはドラッグ&ドロップでフォルダを選択
2. ファイル一覧がTreeviewに表示
3. チェックボックスで対象ファイルを選択
4. 一括操作ボタン（プレフィックス/サフィックス/連番/置換）を実行
5. プレビューボタンで確認
6. 「名前変更実行」で確定（別スレッドで実行）
7. 必要に応じて「元に戻す」でUndo

#### 法律事務所テンプレート操作

1. フォルダ種別をドロップダウンから選択（例: "04 訴訟"）
2. サブ種別を選択（例: "書証"）→ 動的フィールドが自動生成
3. 各フィールドに値を入力（日付は今日の日付が自動入力）
4. 「テンプレート適用」をクリック
5. 選択ファイルに生成されたファイル名が適用
6. 同名ファイルには枝番（`_2`, `_3`）が自動付与

#### 証拠番号操作

1. 甲/乙をラジオボタンで選択
2. 開始番号を入力
3. 「証拠番号を適用」をクリック
4. 選択ファイルに `{日付}_{甲01}_{文書名}` 形式で適用

### 4.4 主要UI要素

| 要素 | 詳細 |
|------|------|
| ファイル一覧 | Treeview 4列（selected, original, new, extension） |
| ダブルクリック | 変更後ファイル名の直接編集ダイアログ |
| シングルクリック | 選択チェックボックスの切り替え |
| プログレスバー | determinate モード（リネーム進捗表示） |
| D&D | TkinterDnD利用（利用可能時のみ） |
| ヘルプ | ?ボタンでモーダルダイアログ（600x500）表示 |
| ToolTip | 全13ボタンにツールチップ設定 |

### 4.5 スレッド設計

- リネーム実行は `threading.Thread` で別スレッド
- メインスレッドとは `queue.Queue` 経由で通信
- メッセージ種別: `rename_progress`, `rename_complete`, `rename_error`

- `default="today"` が `YYYYMMDD` に自動変換されること
- 不正なフォルダ種別/サブ種別でValueError
- 必須フィールドが空の場合にValueError
- 前後空白の自動トリム
- 未入力フィールドが残っている場合のエラー検出

#### DateHelper（11ケース）

- YYYYMMDD形式の検証
- 無効なフォーマット（ハイフン含み、文字混入、桁数違い）
- 存在しない日付（平年の2/29, 13月, 32日）
- 枝番の衝突回避（`_2`, `_3`, `_4`...）
- 欠番検出と埋め（`_2`, `_4` が存在する場合 `_3` を返す）
- 空文字・空白のみでのValueError

#### EvidenceNumberer（14ケース）

- 甲/乙の連番生成（2桁ゼロ埋め）
- 不正な当事者区分でValueError
- start < 1 でValueError
- count < 1 でValueError
- 証拠ファイル名のフォーマット
- 各引数が空の場合のValueError
- 前後空白の自動トリム

### 6.3 テストから分かる設計思想

1. **入力は信頼しない**: 全ての入力値にバリデーションとトリム処理
2. **明示的なエラー**: ValueErrorで詳細な日本語エラーメッセージ
3. **immutable data**: frozen dataclasses + tuple で不変性保証
4. **GUI非依存**: コアロジックはGUIなしでテスト可能

## 8. テンプレート改善に向けた論点整理

殿のテンプレート改善方針判断に資する論点を以下に整理する。

### 論点1: テンプレート定義の外部化

**現状**: 6フォルダ種別のテンプレート定義がlegal_renamer.pyのPythonコードにハードコード。
**課題**: 事務所ごとのカスタマイズや新テンプレート追加にコード変更が必要。
**改善方向**:
- YAML/JSONファイルでテンプレート定義を外部管理
- ユーザーがGUI上でテンプレートを追加・編集できる機能
- **判断ポイント**: 現在の6種類で十分か、カスタマイズ需要があるか

### 論点2: テンプレートパターンの重複

**現状**: 02_協議と05_送付書類の発信/受信パターンが完全に同一。
**課題**: コード上の重複。ただし法律実務上はフォルダの意味が異なるため分離は妥当。
**改善方向**:
- パターン自体を共通化しつつ、フォルダ種別の区別は維持
- **判断ポイント**: 重複を解消する価値があるか、現状の明快さを優先するか

### 論点3: 証拠番号の枝番自動化

**現状**: `甲01`, `甲02` は自動採番。`甲01-1`, `甲01-2` の枝番は手動。
**課題**: 実務では追加提出等で枝番が頻繁に使われる。
**改善方向**:
- EvidenceNumbererに枝番生成機能を追加
- 既存の証拠番号リストを入力として、次の枝番を自動決定
- **判断ポイント**: 枝番の使用頻度と、手動入力の許容度

### 論点4: file_renamer.pyへのテンプレート変数導入

**現状**: file_renamer.pyには `{date}`, `{counter}` 等のテンプレート変数が未実装。
**課題**: 汎用リネーマーとしてはプレースホルダー展開がないため、カスタムパターンが作れない。
**改善方向**:
- `{date}`, `{time}`, `{original}`, `{counter}`, `{ext}` 等の基本変数を実装
- ファイルメタデータ（`{created}`, `{modified}`, `{size}`）の参照
- **判断ポイント**: 法律事務所向け機能がメインなら、file_renamer.pyのテンプレート変数は優先度低

### 論点5: 正規表現ベースの検索置換

**現状**: find_replace()は単純な文字列置換のみ。
**改善方向**: 正規表現パターンのサポート（グループキャプチャ、後方参照等）。
**判断ポイント**: 対象ユーザーのリテラシーレベルと需要

### 論点6: バックアップディレクトリ名の修正

**現状**: `pdf_merger_tool` という名前で固定。プロジェクト名（hananoen）と不一致。
**改善方向**: `hananoen` または `file_renamer` に変更。設定で可変にする。
**判断ポイント**: 既存のバックアップとの互換性への影響

### 論点7: file_renamer.pyのテスト補強

**現状**: rename_files()の実行テスト、Undoテスト、エラーハンドリングテストが未実装。
**改善方向**: 実ファイル操作を含むテストの追加。
**判断ポイント**: テンプレート改善前にテスト基盤を固めるか、改善と並行するか

### 優先度マトリクス（提案）

| 優先度 | 論点 | 理由 |
|--------|------|------|
| **高** | 論点3: 証拠番号枝番自動化 | 法律実務での使用頻度が高く、手動入力の負荷軽減効果大 |
| **高** | 論点7: テスト補強 | 改善前のリグレッション防止に必要 |
| **中** | 論点1: テンプレート外部化 | カスタマイズ需要次第だが、拡張性の基盤として重要 |
| **中** | 論点6: バックアップディレクトリ名修正 | 小規模だが品質向上に寄与 |
| **低** | 論点2: パターン重複解消 | 現状でも運用上問題なし |
| **低** | 論点4: file_renamer.pyテンプレート変数 | 法律事務所向け機能がメインなら優先度低 |
| **低** | 論点5: 正規表現置換 | 対象ユーザーにとって需要が限定的 |
